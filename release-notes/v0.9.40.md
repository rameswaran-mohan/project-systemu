# v0.9.40 — run-tree reconciliation fires at any genuine terminal (Bug 15 follow-up)

A correctness fix completing the v0.9.39 run-tree work. v0.9.39 made the request
cap genuinely run-tree-wide (verified: a SUB-AGENT run bounded 18 → 8), but its
**reconciliation** half was still incomplete on the dominant SUB-AGENT shape.

## Fixed — the run-tree reconciliation sweep was gated on the root execution

v0.9.39 reconciled the whole run-tree only at the terminal of the execution that
*is* the run-tree root (`execution_id == root_execution_id`). That holds for a
sub-agent **fleet** (the parent is the root and terminates last), but **not** for
a suspend→approve→resume **chain**: there the root is the *first* execution, which
**suspends** (and deliberately does not reconcile, since its request is pending),
while the execution that actually reaches a genuine terminal is the **last
resume** — which is not the root. So the tree sweep never fired, and a chain of
N distinct grants reconciled to a single `request-outcome` event (measured: 8
distinct sub-agent grants → 1 event).

Now the sweep runs at **any** genuine terminal, not just the root's. The
per-request_id collapse and the idempotent `already`-seen dedup (v0.9.37 Bug 11)
make redundant sweeps safe, so the last terminal to run reconciles every
still-open grant in the tree. Replaying the real run's ledgers through the fixed
path reconciles all 8 distinct grants (8/8) instead of 1.

The run-tree **cap** (v0.9.39) is unchanged and already correct. No behaviour
change for an autonomous, single-execution run.
