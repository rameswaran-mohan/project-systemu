# v0.8.5 — Decision dispatcher + scroll revalidation + workshop conditional notification + dry-run asyncio fix

The post-v0.8.4 live-vault audit surfaced four silent-failure bugs in the
rebuild → forge tools → approve shadow flow. v0.8.5 ships fixes for all
four, plus the missing dispatcher wire that was blocking decision-card
resolution from triggering immediate continuation (operators were waiting
up to 60 minutes for the hourly sweep to pick up their clicks).

## Fix A: `DecisionDispatcher` — no 60-min delay on Awaken clicks

**Symptom.** Operator clicks `Awaken` (or `Skip` / `Assign`) on a
`shadow_decision:*` card on `/insights → Pending Actions`. The card
disappears, but no shadow is created. The resolved choice sits in the
vault until the next hourly `hourly_shadow_sweep` re-invokes
`decide_shadow` — operator wait was up to 59:59.

**Root cause.** `OperatorDecisionQueue.resolve()` writes the choice to
disk but has no notion of "now continue the pipeline that posted this
decision". The continuation was implicitly bolted onto the next
scheduled sweep. Three dedup_key namespaces (`shadow_decision:`,
`validator_propose:`, `forge_tool:`) had this gap.

**Fix.** New module `systemu/approval/decision_dispatcher.py` introduces
a `register(namespace, handler)` / `dispatch(decision, choice, config,
vault)` registry. `shadow_decision.py`, `scroll_refiner.py`, and
`tool_forge.py` each register their continuation handler at import
time. The dashboard's `/insights` click handler now spawns a daemon
thread that calls `dispatch(...)` immediately after `queue.resolve(...)`.
Fail-soft: unknown namespaces are no-ops; handler exceptions are
caught + logged, never propagated to the UI thread.

## Fix B: Scroll re-validation on tool deploy + advance from `VALIDATOR_BLOCKED`

**Symptom.** Operator forges a tool that a `VALIDATOR_BLOCKED` scroll
was waiting on. Tool transitions to `DEPLOYED` correctly, but the
scroll stays `VALIDATOR_BLOCKED` forever — no automatic re-validation,
no UI hint that anything changed.

**Root cause.** `tool_service.heal_activities_for_tool()` only swept
`PARTIAL` activities; it had no hook to re-run
`validate_and_propose_tools` on blocked scrolls. Separately,
`_mark_scroll_linked()` (shadow-assignment continuation) only advanced
`ACTIVE → LINKED` — `VALIDATOR_BLOCKED` and `PENDING_APPROVAL` scrolls
that picked up a shadow stayed stale.

**Fix.** Added
`scroll_refiner.revalidate_blocked_scrolls_for_tool(tool_id, ...)`
which iterates every `VALIDATOR_BLOCKED` scroll, re-runs
`validate_and_propose_tools`, and advances satisfiable ones to
`PENDING_APPROVAL` (queueing a fresh "Tools Deployed — Ready for
Approval" notification). `heal_activities_for_tool` now calls this
helper after the partial-activity sweep, wrapped in try/except so
validator failures never block tool deploy. Renamed
`_mark_scroll_linked` → `_advance_scroll_after_shadow_assignment` and
broadened it to also cover `VALIDATOR_BLOCKED` and `PENDING_APPROVAL
→ LINKED`. Backward-compat alias retained for v0.9.0 deprecation.

## Fix C: Workshop conditional notification — no stale Approve card on blocked rebuilds

**Symptom.** Operator uses Workshop to rebuild a scroll that's still
missing tools. Workshop saves the scroll with
`status=VALIDATOR_BLOCKED` (correct, per v0.8.4) but ALSO queues a
notification with `Approve` / `Reject` buttons. Clicking `Approve`
silently errors inside `approve_pending_scroll` because the status
isn't `PENDING_APPROVAL`. Operator sees the card vanish, nothing
happens, no log line visible from the UI.

**Root cause.** `workshop_module.rebuild_scroll` unconditionally queued
a `scroll_approval` notification (`actions=["Reject", "Approve"]`),
regardless of what status the rebuild ended in.

**Fix.** Branch on `new_status`. For `PENDING_APPROVAL`, keep the
existing Approve/Reject card. For anything else (notably
`VALIDATOR_BLOCKED`), queue an info-only card titled
"⏳ Rebuilt Scroll Blocked — Tools Required" with `actions=["OK"]` and
`notification_type="scroll_blocked_info"`. The message directs the
operator to `/tools` to forge missing tools, after which Fix B's
auto-revalidate will queue a fresh Approve card.

## Fix D: Tool dry-run asyncio wrapper — no `RuntimeError` in dashboard event loop

**Symptom.** Initiating a tool dry-run from the dashboard (manual
trigger or tool reconciler) crashes with
`RuntimeError: asyncio.run() cannot be called from a running event
loop`. Dry-run never completes; tool stays in pending state.

**Root cause.** `tool_dry_run.py:371` called `asyncio.run(coro)`
unconditionally. From the dashboard NiceGUI event loop, that's
illegal — `asyncio.run` refuses to nest.

**Fix.** New `_run_coro_sync(coro)` helper checks for a running event
loop in the current thread. If absent (CLI / cron), uses
`asyncio.run` directly (fast path). If present (dashboard), offloads
to a fresh `ThreadPoolExecutor(max_workers=1)` worker that owns its
own loop. Exceptions inside the coro propagate to the caller in both
paths. Line 371 callsite now reads `result = _run_coro_sync(coro)`.

## Acceptance criteria (from spec)

1. From the dashboard, click "Awaken" on a `shadow_decision:*` card → shadow is created within 10 seconds (vs 60 min worst case today).
2. Operator's blocked scroll: after forging the last missing tool via /tools toggle, scroll transitions to `PENDING_APPROVAL` automatically (visible on /scrolls within 5s).
3. When a shadow is assigned to an activity whose scroll is `VALIDATOR_BLOCKED`, the scroll advances to `LINKED` (visible on /scrolls).
4. Workshop rebuild of a still-blocked scroll → no `Approve` button on the resulting notification card. The card title says "Blocked — Tools Required".
5. Tool dry-run can be initiated from the dashboard (via tool reconciler or manual trigger) without `RuntimeError: asyncio.run() cannot be called from a running event loop`.
6. All tests in `tests/test_v0_8_5_bundle.py` pass.
7. All existing tests still pass (no regressions).

## Migration

`pip install --upgrade systemu`. No `.env` changes.

For operators with already-stuck `VALIDATOR_BLOCKED` scrolls whose
missing tools were forged in a v0.8.4-era session: forge any tool (or
re-enable an existing one) on `/tools` — v0.8.5's
`revalidate_blocked_scrolls_for_tool` sweeps ALL blocked scrolls on
every deploy, so any single trigger will catch up the backlog.

For operators with shadow-decision cards that were resolved pre-v0.8.5
but never produced a shadow: the next `hourly_shadow_sweep` will pick
them up as before. No manual recovery needed — v0.8.5 only changes
the latency on NEW clicks.
