# v0.9.35 — record-time generalization toggle (reusable, parameterized SOPs)

A minor feature release: a record-time **generalization** toggle that controls how
abstract a recorded task is, and turns the captured specifics into operator-asked
**parameters** at execution — so one recording becomes a reusable, parameterized
SOP. Default behaviour is unchanged.

## New — generalization toggle (broad / standard / narrow)

When you start a recording (`record --generalize …` or the dashboard record
dialog), you choose how the task is generalized:

- **standard** (default) — today's behaviour, unchanged. A no-op end to end.
- **broad** — the recorded intent is abstracted to the goal ("Buy something
  online."), and the few clear **what/where** specifics (e.g. the product, the
  site) are lifted into **parameters**. At execution the operator is asked for
  each, **with the captured value pre-filled as an editable default** — accept it
  or change it. So a recording of *buy a Samsung Galaxy S24 on amazon.com*
  becomes a reusable "buy {product} from {site}" SOP. Incidental values
  (quantities, timestamps, keystrokes, scroll positions) are deliberately not
  lifted.
- **narrow** — the category and site are baked into the intent ("Buy a phone
  online from Amazon."); no parameters.

The parameters ride the existing structured-input (elicitation) rail, so the
operator answers them in the same form used for tool inputs; secrets route through
the URL-mode path and captured values are redaction-aware (no secret defaults).

The chosen mode is recorded per session/scroll, threaded through the analysis
(intent extraction → instructions → scroll) onto `Scroll.parameters` /
`Scroll.generalization`, and resolved at run time so the operator's confirmed
values reach the agent's actual prompt.

## Changed — capture filter renamed `scope` → `sources`

The v0.9.34.1 capture-app filter (record only one app vs everything) is renamed
to **sources** with value tokens `all` / `single`, freeing `broad`/`narrow` for
the generalization toggle above. The old surface is kept as **back-compat
aliases** for one release: the `--scope` / `--app` CLI flags (now hidden), the
`SHARING_ON_CAPTURE_SCOPE` / `_APP` / `_TITLE` env vars (read as fallback), and
the old `session.json` keys (still written + tolerated). New equivalents:
`--sources` / `--source`, `SHARING_ON_CAPTURE_SOURCES` / `_SOURCE_APP` /
`_SOURCE_TITLE`.

## Known limits (by design, this release)

- The captured-value-as-editable-default prompt is rendered by the dashboard
  form; a fully headless/CLI run has no typed-field renderer, so it proceeds with
  the captured defaults rather than presenting an editable form.
- "broad" salience is LLM-judged (product/site-style what/where entities); the
  prompt encodes the salient-only rule + a don't-lift list, but the exact lift is
  the model's call.
- The SQLite/Postgres backends persist scroll structure as JSON blobs (the file
  vault is the primary store); the new `parameters`/`generalization` fields ride
  the blob, consistent with how `adherence`/`raw_request` were added.
