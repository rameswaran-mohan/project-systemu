# v0.8.2 — Fix `sharing_on init` seed-copy filename bug

Closes a silent failure mode that has been present since v0.7.4 when
`sharing_on init` was added: **the init command copied the starter
tool/skill INDEX but skipped every per-entry body file**, leaving operators
with a vault that promises 40 tools but contains 0 body files. Every
downstream `vault.get_tool()` raised KeyError, breaking activity
extraction, shadow assignment, scroll execution — silently.

## Root cause

`sharing_on/cli.py:849` (pre-v0.8.2):

```python
src_file = pkg_vault / kind / f"{entry_id}.json"
```

But:
- `entry_id` from the index is already `tool_abc123` (with `tool_` prefix)
- `Vault.save_tool` writes to `tools/tool_{tool.id}.json` — produces
  `tools/tool_tool_abc123.json` (DOUBLE prefix)
- `src_file.is_file()` returned False for every tool in the package
- Init silently skipped the copy, no error, no warning

**This is exactly what bit the user during v0.8.1 UAT:**

```
[Extract] existing_id tool_a1f69543 not found — creating new tool
Error: 'Tool not found: tool_a1f69543'
```

The user's vault had `index.json` with 40 entries and ZERO body files.

## Fix

```python
kind_singular = kind.rstrip("s")   # "tools" → "tool", "skills" → "skill"
src_file = pkg_vault / kind / f"{kind_singular}_{entry_id}.json"
```

Matches `Vault.save_tool` / `save_skill`'s actual filename convention.

Also adds a **WARNING log** when an index entry has no body file —
silent skip was the root reason this bug stayed hidden for ~3 weeks.

## Migration

```powershell
pip install --upgrade systemu      # → 0.8.2
sharing_on daemon stop --all
cd $HOME\systemu-tryout
sharing_on init --force            # re-seed with the fix
sharing_on daemon start
```

The `--force` flag re-copies all per-entry body files. The fix is
idempotent — existing scrolls/captures/notifications are not touched.

After init --force, the **startup recovery sweep** will pick up your
approved-but-stuck scrolls and re-run activity extraction (now succeeds
because tool bodies are present).

## Tests

`tests/test_v0_8_2_init_seed_copy.py` (4 tests):
- Init copies per-entry body files using the correct filename
- The OLD filename pattern matches zero files (proves the bug)
- Skills follow the same convention
- **Regression**: locks down that the real installed package vault has
  the expected file layout — if a future release changes the naming
  convention, this test fails immediately rather than silently breaking
  init for every operator.
