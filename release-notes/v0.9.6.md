# v0.9.6 — Proactive Surfacing + Scalability Correction (Layer 7)

**Release date:** 2026-06-07
**Layer:** 7 of 7 in the assistant-foundation arc — **arc complete**
**Headline:** Skills are now EARNED via auto-extraction, not prescribed. The curator + process registry + memory consolidator round out the L1–L7 foundation. The arc is complete; what ships from here is intent-matching, scale, and polish.

## Architectural correction (the most important change in this release)

Before v0.9.6, we shipped `burrito-delivery` and `find-nearby` SKILL.md recipes in `systemu/skills/` as "primary acceptance" examples. After a scalability review (cross-referencing a large generic-capability-tool catalog of ~60 tools and an auto-skill-extraction approach), we recognised this as overfit:

- **Generic capability tools** (read_file, web_extract, send_message, etc.) belong in `systemu/runtime/tools/` (v0.9.3+ code-registered batch).
- **Skills** should be EARNED via `auto_skill_extractor` from real successful runs (≥2 rounds OR ≥2 tool calls, confidence ≥ 0.6) — not PRESCRIBED by us.
- The `systemu/skills/` directory now ships **near-empty by design**: a README explaining the architecture + one truly generic starter (`summarize-page`, works for any URL).
- The earlier overfit examples (`burrito-delivery`, `find-nearby`) are preserved at `docs/skill-examples/` as format references for operators copying the SKILL.md shape into their own libraries.

This is the proven design: ship capabilities, let skills grow organically. The acceptance test going forward is **"after N varied runs, did the skill library grow with N relevant skills the agent now reuses?"** — not "did the burrito test pass."

## Post-review wiring correction (read this)

The initial T0–T4 commits landed the L7 modules with green unit tests — but a diligent review found that **all four feature modules had zero production call sites**. They were dead code: the unit tests passed only because they imported/called the functions directly. This is the same silent-failure pattern caught earlier in the arc (v0.9.1.1 dead fixes, v0.9.5 visibility gap). The following fixes make L7 actually run, each pinned by an `inspect.getsource` (or behavioural) wiring-assertion test so it can't silently regress:

- **process_list / process_check were invisible in production** (concrete bug). The registrations lived in `systemu.runtime.process_registry`, but boot discovery (`_discover_v2_tools`) only AST-scans `systemu.runtime.tools`. Fix: moved the `register()` calls into a boot-discovered shim `systemu/runtime/tools/process_tools.py`. Verified: the tools now appear in the registry after a production-style boot.
- **curator never ran**: `should_run()`/`mark_run_complete()` had no callers. Fix: added `jobs.curator_review_job()` (gates on `should_run()`, runs the real consolidation + skill-graduation lifecycle pass, records the run) and registered it as an hourly daemon job (heavy pass fires only when due, default weekly).
- **auto_skill_extractor — the "PRIMARY skill source" — never fired**, and **consolidate_run was dead**. Fix: `direct_task._maybe_extract_skill_and_consolidate()` at the chat-run completion seam. On a successful multi-step run it extracts + persists a SKILL.md (so skills are genuinely EARNED); on every completion it consolidates run memory. `build_result` was enriched (additively) with `tools_called`/`tool_calls`/`rounds` so the skill-extraction threshold can be applied.

## What changes (T0–T4)

### T0 — Close the v0.9.5 visibility gap
The v0.9.5 boot-time augmentation at `shadow_runtime.py:1759` augmented `tool_index` with v2-registered tools, but the live burrito test against v0.9.5 showed the LLM still didn't see them. T0 adds a **per-iteration** augmentation guard at `shadow_runtime.py:~2088` so `_user_payload["available_tools"]` is always rebuilt with v2 tools immediately before the LLM call. Belt-and-suspenders for any future filtering between boot and prompt.

### T1 — Inactivity-triggered curator (idle-curator pattern)
- `systemu/runtime/curator.py`: `.curator_state` JSON file format (`last_run_at`, `last_run_summary`, `paused`, `run_count`).
- `should_run(vault_root, config)` honors `curator_enabled`, `paused`, and `curator_interval_hours` (default 168 = 7 days).
- `mark_run_complete()`, `pause()`, `resume()`.
- Atomic writes via tempfile + rename.
- **NOT a cron daemon** — the daemon's `curator_review_job` (registered hourly) calls `should_run()` and only runs the lifecycle pass when the interval has elapsed. The pass reuses the proven `run_consolidation_for_all` (consolidation + skill-graduation) machinery; the richer forked pin/archive review remains Task 5 future work.
- 3 new Config knobs: `SYSTEMU_CURATOR_ENABLED`, `SYSTEMU_CURATOR_INTERVAL_HOURS`, `SYSTEMU_CURATOR_MIN_IDLE_MINUTES`.

### T2 — Process registry as v2 LLM tool
- `systemu/runtime/process_registry.py` — turn-surviving registry of background processes the agent started (long curl, build, async pipeline).
- Public API: `register_process()`, `mark_done()`, `check_process()`, `list_processes()`, `clear_completed()`, `clear_all()`.
- Two v2-registered LLM tools (`toolset="process"`): `process_list` + `process_check` — the LLM can inspect what it spawned across turns.
- In-process singleton (threading.RLock). Cross-process persistence (vault-backed) is a v0.9.7 enhancement.

### T3 — auto-skill-extractor (auto-skill-extraction pattern, **the primary skill source**)
- `systemu/runtime/auto_skill_extractor.py` — on a **successful** multi-step run, threshold-gates by `n_rounds >= 2 OR n_tool_calls >= 2`. Below threshold: no LLM call, return None.
- **Wired** at the chat-run completion seam (`direct_task._maybe_extract_skill_and_consolidate`), best-effort and never-raising.
- Calls Tier-1 LLM with `systemu/prompts/extract_skill_from_run.md` (conservative: return null when uncertain).
- Confidence-gated by `SYSTEMU_AUTO_SKILL_EXTRACT_MIN_CONFIDENCE` (default 0.6).
- `persist_skill_candidate()` writes a fully round-trippable SKILL.md to `SYSTEMU_SKILLS_USER_DIR` (or a vault-local `skills/earned/` directory when unset).
- This replaces the prior v0.9.4 "bundle pre-baked recipes" approach.
- 2 new Config knobs: `SYSTEMU_AUTO_SKILL_EXTRACT_ENABLED`, `SYSTEMU_AUTO_SKILL_EXTRACT_MIN_CONFIDENCE`.

### T4 — Memory consolidator
- `systemu/runtime/memory_consolidator.consolidate_run()` augments the v0.9.0/v0.9.1 fact_extractor with broader memory surface: facts_learned + patterns_observed.
- **Wired** at the chat-run completion seam (alongside T3), config-gated, best-effort.
- Idempotent via SHA256 fingerprint of chat_history (no double-pay on no-op re-runs); persists to `<vault>/memory_cache/<fp>.json`.
- Distinct from the pre-existing daily `consolidate_shadow_memory` job (which folds shadow buffers into `SHADOW_MEMORY.md`); this is a lightweight per-run consolidation surface.
- 1 new Config knob: `SYSTEMU_MEMORY_CONSOLIDATION_ENABLED`.

## Migration

- No vault migration. New surfaces (`.curator_state`, `memory_cache/`, extracted SKILL.md files) are created on first use.
- Existing v0.9.0–v0.9.5 features untouched.
- The overfit bundled skills (`burrito-delivery`, `find-nearby`) are NO LONGER in `systemu/skills/` — they live at `docs/skill-examples/` as format references. Operators who relied on them being auto-loaded must copy them into `SYSTEMU_SKILLS_USER_DIR`.

## Known limitations

- **v2 tools visible but underused**: T0 + the process-tools boot-discovery fix make all v2 tools (including `process_list`/`process_check`) present in the LLM catalog, but live testing still shows the LLM doesn't reliably call `skill_list_skills` first, sometimes ignores a forged tool's declared parameter names (e.g. calls `write_file(path=…)` when the schema says `output_path`), and sometimes doesn't emit a tool call at all. These are LLM **inference-reliability** gaps — prompt-engineering + intent-matching territory, v0.9.7. The runtime's verifier + stuck-loop guard correctly catch these and park the run rather than fake success.
- **Skill library starts empty**: per the architectural correction, no domain skills are pre-bundled. Operators see the library grow as they run varied tasks. First-run experience is "look at docs/skill-examples/ to understand the format."
- **Auto-skill extraction only fires on a clean success seam**: a run that parks on an operator decision (`PENDING_DECISION`) returns before the post-run hook, so no SKILL.md is extracted for parked runs. Broadening consolidation to parked/failed terminal states is a follow-up.
- **Curator review action is the consolidation+graduation pass, not yet the forked pin/archive review**: the curator is wired and runs on its idle cadence, but the richer review fork (pin hot skills, archive stale ones, dedup) is Task 5 future work that slots into `curator_review_job` ahead of `mark_run_complete`.
