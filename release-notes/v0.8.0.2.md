# v0.8.0.2 — UX guardrails

Hotfix for the four silent failure modes discovered during v0.8.0.1 manual UAT
on Windows. None of these change behavior for already-correctly-configured
installs; they just turn previously-silent failures into visible errors so
operators can fix them.

## Fixed

- **Daemon refuses to start when port is in use.** Previously a 2nd
  `sharing_on daemon start` would silently spawn an orphan; the original
  leftover would win the port-race and serve the new user's dashboard with
  stale config — causing recordings to land in the wrong vault, decisions to
  go missing, and the user to see "nothing happened" with no error.  Now the
  daemon prints a clear error with the offending PID and remediation steps,
  then exits 1.

- **`sharing_on daemon stop --all`** — new flag. Kills every
  `python.exe -m systemu.scheduler.daemon` process, not just the one tracked
  in the pidfile. Use when orphan daemons have accumulated.

- **Every CLI verb loads `.env` from CWD.** Previously only `daemon start`
  did this implicitly; `sharing_on analyze`, `init`, `decisions`, `scrolls`,
  `tools`, etc. all ignored the user's working-directory `.env` and reported
  "OPENROUTER_API_KEY not set" even when the key was right there.

- **Dashboard health banner.** Top-of-page banner now surfaces three issues
  before they bite:
  - 🛑 More than 1 systemu daemon running (with `daemon stop --all` CTA)
  - ⚠️ OPENROUTER_API_KEY missing or empty (LLM steps would silently fail)
  - 🛑 Vault directory read-only

  Banner is silent when healthy.  Renders on every page load (no auto-poll
  yet — deferred to v0.8.1).

## Not yet fixed (deferred to v0.8.1)

- Record dialog showing the absolute save path before the recording starts
  (so the user can visually verify the daemon's CWD before committing).
- Sticky red toast when a JobManager subprocess ends with non-zero exit OR
  `jobs.log` contains "LLM generation failed".
- Banner auto-poll (currently page-load only).

## Migration

`pip install --upgrade systemu`.  No env changes needed.

If you have orphan daemons from previous installs:
```
sharing_on daemon stop --all
sharing_on daemon start
```

## Background

The four guardrails landed after a manual UAT session caught a multi-daemon
port-race that silently swallowed an entire recording.  Full debug trace in
the internal repo (`docs/superpowers/reports/2026-05-26-uat-pypi-v0.8.0.md`).
