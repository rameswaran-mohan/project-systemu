# v0.8.3 — queue-mode no longer gated on `not isatty()`

Closes a Windows-specific silent failure that survived v0.8.0 through v0.8.2:
shadow-creation prompts (and any other `notify_user` call with a `dedup_key`)
silently auto-selected `actions[0]` instead of routing to the dashboard's
Pending Actions queue, even when the operator had `SYSTEMU_DECISION_QUEUE=true`
explicitly set.

## Root cause

`systemu/interface/notifications.py:282` (pre-v0.8.3):

```python
if _decision_queue_enabled and not _sys.stdin.isatty():
    # queue-mode branch
```

A background daemon spawned from an interactive PowerShell inherits the
parent's TTY stdin handle, so `sys.stdin.isatty()` returns True even when
no actual interactive operator is reachable. Queue-mode SKIPPED.

The legacy headless-detection block immediately after uses a broader
OR-of-signals predicate:

```python
is_headless = (
    not sys.stdin.isatty()
    or os.environ.get("SYSTEMU_HEADLESS") == "1"
    or _console_non_unicode      # ← cp1252 stdout encoding (Windows default)
    or non_interactive
)
```

So in the same daemon: queue-mode skipped (isatty=True), headless triggered
(cp1252=True). Result: every `shadow_decision` prompt silently auto-selected
"Skip", scrolls never progressed past activity assignment.

## What the user actually saw

The v0.8.2 user, after init-fix landed and activity extraction succeeded:

```
[Shadow] LLM decision: CREATE_NEW | hint: WeatherDoc
[Notify] Headless mode — auto-selecting 'Skip' for: New Shadow Recommended
[Shadow] User skipped shadow creation
```

The user never saw a prompt. Scroll execution dead-ended at "no shadow".

## Fix

```python
if _decision_queue_enabled:    # dropped: and not _sys.stdin.isatty()
    # queue-mode branch
```

When the operator EXPLICITLY sets `SYSTEMU_DECISION_QUEUE=true`, queue-mode
ALWAYS fires regardless of TTY status. The dedup_key + queue availability
already gate it appropriately for legitimate cases. Interactive operators
who want TTY prompts simply don't set the env var.

## Migration

`pip install --upgrade systemu`. No `.env` changes needed — your existing
`SYSTEMU_DECISION_QUEUE=true` will now actually route shadow-decision
(and tool-forge, scroll-approve) prompts to `/insights → Pending Actions`.

For your stuck scroll: restart the daemon, the recovery sweep will re-run
shadow-decision, and the "New Shadow Recommended" prompt will appear as a
decision card on the dashboard.

## Tests

`tests/test_v0_8_3_queue_mode_ignores_isatty.py` (4 tests, all pass):
- Queue-mode fires when `isatty()=True` (the daemon-from-PowerShell scenario)
- Queue-mode still fires when `isatty()=False` (the CI/Docker scenario)
- Queue-mode skipped when `SYSTEMU_DECISION_QUEUE` env var is unset
- Resolved decisions returned even when `isatty()=True`
