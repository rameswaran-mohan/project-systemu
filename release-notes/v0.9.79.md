# v0.9.79 — R-A14a (§15.1): the forged-network hard-DENY release gate

The release gate that binds R-A14a and every release until the OS-kernel egress jail (S2) ships.
Before this, a **forged/untrusted network tool** — one the daemon wrote for itself, carrying an
effect that reaches the network — was merely *approval-gated*: an operator could rubber-stamp it and
it would run with unrestricted network access, because no egress jail exists yet. That is now a
**hard refusal**, not a stampable card.

## What ships

- **Forged-network HARD-DENY (IMPL-13 / DEC-1).** A tool that is forged-by-the-daemon *and* carries a
  network-egress effect (net read/mutate, send-message, money-move, oauth) is now **refused outright**
  when no egress enforcer is present — refused **before** the approval gate (no rubber-stampable card)
  and **before** any subprocess is spawned (never launched-then-denied). Fail-closed: a forged tool the
  classifier can't clear is refused, never run.
- **The refusal is surgical.** It does **not** touch: the operator-connected MCP tier-2 path (those
  tools are never forged and actuate via the MCP call path, never the forged spawn); a forged
  *local-only* tool (no network egress → unchanged, still approval-gated); or a non-forged built-in
  network tool (vetted repo code → gated, not denied).
- **No forged/registry actuation rung pre-S2.** The new actuation-modality selector admits exactly the
  operator-connected `mcp` rung — a forged-tool rung and a registry-install rung are withheld until the
  egress jail exists, kept as one enforceable source of truth so the gate can't silently grow a rung.
- **Registry/untrusted stdio launches refused pre-jail.** Launching an MCP server subprocess from an
  untrusted/registry source is refused before the transport is even constructed. Operator-connected
  servers (the vouched default) are unaffected.

## Compatibility & scope

- **Behavior change, by design:** a forged network tool that previously reached an approval card now
  hard-DENYs. This is the intended IMPL-13 posture ("no kernel enforcer ⇒ forged-network DENY; the
  capability is absent, never silently ungated"). Non-forged tools, local-only forged tools, and the
  operator-connected MCP path are byte-identical.
- Consolidated CI release gate in `tests/test_ra14a_hard_deny.py` (the three §15.1 assertions). Full
  suite green. A focused three-lens adversarial pass on the gate ran before release and found **two
  high-severity holes**, both fixed at root: the DENY originally keyed only on a tool's stored effect
  tags, but a daemon-forged tool ships with **no tags** (never stamped) and a forged tool can
  **declare benign tags** in its own metadata — either way a net-exfiltrating forged tool slipped past
  to a rubber-stampable card. The gate now **re-derives network egress structurally from the source the
  backend is about to run**, ignoring the untrustworthy declared tags (mirroring the dry-run path's
  existing source re-derivation). A forged tool whose body reaches the network is denied whether or not
  its tags admit it.
