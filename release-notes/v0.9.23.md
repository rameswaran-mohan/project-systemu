# v0.9.23 — Set up in one command; never bricked by a model name

A focused release: secure first-run setup for `pip install` users, and a
runtime that survives a bad/renamed model ID instead of failing every task.

## `sharing_on setup` — safe, secure, one command

Until now, `pip install systemu` users never hit the installer's key flow
(that lived in the git-clone `install.py`), so they reached `daemon start`
with no key and a dead dashboard. New:

```
pip install systemu
sharing_on setup      # or just `sharing_on daemon start` — it runs this first
```

The wizard:

* **Key entry is hidden** (getpass — never echoed to your terminal or
  scrollback), **validated live** against OpenRouter *before* it's stored,
  written to a **0600-permission `.env`**, and only ever shown masked.
* **Model preset** stored as the *name* (`SYSTEMU_MODEL_PRESET=balanced`),
  not resolved model IDs — so when we fix a model name in a release, your
  install picks it up automatically (see below for why that matters).
* **Output folder** chosen and created.

`daemon start` now runs setup for you the first time if no key is
configured (in a terminal), or points you at it (headless) — it never
boots into a dashboard that fails every task.

## A dead model name no longer bricks your tasks

Field report: the `balanced` preset's tier-1 was `deepseek/deepseek-v4`,
an ID OpenRouter rejects ("not a valid model ID") — so every task on that
preset hard-failed. Two fixes:

* **Runtime fallback:** an "invalid/unknown model" error now degrades to
  the proven budget-default model for that tier (logged loudly) instead of
  failing the task. This auto-rescues installs whose `.env` already baked
  the dead ID, and makes the system resilient to future catalog drift.
* **Presets repaired:** the dead ID is gone; presets now use only model
  IDs with positive live evidence this cycle.

> **Upgrading?** Restart the daemon after `pip install -U` — a running
> daemon keeps the old code in memory until you `sharing_on daemon stop
> --all` and start again.
