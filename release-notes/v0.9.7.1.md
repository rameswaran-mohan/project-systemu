# v0.9.7.1 — Hotfix: per-objective verifier baseline timing

**Release date:** 2026-06-09
**Type:** Correctness hotfix (applies to both the intent engine and the legacy per-objective engine)

## The bug

The per-objective durable-outcome verifier (v0.9.1 Layer-4 contract) captured its
**baseline at the wrong moment**, so it rejected objectives that were genuinely
completed.

- `ObjectiveState.baseline` was never populated, so `process_completion_claim`
  always fell through to `state_delta.capture_baseline()` **at verify-time** —
  i.e. *after* the agent's tool call had already written the deliverable that same
  turn.
- The baseline therefore **absorbed the just-written file**, `compute_delta`
  reported nothing new ("state delta is entirely empty"), and the verifier
  rejected the claim with *"no durable evidence"* — for a file that was sitting
  right there.
- The agent (a tier-2 reasoning model) then re-attempted the same objective,
  hit the stuck-guard, and **never advanced to later objectives**. On
  multi-artifact tasks the run stalled before producing all artifacts.

This was masked for **single-artifact** goals by v0.9.7's goal-level acceptance
(which uses an epoch baseline, so everything present counts). **Multi-artifact**
goals exposed it.

## The fix

`execute()` now captures **one run-start baseline** *before* the agentic loop and
assigns it to each `ObjectiveState`, so `compute_delta` correctly sees everything
the run produces. The lazy post-write capture no longer runs.

- File: `systemu/runtime/shadow_runtime.py` (run-start `capture_baseline` + per-objective assignment).
- Applies to **both** engines — this is the base Layer-4 contract, not new-engine-only.

## Validation

A deterministic two-file task (`quotes.txt` → `summary.txt`):

| | Before fix | After fix |
|---|---|---|
| OLD engine success | **0 / 4** | **2 / 2** |
| NEW engine success | 2 / 4 (1 was a goal-level false-positive) | **2 / 2** |
| `summary.txt` produced | ~1 / 8 runs | **4 / 4 runs** |

Root cause was **not** model capability — the model wrote the file and claimed the
objective correctly every time; the verifier rejected its correct work.

## Tests

- `tests/test_v0_9_7_verifier_baseline.py` (3): pure `state_delta` baseline-timing
  invariant, `process_completion_claim` behavioral (mocked verifier), and an
  `execute()` wiring guard.
- Full regression: 8 pre-existing baseline failures, **0 new regressions**.

## Migration

- No migration. Behavior change is strictly a correctness improvement: objectives
  satisfied by files written in the claiming turn are now credited instead of
  falsely rejected.
