# v0.8.0.1 — Dashboard Pending Actions fix

Hotfix for v0.8.0. The marquee dashboard surface introduced in v0.8.0 —
`/insights → Pending Actions` — was non-functional in the default config
(`SYSTEMU_STORAGE=file`) due to a missing 2-method proxy in the FileVault
adapter. Discovered during post-publish UAT.

## Fixed

- **`FileVault.save_decision` / `get_decision`** — proxy methods added so
  the dashboard's `AppState.vault` (a FileVault wrapper around the raw
  Vault) can read and write OperatorDecision records. Without these, the
  Pending Actions tab silently rendered the empty-state message even when
  the CLI `sharing_on decisions list` clearly showed pending records.
  `SYSTEMU_STORAGE=sqlite` was unaffected (SqliteVault is used directly,
  no wrapper).
- **`OperatorDecisionQueue.list_pending` log noise** — the per-item
  `get_decision` failure was being silently swallowed at `logger.debug`.
  Promoted to `logger.warning` so future similar wrapper-gap bugs surface
  immediately in dev / UAT instead of being invisible.

## New regression test

`tests/test_decision_queue.py::test_file_vault_wrapper_proxies_decisions`
exercises the FileVault adapter directly (the previous test used the raw
Vault and missed the wrapper hole). Asserts the proxy methods exist AND
that a full round-trip through the wrapper works.

## Migration

No action required. `pip install --upgrade systemu` picks up the fix.

If you've been on v0.8.0 and saw the empty Pending Actions tab even when
queued decisions existed, this release fixes it. The decisions you queued
under v0.8.0 are still in the vault — they'll appear on the dashboard as
soon as you upgrade and restart the daemon.

## Not changed in this hotfix

- Daemon-multi-spawn issue (pre-existing; `sharing_on daemon start` can
  spawn under both venv Python AND system Python if both have `systemu`
  installed) — deferred to v0.8.1.
- Cosmetic `decisions resolve <bad_id>` shows the raw `KeyError` repr —
  deferred to v0.8.1.
