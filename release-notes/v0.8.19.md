# v0.8.19 — Agentic-UX hardening (validate → ask → show)

Three additive improvements to how the agent uses its tools, consults the operator, and surfaces its
plan. Where v0.8.18 made keyed tools *connectable*, v0.8.19 makes the agent's tool calls *trustworthy*
to watch: it validates its own arguments before running a tool, asks structured multi-option questions
instead of free-text-only prompts, and shows a live checklist of the objectives it's working through.
All three are safe-by-default and require no migration.

## R4 — Parameter validation → re-prompt

Previously, a tool call with a wrong-typed or missing argument failed *inside* the tool — a `KeyError`,
a `TypeError` deep in the implementation, or a confusing upstream error — with nothing telling the model
*which* argument was wrong or how to fix it. The model would often retry the same malformed call.

Tool calls are now validated against the tool's `parameters_schema` **before** execution. A wrong-typed
argument returns a precise, self-correcting error envelope (`error_type: "tool_param_invalid"`) that
names the offending parameter, the expected type, and what was received — so the model can re-prompt
itself and fix the call on the next turn instead of spinning.

The gate is deliberately conservative:

- **Type-checks always.** Any declared parameter with a wrong runtime type is caught.
- **Required only when explicit.** A parameter is treated as required *only* when the schema declares
  `required: true` for it. There is no risky "no default ⇒ required" inference, so a tool whose schema
  is loose won't suddenly start rejecting calls it used to accept.
- **Empty schema is a no-op.** A tool with no `parameters_schema` (every tool that predates this
  release) skips the gate entirely and behaves exactly as before.
- **Complements the existing backstop.** The pre-existing `TypeError` "Parameter mismatch" handler stays
  in place; the new gate just catches the common cases earlier, with a clearer message.

## R3 — Structured ask-user

The operator decision queue could only carry a plain question with a free-text answer. When a tool or
the runtime needed the operator to *choose* among a known set of options, it had to flatten that into
prose and parse a typed reply.

A new `request_choice()` posts a **structured** question through the same decision queue: a prompt plus a
list of named options, with a free-text **"Other"** escape hatch. The `/insights` pending-actions card
renders these as option pickers with a **Submit** button instead of a bare text field. `resolve()` now
accepts the JSON answer shape for structured questions, while plain (free-text) decisions are resolved
exactly as before. Nothing about the existing plain-decision path changes.

**Live caller — clarifying questions during scroll refinement.** R3 now has its first production caller:
the scroll refiner asks a structured clarifying question (through the operator decision queue) when a
captured request is **genuinely ambiguous** in a way that would materially change the outcome. The
behaviour is deliberately conservative — the Tier-1 refinement prompt emits `clarifying_questions` only
for real ambiguity (defaulting to none), so most refinements never ask. When the operator answers on
`/insights`, the refinement re-runs with those answers folded in; when the run is unattended (no operator
decision queue available, headless/batch), it **degrades gracefully** and proceeds with the original draft
rather than blocking. The session `scrolls refine` path is wired (its CLI now parks cleanly on a pending
question and resumes — exit 75 — on re-run, the same as `scrolls approve`); the chat refinement path is
unchanged for now.

## R2 — Live objective checklist

The runtime builds and completes objectives as it works through an activity, but that progress was only
visible after the fact in the event log.

The runtime now emits `objective_state` events as objectives are built and as each one completes. A new
**Live Objectives** pane on the Console renders these as a live `☑ / ▣ / ☐` checklist — done /
in-progress / pending — so the operator can watch the plan fill in as it executes. This is **read-only
observability**: it does not change execution behaviour, ordering, or retries in any way.

## Compatibility

All three changes are additive and safe-by-default:

- **R4** no-ops on an empty `parameters_schema` and only enforces `required` when a schema explicitly
  declares it — legacy tools are unaffected.
- **R3** leaves plain decisions untouched; only the new `request_choice()` path uses the structured
  shape.
- **R2** is pure observability — a new event stream and a new read-only pane, with no change to how
  activities run.

No new dependencies were added. No migration is required.

## Migration

```
pip install --upgrade systemu==0.8.19
```

No new dependencies, no vault reseed, and no configuration changes are required. Existing tools,
activities, executions, and decision-queue behaviour are fully compatible — the three improvements only
add to what was there.
