# v0.8.12 — names instead of ids across operator surfaces

The dashboard now shows human-readable entity **names** wherever the operator
used to see a raw id. A small grey monospace **short-id** sits beside or beneath
the name as a companion, so the underlying id is still one glance away when you
need it. Entities that genuinely have no name — executions, submissions,
decisions — stay as short ids, but they are paired with a resolvable name
wherever one is in scope.

## What changed

A single shared resolver (`systemu/interface/name_resolver.py`) maps an entity
id to its name. It is **prefix-dispatched** (`shadow_` → `vault.get_shadow`,
`scroll_` → `vault.get_scroll`, and so on), **TTL-cached** (30s, so a busy live
feed doesn't hammer the vault), and **fallback-safe**: any miss, error, or
non-string name returns the raw id rather than breaking a render. Evolutions
have no name, so they resolve to a `"<target_type> evolution"` summary.

The change touches every operator-facing surface:

- **Supervisor live feed** — queued / executing / completed / dead-lettered /
  diagnosis events name the activity; the raw `activity_id` is kept in the event
  `context` for tooling.
- **Activities** table + detail — Scroll and Shadow columns show the name with a
  grey short-id beneath; unassigned shadows read "—" / "Unassigned".
- **Army** shadow detail — Skills / Tools / Activities joins render names instead
  of id lists.
- **Evolution cards** — both the dashboard proposal card and the engine's
  approval notification name the target entities.
- **Recover panel** — the panel title names the scope entity with a short-id
  companion line.
- **Chat** history — each row names the shadow that ran it; the execution stays a
  short id.
- **Scrolls** — the background-approval job label and toast name the scroll.
- **Skills** — evidence scrolls show the scroll name with a grey short-id.
- **Workflow detail** — linked-entity rows show names (the execution row, having
  no name, shows the short id only); the re-queue toast names the activity and
  swapped shadow, keeping the submission id as a short companion.
- **TERMINATE approval card** — the supervisor's operator card names the scroll
  and shadow; the execution id stays as a short companion.

Backend event builders (supervisor feed, evolution notify, TERMINATE card)
resolve names **at publish time** and keep the raw id in the event context, so
downstream tooling and logs are unaffected.

## Migration

`pip install --upgrade systemu`. No data migration; this is a presentation-layer
change. The resolver is fallback-safe, so a stale id (renamed or deleted entity)
simply renders as the id, exactly as before.
