# v0.8.14 — one click from the Console to the exact tool approval

A discoverability bundle for the approval loop that v0.8.13 introduced. v0.8.13
made chat tasks park on `waiting_on_tools` and close the unblock loop through a
review-then-approve flow (Fix 6) — but the operator still had to *find* the
blocking tool, hunt through the registry, and notice it amid stale chat history
that read like the current state. This release surfaces the approval one click
away, makes it prominent on the Console, and stops finished tasks from
masquerading as live. No new runtime capability — these are UX fixes for the
file-mode interface.

## Fix 1 — per-tool "Review & forge →" deep-link

Every proposed-tool row in the **Pending Tools** component now carries a
per-tool **"Review & forge →"** button that navigates to
`/tools?forge=<tool_id>`. The existing catch-all "Review proposed tools →"
footer button stays; the new button is per-row so the operator goes straight to
the right tool.

The `/tools` page reads the `forge` query param (NiceGUI passes query params as
page-function args, the same pattern as `page_insights(tab=…)`): `page_tools`
gained a `forge` arg that threads into `build_tools_page(forge_tool_id=…)`. When
present, the page **auto-opens the spec/code review dialog** for that tool after
render (via a one-shot `ui.timer`, so the dialog opens after the slot stack has
settled). This reuses the existing v0.8.13 forge dialog and its reviewed
**Approve & Sign Off** path unchanged — review-then-approve still installs deps,
enables the tool, and heals waiting activities. A bad/unknown id is tolerated
(no-op + notify), it does not crash the page.

End to end from a notification or the Console: *click "Review & forge →" → land
on `/tools` with the spec/code dialog already open → review spec + code + deps →
Approve & Sign Off → tool enabled → parked chat task heals and runs.*

## Fix 2 — Console "Pending Approvals (N)" section, auto-expanded when pending

The Console gains a top-level **"Pending Approvals (N)"** expansion rendered
right after "Pending Actions". It renders the **Pending Tools** and **Tool
Dependencies** components together, and is **auto-expanded** (`value=N>0`)
whenever there is anything to approve — so a pending approval is visible on
arrival rather than buried in a collapsed "More" section.

`N` = proposed-tools count + pending-deps count, computed by a pure,
non-raising `_pending_approvals_count(vault)` helper (file-mode `DepApprovalStore`
read is wrapped in try/except so a missing `data/dep_approvals.json` yields `0`,
not an error). To avoid a double render, the now-superseded "Pending Tools" and
"Tool Dependencies" entries were **removed from the "More" expansions**
(`_build_expansions` keeps Learning Curves, Memory Status, Skills Snapshot).

## Fix 3 — chat page "Clear history" + de-emphasis of stale terminal entries

Old, finished chat tasks were rendering with the same weight as the live one,
so a stale `failure`/`success` row read like the current state. The chat page
now:

- de-emphasises **stale terminal entries** — rows whose status is terminal and
  which are *not* the newest entry are dimmed (reduced opacity) and tagged as
  previous, so only the current task reads as current. A pure
  `_stale_terminal_ts(entries)` helper returns the timestamps to dim. Terminal
  statuses include both `failed` (extraction/decision crash, written by
  `direct_task`) **and** `failure` (runtime result), plus `success`, `partial`,
  `skipped_no_shadow`, `cancelled` — so neither failure spelling slips through.
- adds a **"Clear history"** control (shown only when history is non-empty) that
  calls the new `Vault.clear_chat_history()` and re-renders. The vault method
  truncates the `elder/chat_history.jsonl` file and is safe to call when no
  history exists.

## Migration

`pip install --upgrade systemu`. No data migration. "Clear history" deletes the
local `elder/chat_history.jsonl` on demand; all other changes are interface-only.
This builds directly on v0.8.13's `waiting_on_tools` parking and Fix 6
review-then-approve loop — the same forge dialog and Approve & Sign Off path are
reused unchanged, only made reachable in one click and surfaced prominently.
