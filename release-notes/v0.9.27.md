# v0.9.27 — "near me" tasks no longer dead-end on a geocode miss

A small fix for a real failure: a *"best punjabi restaurant near me"* quick
task failed with **"iteration budget exhausted (12) without an answer"** and
produced nothing.

## Root cause (reproduced live)

The agent built the location string `"santhoshpuram, Chennai"` from the
operator profile. **Nominatim returns `200 []` for that combined
neighborhood+city string** — even though `"Chennai"` on its own resolves
fine. `geocode()` did a single lookup and gave up, so `find_places` returned
`"could not resolve location"`, the agent lost its one structured-results
tool, and flailed through web searches until it hit the 12-step budget.

Nothing else was wrong: the daemon was current, the LLM ran cleanly on every
call, and the provider path was untouched. One geocode miss cascaded into the
whole failure.

## The fix

`web_access.geocode()` now degrades gracefully: it tries the full string,
then **progressively coarser comma-trimmed fallbacks** (drops the leading,
most-specific segment each step), so `"santhoshpuram, Chennai"` →
`"Chennai"` and the lookup succeeds. Comma-free strings make exactly one
lookup as before — no extra Nominatim traffic for the common case. This fixes
the entire *"X near me"* class, not just one query.

Verified live after the fix: `geocode("santhoshpuram, Chennai")` resolves to
the Chennai center and `find_places` returns real nearby POIs.

## Known follow-ups (not in this release)

- Overpass cuisine-tag matching is loose — *"punjabi restaurant"* returns
  nearby restaurants, not strictly Punjabi ones.
- When a tool genuinely can't help, the quick lane still burns its full
  iteration budget rather than degrading to a useful partial answer sooner.
