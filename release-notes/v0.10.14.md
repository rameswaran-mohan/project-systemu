# v0.10.14 — The assistant sees its most relevant tools first (R-CAP1, live selection)

The capability index (v0.10.13) now feeds the assistant's tool selection. When it picks a tool for a
task, the most relevant candidates are shown **first** — so it reasons over a well-ordered list
instead of a flat dump.

## What changed

- **Relevance-ordered tool lists.** Two selection surfaces — the quick task lane and the skill/tool
  extraction pass — now order their tool list most-relevant-first for the task at hand, ranked
  deterministically by capability slot and name match.
- **Nothing is hidden.** This is strictly a reordering: every tool keeps its full schema and stays in
  the list, so the assistant can still choose any of them. When a task has no clear signal, the order
  is left exactly as it was — the reorder only kicks in when relevance actually differs, and never
  reshuffles into an arbitrary order.
- **Safe by construction.** A tool's own description is never a ranking signal (so it can't be gamed),
  the ordering is deterministic and reproducible, and any hiccup falls back to the original order
  untouched. Tool *resolution* is unchanged — this only affects the order the assistant reads.

An adversarial review (2 lenses × refute-by-default panels) confirmed every safety invariant holds
(never-subtract, defensive fallback, determinism, no order-dependent side effect) with zero real
findings; a dismissed observation prompted the "leave order unchanged when there's no relevance
signal" refinement anyway. Full suite green.

## Scope

The forge side of capability governance — refusing to forge a tool that already exists and offering
to extend one instead (CAP-5/CAP-6) — is the next slice.
