# v0.9.47 — stop the runaway forge / stuck-run loops

When the agent pulled at a goal it couldn't actually satisfy (e.g. *"encrypt
`report.bin`"* when no such file exists), it could grind for minutes — forging
tool after tool, re-opening the forge review, and re-searching the filesystem —
sometimes badly enough to need a daemon restart. This release makes those paths
**fail fast** instead of looping.

## Fixed

- **Forge review no longer re-opens at gate 1.** Approving a forged tool from the
  Tools page could bounce you back to the spec gate and loop. The `?forge`
  deep-link is now status-guarded (only an actually-*proposed* tool opens the
  review) and the query is cleared after approval, so a websocket reconnect mid-
  heal can't re-trigger the dialog.
- **A stuck objective is force-finalized, not re-parked.** Once the auto-coach
  budget is spent and an objective still shows no progress, the run finalizes as
  *partial* instead of re-parking — which previously spawned more forge requests
  each round.
- **The loop guard catches futile-but-varied repetition.** The same tool failing
  over and over with *different* arguments (e.g. searching path after path) now
  trips the guard; before, only identical-signature loops were caught.
- **No-progress detection survives a resume.** The no-progress counter is now
  carried across a resume, so a resumed run doesn't restart stuck-detection from
  zero.
- **The harness-request cap is cumulative per activity.** A new per-activity cap
  survives resumes and retries; the previous per-run cap reset each cycle, which
  let the total number of pulled requests grow unbounded.
- **A grant-resume doesn't burn a retry.** Approving a pulled capability is
  forward progress — it no longer pre-consumes one of the run's limited retry
  slots.
- **Recursive `file_list_dir` is bounded.** A "search the whole drive" step used
  to hang on a 30-second sandbox timeout. Recursive scans now have depth and
  result caps and refuse a bare drive-root / home-root scan, returning
  immediately.

## Under the hood

These are runtime-reliability fixes; the happy path is behavior-preserving. Each
fix landed test-first (18 new tests).

No eval/paper changes.
