# v0.9.29 — web_search actually returns snippets

The deferred follow-up from v0.9.28 (point 4). Keyless `web_search` was
returning results with **empty snippets** — only titles and URLs — so any task
that needed to *read* or *rank* results (like "best X near me") got link lists
with no substance and kept drilling. The v0.9.28 loop redesign already
prevented that from ending in a bare failure; this makes the search itself
useful.

## Root cause

`web_access._parse_ddg_results` deduped results by URL and kept only the title
link, hard-coding `snippet=""`. Real DDG-via-Jina markdown renders **~3 links
per result, all pointing at the same redirect**: the title, the display URL,
and the snippet paragraph. The parser was throwing the paragraph away.

## Fix

Group the markdown links by their decoded target URL (first-seen order) and
surface the **longest prose anchor** as the snippet — excluding the bare
display URL (no spaces) and the title itself. Single-link / lite renderings
still get an empty snippet (safe fallback).

Live-verified: a real `best punjabi restaurant chennai` search now returns
snippets like *"Here is our list of the Best Places in Chennai to savour
Punjabi food. 1. Jamavar — The Leela Palace…"* instead of blank.

## Verification

New parser test on the real 3-link-per-result structure (title kept, prose
snippet surfaced, display URL excluded, one row per target). Existing
single-link parse test unaffected. Full suite green (3362 passed).
