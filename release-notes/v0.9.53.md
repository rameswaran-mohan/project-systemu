# v0.9.53 — EffectTag vocabulary + legacy backfill (G0)

**The shared, extensible effect language the open-world go-getter's safety net is
built on. Purely additive: no runtime behavior changes yet — it lays the vocabulary
that the action-governance gate, requirement stamping, and external-effect
verification will consume in later releases.**

This is milestone **G0** of the Open-World Go-Getter build (spec `UNIFIED-v2` §5.7 / §9).

## What ships

1. **`systemu/runtime/effect_tags.py` — the vocabulary.** `EffectTag` is a str-valued
   enum (`local_read/write/delete`, `shell_exec`, `net_read/net_mutate`,
   `send_message`, `money_move`, `oauth_call`) with an **`UNKNOWN` sentinel**. Two
   load-bearing properties:
   - **Open vocabulary.** `coerce()` maps any unrecognized effect to `UNKNOWN`
     (which the gate treats as *dangerous-until-proven* ⇒ REQUIRE_APPROVAL — never a
     refusal), and `register_effect_tag()` lets a future modality declare its own
     effect class at runtime. Determinism *classifies and gates* a proposal; it never
     rejects the plan.
   - **High-severity predicate.** `is_high_severity()` backs the spec's §5.7 two-band
     DENY floor (an `UNKNOWN` effect that *also* carries a high-severity signal —
     delete/net-mutate/send/money — fails closed to DENY rather than a rubber-stampable
     card). `UNKNOWN` alone is not high-severity.

2. **`classify_source(code)` — a deterministic AST classifier.** Scans tool source for
   effectful sinks (network read/mutate by HTTP method, shell, local read/write/delete)
   and returns a tag set. It is an **advisory, escalate-only SIGNAL**: unparseable
   source ⇒ `{UNKNOWN}`, and the **absence of a tag is never "no effect"** (the real
   network boundary is the OS-kernel egress jail, a later release).

3. **`Tool.effect_tags: List[str]`** — a new, default-empty field on the tool model.
   Stored as plain strings (not `List[EffectTag]`) so the foundational `core.models`
   stays free of a runtime-layer import cycle.

4. **Legacy backfill** — `vault_migrator.backfill_effect_tags()` classifies every
   existing vault tool from its implementation source and stamps `effect_tags` onto its
   body JSON. Version-gated via a `.effect_tags_seed` marker (independent of the
   seed-tool fast path), so it runs **once per version bump** and is a no-op on every
   boot after. It **never raises** — a per-tool classification failure leaves that
   tool's tags empty and moves on, so it can't break daemon boot.

## Why additive-safe

`effect_tags` defaults to `[]` and is ignored by every existing code path; the backfill
only writes a body key that wasn't there before. No gate consults it yet — that starts
with S1 (`evaluate_action`). Existing tools with no readable source stay `[]` and are
treated as `UNKNOWN`-at-gate when the gate lands.

## Tests

- `tests/test_effect_tags.py` (15 tests) — the vocabulary + open-world coercion +
  extension registration, the high-severity predicate, the AST classifier per sink
  class (incl. `self.session.post(...)` attribute chains and unparseable ⇒ UNKNOWN),
  and the backfill (a legacy tool gains tags; idempotent second run; never raises on a
  missing impl file).
