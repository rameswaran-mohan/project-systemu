# v0.9.41 — cap-denied requests are a first-class taxonomy entry (Bug 15 follow-up)

Completes the request-outcome denominator. With the run-tree cap (v0.9.39) now
genuinely bounding a sub-agent run, the requests it turns away were invisible:
a cap-exceeded DENY wrote no ledger row at all (only the grant path logs, via
materialise), so over-delegation never appeared in the taxonomy.

## Added — `cap_exceeded` pull-failure category (outcome `denied_cap`)

A per-run-cap DENY is now recorded and classified as its own thing:

- The arbiter flags a cap-driven DENY (`HarnessVerdict.cap_exceeded`).
- The loop records the arbitration row for that DENY (the grant path already
  logged via `materialise`; the deny path now logs explicitly), with a
  `cap_exceeded` marker on the outcome.
- Reconciliation emits outcome `denied_cap` with pull-failure category
  `cap_exceeded` — **never** folded into `wasted_request`. `cap_exceeded` is the
  highest-precedence rule for a deny, so it wins even when a fallback existed or
  the request looked premature.

Net effect: the request-outcome denominator now covers every arbitrated request —
grants **and** cap-denials — so a bounded sub-agent run reports both how many
grants it used and how hard it pushed past the cap (the over-delegation signal),
the most informative taxonomy cell for the sub-agent family.

Regular (non-cap) denials are unchanged (`denied_fallback_ok` /
`denied_fallback_failed`, `wasted_request`).
