# v0.8.4 — three UX-correctness fixes from the post-v0.8.3 audit

Bundles three "default path = failure path" fixes identified in the v0.8.3
audit of silent UX failures. Each was a real operator-visible bug where the
default behavior produced a dead-end state with no error or notification.

## Fix #1: Workshop's `rebuild_scroll` runs the validator + propose-bridge

`systemu/pipelines/workshop_module.py:rebuild_scroll()` previously saved the
LLM-rebuilt scroll with status=PENDING_APPROVAL **without running the
validator**. The v0.8.1 propose-bridge (which surfaces missing tools to the
operator via `/tools` page + decision card on `/insights → Pending Actions`)
only fired on the CLI refine path. Operators who used the Workshop UI to fix
blocked scrolls never saw proposed tools or decision cards.

**Fix:** extracted the validator+propose-bridge from `scroll_refiner` into a
shared `validate_and_propose_tools()` helper. `refine_scroll()` now delegates
to the helper (no behavior change). `workshop_module.rebuild_scroll()` calls
the same helper after rebuilding — if validator passes, status =
PENDING_APPROVAL; if it blocks, status = VALIDATOR_BLOCKED + tools proposed
+ queue card.

## Fix #2: Evolution APPROVED → auto-applies

`systemu/pipelines/evolution_engine.py:_notify_evolution` previously set
status=APPROVED on operator approval, then logged
*"Note: actual application … is Phase S3+"* and stopped. `apply_evolution()`
existed but was only reachable via `sharing_on evolve apply <id>` CLI with
no scheduler job invoking it. Operators clicked Approve in the dashboard
and nothing else happened.

**Fix:** immediately after `status=APPROVED`, call `apply_evolution()` with
the same scroll's config. Failure is logged but doesn't roll back — operator
can still retry via CLI. `_notify_evolution` signature now takes `config`
(matching its caller `run_evolution_check`).

## Fix #3: `OperatorDecisionQueue.get_resolved_choice` WARNs on corrupt skip

`systemu/approval/decision_queue.py:get_resolved_choice` silently skipped
index entries whose body file couldn't be loaded (the v0.8.2 init-bug class).
Operators would be re-prompted for already-resolved decisions and their
original click would be lost. The silent `continue` masked vault corruption.

**Fix:** changed silent `continue` to a `logger.warning` that names the
corrupt header id, dedup_key, and the underlying exception. Still continues
(so a single corrupt entry doesn't break lookup), but now surfaces the issue
in daemon logs.

## Tests

`tests/test_v0_8_4_bundle.py` (7 tests, all pass):
- `validate_and_propose_tools` helper is importable at module scope
- Workshop rebuild invokes the helper
- Workshop rebuild sets VALIDATOR_BLOCKED when validator blocks
- Evolution Approve triggers auto-apply
- Evolution Reject does NOT trigger apply
- `get_resolved_choice` WARNs when body is missing
- `get_resolved_choice` returns choice + emits no warnings on healthy vault

Full sweep: 1186 pass / 9 skipped / 0 failed.

## Migration

`pip install --upgrade systemu`. No `.env` changes.

For operators with already-stuck scrolls from a v0.8.3-era Workshop rebuild
that ended at PENDING_APPROVAL without going to VALIDATOR_BLOCKED: open
`/scrolls` → click rebuild with any prompt. v0.8.4 will run the validator
and either pass it through or surface proposed tools.

For evolution proposals stuck at APPROVED: `sharing_on evolve apply <id>`
to apply manually. v0.8.4+ approvals auto-apply.
