# v0.9.44 — forge "Enable & run" no longer hangs when you click fast (the dry-run race)

A follow-up to the v0.9.43 forge re-dispatch fix. v0.9.43 wired the heal sweep
into the Inbox "Enable & run" resolver, which works **when the tool is already
dry-run-passed at resolve time**. But the "Enable & run" gate appears right after
the forge gate, so an operator commonly approves it **before** the background
reconciler finishes the tool's dry-run.

In that case Gate-3.5 correctly refuses to enable a not-yet-dry-run-passed tool —
but nothing retried the enable once the dry-run passed. The tool ended up
`DEPLOYED`-but-`disabled`, the parked task hung, and the resolver's heal sweep had
already run as a no-op (the tool was still disabled).

## Fixed — the reconciler completes a deferred "Enable & run"

`ToolReconciler` now runs a sweep on every pass: for each **resolved** "Enable &
run" (`tools_blocked`) gate, it enables any of that gate's tools that are now
dry-run-passed-but-disabled and fires the heal sweep so the parked activity
re-dispatches. The reconciler is the right place — it's the point where a tool
actually becomes dry-run-passed.

- Closes the race in **both** orderings: approve-before-deploy (this fix) and
  approve-after-deploy (v0.9.43's resolver heal).
- **Recovers runs already stuck by it** — a tool sitting `DEPLOYED`+`disabled`
  with a resolved "Enable & run" is enabled and healed on the next 30s tick.
- Idempotent (already-enabled tools are skipped); a tool whose dry-run hasn't
  passed (or failed) is left disabled, as Gate-3.5 intends.

No eval/paper changes.
