# v0.10.13 — A searchable capability index + `find-tools` (R-CAP1, foundation)

Your tool catalog gets a deterministic index and a way to search it. This is the foundation for
smarter tool selection at scale (§5.5.1 Capability Slots) — landed safely, without changing how any
task currently picks tools yet.

## What ships

- **A capability index.** A background job (every 60s) derives `capabilities/capability_index.json`
  from your whole catalog — every enabled tool plus every connected MCP tool — tagging each with a
  canonical *slot* (`create:issue`, `send:email`, …), its effect tags, and usage. It's a pure derived
  cache: fully rebuildable, and it holds names and shapes only — never a secret value.
- **`sharing-on find-tools <query>`** — search your catalog deterministically (no model call, no
  budget). It ranks the **complete** store, so every tool is reachable — the ranking only *orders*,
  it never hides. Example: `sharing-on find-tools create an issue`.

## Safety & discipline

- **Single writer.** Only the background reconciler writes the index; the search reads a fresh
  in-memory view without writing. The concurrency map's CI guardrail enforces this.
- **No rank inflation.** A tool can't climb the results by reporting its own success, and for an
  action-bearing query (create/send/delete/…) a connected third-party tool can't outrank a built-in
  one by stuffing its name or description with keywords — trusted origin wins, and a tool's
  description is never a ranking signal.
- **Additive.** This release does **not** change how the assistant currently selects tools during a
  task — the index and search are new surfaces alongside the existing behavior. Wiring the smarter
  top-K selection into the live task path is the next, separately-validated step.

An adversarial review (3 lenses × refute-by-default panels) caught four real issues before ship —
including one that would have left the production index empty (reading tool records the wrong way) —
all fixed and regression-tested. Full suite green.
