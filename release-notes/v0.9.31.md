# v0.9.31 — recorded tasks fail honestly and recover correctly

Four fixes for a real recorded-task failure where the run got 2 of 3 objectives
done, then died on a broken browser tool, retried into the same wall three
times, left the activity stuck "assigned", and "self-healed" by injecting tool
names that don't exist. RCA: it wasn't one bug — it was four.

## What changed

1. **`web_act` works again.** It perceived the page via `page.accessibility.snapshot()`,
   which Playwright removed (1.59 installed) → `'Page' object has no attribute
   'accessibility'` on every call. It now perceives via `aria_snapshot()` — the
   computed accessible names that `get_by_role(role, name)` already matches —
   so the one browser-action tool is functional again.

2. **Runs reach a terminal state and stop storming.** A run that hits the
   iteration cap now returns an **honest partial summary** (objectives done /
   total, the uncompleted goals, the tools that structurally failed). A tool
   that *structurally* fails (a non-transient circuit-breaker trip — not a
   timeout/5xx) flags the run, and the supervisor **skips the retry** (re-running
   won't help) instead of looping three times. Either way the activity is marked
   terminally **FAILED**, not left orphaned at "assigned".

3. **The refinery stops inventing tools.** Its failure appraisal was never given
   the tool registry and its own prompt example hard-coded a non-existent tool,
   so "PRIOR FAILURE" hints recommended tools that don't exist. The appraiser is
   now **grounded in the real registry** (told to only name available tools, or
   say the capability is missing), and any tool-shaped name not in the registry
   is **flagged** before it's written into the scroll.

4. **A refined scroll re-maps its tools/skills.** Refinement used to append a
   text hint but leave the activity's frozen (broken) tool mapping — the
   extractor's idempotency guard always returned the existing activity. On a
   structural failure the refinery now clears the activity link + marks the
   scroll `APPROVED`, so the next extraction recomputes the mapping with the
   refined hints (the recovery sweep / re-approval already handle that state).
   Transient failures keep the activity for the normal retry path.

Together: the broken tool is fixed; a genuinely-stuck run fails honestly once
instead of three times; the system never recommends a tool that doesn't exist;
and a refined scroll actually gets a fresh capability mapping.

## Verification

TDD throughout — live-verified the `aria_snapshot` format + that `get_by_role`
resolves the parsed names; deterministic structural-failure signal (not a fuzzy
heuristic); idempotency-robustness preserved. Full suite green.
