# v0.9.77 — security hardening: pin the external read-back to its validated address

A defense-in-depth hardening of the external-verification net's independent read-back channel. The
read-back already refused any non-public or non-HTTPS target before connecting; this release also
**pins the connection to the exact address it validated**, closing a DNS-rebind window in which a
hostname could have resolved to a safe public address during the safety check but to an internal
address a moment later at connect time.

## What ships

- **The read-back connects to the address it vetted.** For a hostname target, the client now resolves
  it once, confirms the resolved address is public, and connects to *that* address — with no second
  DNS lookup that a rebind could hijack. TLS is unchanged: the certificate and SNI are still validated
  against the original hostname, so only the socket destination is pinned, not the identity check.
- Literal-IP targets were never a rebind risk and are unaffected; all existing SSRF refusals (private,
  loopback, link-local, cloud-metadata, and IPv6 literals embedding an internal IPv4) still apply
  before any connection.

## Compatibility & scope

- **No behavior change beyond the hardening:** the read-back's request shape, token matching, redirect
  refusal, and size cap are unchanged; only the socket destination is pinned. The net remains off by
  default and this path only runs when the net is armed. The full suite (5505 tests) is green.
