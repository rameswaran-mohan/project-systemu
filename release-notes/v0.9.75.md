# v0.9.75 — R-A13b-2ii-b: tool-declared effect classification + OAuth recognition

Completes the external-verification net's effect-classification layer. v0.9.74 taught the system to
recognize money-moving and message-sending effects from a curated signal map; this release lets a
tool **declare its own effect classification** in its metadata (for effects an automated scan can't
see), and adds recognition of **OAuth / identity** flows. Read/shadow-only — the net stays off and no
run outcome changes.

## What ships

- **Tools can declare their effect classification.** A tool may state its effect classes in its
  metadata, which the system now honors — useful for tools whose behavior an automated source scan
  can't infer. This is the intended path for tools that reach a payment or messaging capability
  through an opaque or generated call the scanner can't read.
- **A declaration can raise, but never lower, the money-move classification.** If a tool's own
  code shows a money-moving signal, that classification is applied **regardless of what the tool
  declares** — a tool can never declare-away a detected money-move. So the guarantee that a
  money-moving effect is only ever confirmed by an independent, host-pinned read-back stays intact.
- **OAuth / identity flows are recognized.** A curated, conservative set of well-known OAuth SDKs and
  identity endpoints lets the system classify authorization flows as their own effect type.

## Compatibility & scope

- **Read/shadow only, no behavior change:** classification feeds the shadow meter and the effect
  gate's existing stamp set; no run outcome changes in any mode, and the net remains off. The full
  suite (5478 tests) is green. No shipped tool declares effect classes today, so real behavior is
  unchanged; the declaration path activates for tools (and future generated tools) that opt in.
