# v0.9.61 — S4 + S3: fail-closed external-effect credit & the independent verifier

**A run can no longer report an external effect as "done" without an independent, deterministic
confirmation.** This bundles two security substrates of the actuation net — **S4 (R-A6, fail-closed
credit)** and **S3 (R-A7, the independent external verifier + IMPL-6)** — onto v0.9.60. Together they
close the "fabricated done" gap: an objective that mutates external state (a money-move, a net-mutation,
or any unclassifiable effect) is credited only when a real, deterministically-matched artifact proves it
happened — and a lost-response ambiguity can never silently re-submit a money-move.

This ships the **credit/verification** half of the actuation safety substrate. It does not by itself
enable any actuation tier — that remains gated on all of S1–S5 existing (S2, the OS-egress jail, is the
remaining one). Every non-external run is byte-identical to before.

## S4 — Fail-closed external-effect credit (§5.8)

- **The trigger.** An objective is stamped `requires_external_verification` from its capability's
  **EffectTag** (not the LLM's `output_type`); an **UNKNOWN/unclassifiable effect stamps it True too**
  (dangerous-until-proven — BLOCKER-3).
- **Fail-closed at every finalization site.** Such an objective is credited **only** on a persisted
  `ExternalEvidence.confirmed` bit. Any exception in the credit path, or its absence, ⇒ **not credited**
  + an `UNVERIFIED_EXTERNAL` operator card — never a fall-through. This holds at **all four** routes a run
  can finalize: the per-iteration credit, resume re-credit, and both the COMPLETE and stuck-park
  goal-level accepts. The local verifier is **advisory-only** for these objectives (a soft-pass alone
  never credits; and a false local reject can't block a *confirmed* effect — closing a double-submit
  hazard).
- Persisted via `ExternalEvidence` + `ExecutionSnapshot` schema **v5** (`_migrate_4_to_5`), additive and
  backward-compatible.

## S3 — Independent external verifier + IMPL-6 (§5.8 / §11.5)

- **Deterministic-only `confirmed`.** A new `ExternalVerifier` sets the `confirmed` bit **only** on a
  deterministic match — no LLM/model output can set it. Strategies: `api_readback` (same authenticated
  host, https, and **token-freshness** — a token that existed before the submit can't confirm),
  `email_confirm` (predicate match), `web_assertion` (advisory — can never alone confirm a money-move),
  `operator_attest` (raw, redacted, agent-uninterpreted evidence).
- **Money-move net (BLOCKER-3), fail-closed.** A money-move — `MONEY_MOVE`, a financial signal, **or an
  external effect that can't be classified** — must be confirmed by a strong channel (`api_readback`/
  `email_confirm`); advisory strategies are hard-gated out. The classifier biases toward gating when an
  external effect is financially ambiguous.
- **IMPL-6 — no double-submitted money-move.** On a transport-ambiguous failure of an effectful call
  (timeout-after-send, connection reset, 5xx-after-send), the runtime runs a read-back **before any
  retry**, keyed to a **client-generated idempotency key** written into the request before send (never a
  server-assigned token — the case where a server token was never received is exactly the double-submit
  hazard). Confirmed-present ⇒ credit with no re-submit; confirmed-absent ⇒ a retry is safe;
  indeterminate, or a target with no idempotency primitive ⇒ an operator card — **never** a false
  "absent", never a silent retry.
- **MASK.** Evidence is redacted (Authorization/Cookie/storage-state, plus token shapes incl. AWS/JWT/
  high-entropy values) before any persist, audit-write, or operator render.
