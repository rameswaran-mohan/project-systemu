# v0.9.0 — User Model + Persistent Context (Layer 1 of the assistant foundation)

systemu now knows who you are.

This is the first release of the v0.9.x assistant-foundation arc — a 7-layer
shift from "agentic task runner" to "intelligent personal AI assistant". Layer 1
introduces a persistent user model so every task can reference what systemu
knows about you instead of asking the LLM to re-derive it.

## What's new

### `sharing_on user init` — the wizard

Run it once. Answer 4 questions (name, location, timezone, default output dir).
Now every task knows you.

```bash
$ sharing_on user init
Your name [Rameshwaran]:
Where are you? (e.g. 'Bangalore, India'): Bangalore, India
Your timezone (IANA, e.g. 'Asia/Kolkata') [Asia/Kolkata]:
Default output directory [/home/rameshwaran/systemu-output]:
✓ Profile saved to /home/rameshwaran/systemu/vault/user_profile.json

Next: `sharing_on chat submit "..."` — systemu now knows you.
```

### Persistent facts about you

Beyond the 4-field profile, systemu accumulates **facts** about you in
`vault/user_facts.jsonl`. Facts are freeform statements with provenance,
tags, and confidence scores.

You can add facts explicitly:

```bash
$ sharing_on user remember "I prefer Italian food" --tag preference --tag cuisine
✓ remembered: [fact_a1b2c3d4] I prefer Italian food
```

By default, systemu also extracts facts automatically from your chat
conversations after each task resolves. The fact extractor is a small Tier-1
LLM call that's intentionally conservative. To turn it off:

```bash
export SYSTEMU_AUTO_EXTRACT_USER_FACTS=false
```

### `sharing_on user facts list / forget`

Review and prune:

```bash
$ sharing_on user facts list --recent 10
  [fact_a1b2c3d4] (explicit_user, conf=1.00) #preference #cuisine
    I prefer Italian food
  [fact_e5f6g7h8] (auto_extract, conf=0.85) #location
    Lives near Indiranagar, Bangalore

$ sharing_on user forget fact_e5f6g7h8
✓ forgot fact_e5f6g7h8
```

Forgotten facts are marked `superseded_by="forgotten"` in the audit log —
they leave a trail but stop influencing tasks.

## What this fixes

A live v0.8.22.3 run of "find top burrito places near me and rank them"
completed obj_1 and obj_2 in LLM working memory but stalled on obj_3 and
produced no output files. One reason: the scroll-refinement LLM had no way
to know **where** the user was, so a "determine the user's location"
objective frequently leads the task chain. With a populated profile, that
step disappears.

This is just the first layer. Layers 2–7 (episodic memory, capability
ledger, durable-outcome contracts, recipe fast-paths, goal orchestration,
proactive surfacing) follow.

## Privacy

The profile and facts are stored **locally** in `<your-vault>/user_profile.json`
and `<your-vault>/user_facts.jsonl`. They are never sent anywhere except as
context inside LLM calls you've already configured systemu to make.

## Upgrade

```bash
pip install -U systemu==0.9.0
sharing_on daemon stop          # if running
sharing_on user init            # one-time
sharing_on daemon start         # then resume normal use
```

No vault migration is required. Existing tasks continue to work; setting a
profile is purely additive.
