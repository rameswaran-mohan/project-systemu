# v0.8.1 — Pattern 3: validator-propose bridge

Closes the silent-failure gap discovered during the v0.8.0.3 manual UAT:
the operator recorded "Fetch today's weather" → scroll created → validator
correctly identified that no weather-fetching tool existed → scroll marked
`validator_blocked` → **no proposed tool ever appeared on the dashboard** →
the operator had no actionable surface to unblock it without manually
editing `.env`.

## Root cause

`scroll_refiner.py:420` (pre-v0.8.1):

```python
if not v_result.satisfiable and getattr(config, "auto_forge_tools", False):
    specs = getattr(v_result, "missing_tool_specs", None) or []
    # ... forge them
```

The validator-to-forge bridge was gated entirely on
`SYSTEMU_AUTO_FORGE_TOOLS=true` — a dev escape-hatch that bypasses
operator review. For every operator install (the default), the bridge
silently dropped the `missing_tool_specs` the validator emitted. Scrolls
stuck at `validator_blocked` with no actionable UI surface.

This was explicitly tracked as **Pattern 3** in the 2026-05-26 architecture
audit and deferred to "after Pattern 1 (OperatorDecisionQueue) is in
place" — which it has been since v0.8.0.

## What changed

**`systemu/pipelines/tool_forge.py`**

- New `propose_tools_from_specs(specs, scroll, config, vault) -> list[Tool]`
  helper. Creates Tool records with `status=PROPOSED` from validator-emitted
  specs (LLM-fills `parameters_schema`, `return_schema`, etc. via the
  existing Tier-2 `forge_tool_spec.md` prompt). Does NOT generate code —
  that's a separate operator-reviewed step on the `/tools` page.
- `forge_proposed_tools_from_specs` now delegates Step 1 to
  `propose_tools_from_specs` then does Step 2 (code generation) on the
  already-proposed tools. Same behavior as before for the dev escape-hatch
  but DRY.

**`systemu/pipelines/scroll_refiner.py`**

- Replaced the `auto_forge_tools`-gated bridge with a three-tier flow:
  1. **ALWAYS** call `propose_tools_from_specs` when `missing_tool_specs`
     are present (creates Tool records with `status=PROPOSED` so they appear
     on dashboard `/tools` page).
  2. **ALWAYS** post one decision card to the OperatorDecisionQueue
     (`dedup_key=f"validator_propose:{scroll.id}"`) so the operator sees
     the block on `/insights → Pending Actions` without having to navigate
     to `/scrolls` first. Card body lists the proposed tool names +
     descriptions with options `[Skip] [Forge All]`.
  3. **If `auto_forge_tools=True`** (dev escape-hatch preserved for
     unattended CI), also immediately code-generate the proposed tools and
     re-validate the scroll.
- `pipeline_trace.detail` now persists `missing_tool_specs` (was previously
  dropped) so `sharing_on scrolls show <id>` shows what the validator
  suggested.

## Migration

`pip install --upgrade systemu`. No `.env` changes needed.

If you have scrolls already stuck at `validator_blocked` from v0.8.0.x:
re-trigger the refiner via `sharing_on scrolls refine <session-dir>` and
the proposed tools will appear on `/tools` + a decision card will land on
`/insights → Pending Actions`.

`SYSTEMU_AUTO_FORGE_TOOLS=true` still works (auto-forge without review)
for unattended-CI use cases.
