# v0.9.1 — Durable-Outcome Contracts (Layer 4)

**Release date:** 2026-06-06
**Layer:** 4 of 7 in the assistant-foundation arc
**Headline:** "Objective complete" now means "durable evidence exists" — judged by an independent fresh-context Tier-1 LLM.

## What changes

- **Per-objective verifier hints.** During scroll refinement, Tier-1 generates a free-text `verifier` field for each objective describing what durable evidence proves completion (file path + entry count, audit-log action + recipient, chat-reply expectation, etc.).
- **Fresh-context verifier.** At each completion claim, the runtime captures a `StateDelta` (files added/modified, audit-log entries, chat reply, vault records) and hands it — along with the objective and verifier hint — to a fresh Tier-1 LLM with no conversation history. Only when the verifier returns `verified=true` does the runtime credit the objective. (Prior art: Odysseus `src/agent_loop.py`, Hermes `agent/background_review.py`.)
- **Action-tool audit log.** Tools tagged `is_action_tool=True` write an entry to `vault/audit/actions.jsonl` (or the equivalent sqlite/postgres table) on each successful invocation. The verifier reads these entries to credit action outcomes (email sent, message delivered) that don't produce files.
- **Rejection budget.** Three consecutive verifier rejections on the same objective trigger the existing stuck-loop guard. Per-turn verifier cap of 2 with a "fresh effectful work" gate prevents loops.
- **Resume re-credit.** On daemon restart, the verifier re-judges each uncredited objective against current durable state — objectives whose evidence already exists are credited without rerunning their tool path.
- **L3-readiness slots.** `Tool.toolset` and `Tool.max_result_size_chars` ship as free fields now (no enforcement) so the L3 tool-registry refactor doesn't require a model migration. `max_result_size_chars` IS wired into the sandbox truncation path.
- **fact_extractor short-circuit.** SHA256 fingerprint of recent chat history skips redundant Tier-1 calls when the conversation hasn't changed (Odysseus `memory_extractor.py` pattern).

## Operator-facing knobs

8 new Config fields with `SYSTEMU_*` env vars:

| Env var | Default | Purpose |
|---|---|---|
| `SYSTEMU_VERIFIER_ENABLED` | `true` | Global Layer 4 kill switch |
| `SYSTEMU_VERIFIER_PER_TURN_CAP` | `2` | Max verifier calls per LLM turn |
| `SYSTEMU_VERIFIER_REJECTION_BUDGET` | `3` | Rejections before stuck-loop escalation |
| `SYSTEMU_VERIFIER_MAX_CALLS_PER_RUN` | `50` | Reserved knob — full enforcement lands in v0.9.2; field accepted to lock the env var name. |
| `SYSTEMU_VERIFIER_TIER` | `1` | Which LLM tier the verifier uses |
| `SYSTEMU_AUDIT_LOG_ENABLED` | `true` | Action-tool audit on/off |
| `SYSTEMU_STATE_DELTA_FILE_PREVIEW_CHARS` | `200` | Per-file preview length in the delta |
| `SYSTEMU_STATE_DELTA_MAX_FILES_PER_SECTION` | `50` | Max files in each delta section |

## Privacy

Action-audit entries include tool params (e.g. email recipient + subject). Audit log is local-only under `vault/audit/`. Operators should be aware that file previews shown to the verifier LLM are bounded by `state_delta_file_preview_chars`.

## Acceptance

A clean install (`pip install -U systemu==0.9.1`) + `sharing_on user init` + `sharing_on chat submit "find top burrito places near me and rank them"` produces **real files on disk** at the verifier-declared paths. The user can `ls` the output directory and see results. No "complete in working memory only" state survives the verifier.

## Migration

- Existing scrolls without `verifier` fields preserve current behavior (credit on tool success).
- Operators wishing to keep v0.8.22.3 behavior can set `SYSTEMU_VERIFIER_ENABLED=false`.
- No vault migration required for the file backend. sqlite/postgres backends auto-create the `action_audit` table on next vault open.
