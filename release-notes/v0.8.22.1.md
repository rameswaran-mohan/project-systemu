# v0.8.22.1 — Resumable Chat Decisions + Loose Ends

Patch release. v0.8.22 made stuck-loop decisions visible in chat (inline card); this
release makes answering them actually resume the run, and clears four banked fixes.

## Fix R — Resume-after-decision (true snapshot resume)

A chat task that paused on a stuck-loop decision now **resumes** when the operator
answers. The stuck-guard writes a resume snapshot at park; a daemon EventBus
subscriber re-dispatches the activity on resolve (stashing the answer into the
snapshot); the runtime applies the answer at resume-start — "Provide hint" retries the
stuck objective with the hint and completed objectives intact, "Accept partial" /
"Cancel run" finalize cleanly. The stuck dedup-key is now execution-independent so the
answer survives across runs. (`runtime/resume_on_decision.py`, `runtime/shadow_runtime.py`,
`interface/notifications.py`, `scheduler/daemon.py`.)

## Fix 1 — Forge loop closed (three parts)

- **1a** The activity extractor now reuses any existing same-name tool regardless of
  enabled/status, so re-extraction never spawns duplicate PROPOSED tools.
- **1b** The chat readiness gate treats a disabled required tool as not-ready and parks
  the task as `waiting_on_tools` (auto-resumes when the operator enables it) instead of
  executing and burning iterations failing `GATE_3_DISABLED`.
- **1c** The waiting message says "Enable in Tools Registry" for a forged-but-disabled
  tool vs. "forge / approve dependencies" for a proposed one.

## Fix 2 — Queued chat tasks thread `chat_submission_id`

The Supervisor `submit()` and worker now carry `chat_submission_id`, so the inline
decision card + resume work for queued chat tasks, not just the synchronous path.

## Fix 3 — Cancel wording

An operator "Cancel run" no longer reads as a system "stuck" failure and no longer
carries the `StuckLoopDetected` error.

## Fix 4 — `tools_tried`

The stuck-decision prompt's "tools tried" line now excludes tools that ultimately
succeeded.

## Tests
