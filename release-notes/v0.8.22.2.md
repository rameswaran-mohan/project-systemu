# v0.8.22.2 — Tool Usability + Cross-Process Resume + Capture Bug Fix

Patch release. Three independent bug fixes surfaced by live verification of v0.8.22.1
and the F3/G3 architecture review.

## Fix 1 — `web_extract` LLM-friendly `fields` interface

**The bug.** v0.8.22.1 live verification on a real chat task ("find top burrito places near me") looped indefinitely with:
```
WARNING [Runtime] Tool web_extract failed: Invalid parameters for 'web_extract':
missing required parameter 'schema' (type object).
```
The vault tool record marked `schema` (a full JSON Schema for one record) as required with no example or fallback. The LLM kept omitting it because it had no way to construct a valid schema, every call failed Gate 2.5 (param validation), and the resume-with-hint flow couldn't help — the bottleneck was the tool's *interface*, not information the LLM lacked.

**The fix** (`systemu/vault/tools/implementations/web_extract.py`, `systemu/vault/tools/tool_tool_web_extract.json`, `systemu/prompts/extract_skills_tools.md`):
- New `fields: list[str]` parameter — pass `["name", "url", "rating"]` and the tool builds the JSON Schema internally (numeric fields like `rating`/`price`/`count` get `type: number`; everything else `type: string`).
- Heuristic fallback when neither `fields` nor `schema` is supplied: scans for the most-common repeated container (`li`/`article`/`section`/`div`/`tr`) and emits records using inferred field names (`name`/`url`/`price`/`rating`/`description`). Logs an `INFO [web_extract] no schema/fields supplied — used heuristic extraction (got N records)` so the operator can see what happened.
- Explicit `schema=` still wins when both are supplied — full backward compat for one release.
- Extractor prompt now shows the `fields=[...]` form as the worked example, so the Tier-1 LLM picks the new interface from the first attempt.

10 new tests + fixture page; existing 145 web/tool tests stay green.

## Fix 2 — Cross-process resume for CLI-resolved stuck decisions

**The bug.** v0.8.22.1's resume-after-decision feature is wired through an EventBus subscriber registered at daemon boot. `EventBus` is a process-wide singleton, so:
- Dashboard resolve (runs in-daemon) → subscriber fires → resume works ✅
- `sharing_on decisions resolve <id> --choice ...` CLI (a separate process) → `queue.resolve()` publishes to the CLI's EventBus → daemon's subscriber never sees it → task is marked resolved but **never resumes** ❌

The CLI command is the one the `PendingOperatorDecision` error message points operators at, so the gap was silently inconsistent.

**The fix** (`systemu/runtime/resume_on_decision.py`, `systemu/scheduler/jobs.py`, `systemu/scheduler/daemon.py`):
- Refactored the EventBus handler so its core "stash answer + re-submit" logic is callable from any path. New `_dispatch_resume(decision, vault, supervisor, data_dir)` is the shared entry point.
- Added a 15-second daemon-side poll job (`reconcile_resolved_stuck_decisions`) that walks the decisions index for `status=="resolved"` stuck `structured_question` decisions with full resume coords, and re-dispatches any that haven't been dispatched yet.
- Persisted `decision.context["resume_dispatched"]` flag stops double-dispatch across both paths and across daemon restarts.
- EventBus stays as the fast path; the poll is the safety net.

5 new tests + the 15 existing v0.8.22.1 EventBus tests stay green.

## Fix 3 — `capture_to_skill` fresh-extract branch (included here for the release record)

`capture_to_skill.py` called `.get("skill_ids")` on the result of `extract_and_process`, but that function returns an `Activity` (not a dict) — so the fresh-extract branch raised `AttributeError` whenever it was reached. Only the skill-reuse branch worked. The fix uses `Activity.required_skill_ids` and guards the `None` (extraction-failed) case. Landed between v0.8.22.1 and this release.
