# v0.8.22.2 — Capture → Skill fresh-extract bugfix

Patch release. Fixes a latent crash in the capture → portable-skill orchestrator
(`sharing_on capture export-skill ...`) when the captured scroll has no
pre-existing Skill linked to it.

## Fix — capture_to_skill fresh-extract branch

`extract_and_process` returns `Optional[Activity]`, but
`export_skill_from_capture` was treating the result as a dict
(`result.get("skill_ids", ...)`). The defensive `isinstance(result, dict)`
guard prevented an `AttributeError`, but silently coerced every Activity to
`[]` — so the fresh-extract branch always raised
`RuntimeError("produced no Skill")` even on successful extractions. Only the
skill-reuse branch (when a Skill already pointed at the scroll via
`evidence_scroll_ids`) worked.

The bug went unnoticed because the existing happy-path test stubbed
`extract_and_process` to return the same wrong shape (a dict with
`skill_ids`), so test and production were broken together.

Fix: read `activity.required_skill_ids` from the returned Activity and
explicitly handle the `None` (extraction-failed) case with its own
`RuntimeError`. (`systemu/pipelines/capture_to_skill.py`.)

## Tests

`tests/test_v071_capture_export_skill_orchestrator.py` — 3 new regression
tests that pin the real `Optional[Activity]` contract for the fresh-extract
branch (Activity → returns bundle path; empty `required_skill_ids` →
RuntimeError; `None` → RuntimeError). The existing happy-path stub was also
corrected to return a real Activity.

Red/green proven via stash (2 fail → 6 pass). Combined 57/57 pass across the
orchestrator, CLI subprocess, `extract_intent_aware`, `chat_fixes`, and
`scroll_pipeline` suites.
