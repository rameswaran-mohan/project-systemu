# v0.10.6 — Retry robustness + a compliance-controls seed

More backlog cleanup: one reliability fix and the first honest seed of a compliance surface.

## A durable retry can no longer collide across runs

When an activity fails transiently, its retry is armed on a durable record so it survives a daemon
restart. On one edge path (a worker-thread crash with no execution id) the record's key fell back to the
run's *shadow* identity — which is stable within a run but not guaranteed unique across runs, so two
different runs of the same activity failing at the same step could, in principle, collide and silently
drop one retry. The key now prefers the run's **unique submission id**, so distinct runs always get
distinct records, while re-arming the same run stays idempotent.

## A first, honest compliance control map

`compliance/controls.yaml` is a seed of a compliance surface (the full program is picked up later, at an
enterprise trigger). It maps fifteen control families to the **actual** systemu enforcement points —
least-privilege action governance, forged-network containment, fail-closed money verification, secrets at
rest, filesystem confinement, independent effect receipts, the audit trail, SSRF pinning, prompt-injection
containment, and more — each with a grep-verifiable mechanism and an **honest status**. It does *not*
claim everything is done: dashboard authentication and a tamper-evident ledger are marked *planned*, and
capture-consent *partial*. A one-line capture-and-consent disclosure was also added to the README.

## Scope

- Both are additive; nothing loosens. The compliance seed is documentation + a validity test, not a new
  runtime control. Full suite green.
