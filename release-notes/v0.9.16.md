# v0.9.16 — Operator visibility: nothing parks in silence

A run that pauses waiting on you should be impossible to miss. This release
closes a class of bugs where pending operator decisions and finished task
outcomes were invisible across the dashboard shell — and adds a Status surface
so "what happened to my task?" is always one click away.

## The bug this fixes

When a run got stuck and asked you a question (a `structured_question`
decision — "Provide hint / Accept partial / Cancel run"), **every "needs you"
surface reported zero**: the header badge said 0, the right-rail "Needs you"
section said "Nothing waiting on you", and `/inbox` was empty. All three
filtered to `context.kind == "gate"`, so non-gate asks were dropped — two
parked runs looked like idle ones.

## Fixes & features

- **Every pending decision is now counted and actionable.** One accounting
  module backs the header badge, the right-rail section, and `/inbox`: gates
  *and* asks (stuck-run questions, credential requests). Asks render with an
  inline **Answer** dialog (the same option pickers + free text as `/chat`),
  and answering routes through the normal resolve path — so it unsticks the
  parked run, not just the card.

- **New "Status" dropdown** (header, next to "Needs you"): a scrollable list of
  recent tasks with their status, outcome message, a click-through to the
  workflow, and the artifacts folder where produced files land. (Task outcomes
  are now persisted to the task log — previously the final summary lived only
  in memory, so no list could show it.)

- **Live pane, upgraded.** Each event is now a header row — timestamp + event
  name — with an expand arrow for its reasoning / tool params / tool result /
  LLM response, and an **Outcome / Artifacts** view for finished tasks.
  Decision events show their title instead of a blank line, and pending ones
  carry an inline Answer button.

- **`/inbox` History un-blanked.** The History section had been silently empty
  since it shipped — the decisions index stores slim headers without
  `context`, so the gate filter matched nothing. It now hydrates resolved rows
  and shows gates and asks alike.

## Verification

Every change was test-driven and the operator-attention surfaces are now
**seeded into the boot-the-real-dashboard smoke harness** — a stuck-run ask is
posted before launch, so the badge / rail / Status / Live rendering is
exercised populated on every route, and a regression fails CI instead of the
operator. That harness immediately earned its keep: it caught the Answer
dialog silently failing to open (a NiceGUI slot-disposal race that the v0.9.15
log filter was masking), now fixed with a stable-slot dialog host. Full suite:
2,945 passed, 0 failures.
