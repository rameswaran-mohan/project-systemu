# v0.9.56 — action-governance evaluator + trusted_inprocess-bypass close (S1a + S1b.1)

**The single deterministic policy that decides ALLOW / DENY / REQUIRE_APPROVAL for
every effectful call (the safety net's core), plus the first governance-wiring: a
forged tool that mutates externally can no longer run in-process at full daemon
privilege even if operator-marked `trusted_inprocess`.**

Safety-substrate milestone **S1** of the Open-World Go-Getter build (spec
`UNIFIED-v2` §5.7 / §13.3). **Depends on G0 (v0.9.53)** — it consumes the EffectTag
vocabulary + `is_high_severity`; this branch is built on top of G0.

## What ships

`runtime/action_governance.py` — `evaluate_action(ActionContext) -> (Verdict, reason)`,
the deterministic gate. It is the *net*, not the *driver*: the planner proposes
boldly; this decides. The load-bearing rules (all under test):

- **Effect is derived PRIMARILY from the tool's EffectTags (G0) + the target host.**
  The name verb-map and `is_destructive_param` are **positive-only escalators** (add
  danger, never clear it); a **self-declared HTTP method never clears** — a
  network-reachable target is a mutation unless the operator confirmed read-only.
- **Open vocabulary, two-band UNKNOWN.** An unclassifiable effect is
  `REQUIRE_APPROVAL` (gated, dangerous-until-proven) — never a refusal — **except**
  the narrow **DENY floor**: UNKNOWN ∩ a high-severity signal (irreversible /
  destructive-param / money) fails closed to `DENY` with an honest handoff, not a
  rubber-stampable card. This bounds *blast radius*, not *world breadth*.
- **Whole-token verb map** (`submit`/`post`/`upload`/`charge`/`file`/`issue`/`rsvp`
  → mutation/money/message) with **no false positive**: a local tool named
  `send_summary_to_log` (tool-side tags say local-only) is **not** escalated by its
  name — it stays `ALLOW`.
- Discovered/registry/first-use tools (`classification_trusted=False`) are gated on
  any effectful call regardless of a self-declared read-only hint.

`should_mask()` flags secret-bearing args for the evidence-redaction layer (§4.4);
MASK is orthogonal to the action verdict.

### The first wiring — trusted_inprocess-bypass close (§13.3)

`requires_subprocess_isolation()` now consumes a governance policy: a **forged** tool
carrying a dangerous effect-tag (`net_mutate` / `send_message` / `money_move` /
`oauth_call`) is **never eligible for the in-process fast path**, even if the
operator marked it `trusted_inprocess`. `trusted_inprocess` is a *speed* grant, not a
*governance* grant — it must not let a money-capable actuator run in-daemon with an
ambient secret. Narrow and defensive: it changes behavior **only** for a forged,
`trusted_inprocess`, dangerous-tag tool (forcing it out-of-process); built-ins and
benign forged tools are unchanged. `action_governance.requires_isolation()` /
`MUST_ISOLATE` is the policy.

## Why low-risk

The evaluator is additive (nothing routes verdicts to a card yet). The isolation
change only *adds* isolation for the narrow forged-∩-trusted_inprocess-∩-dangerous
case (safe direction), leaving every existing flow intact — verified against the
existing `test_wave2_sandbox_isolation` suite.

### Deliberately deferred — per-call forged-tool gating (a scope decision, not wiring)

Routing every effectful forged/tool call through `evaluate_action` to a decision card
is **not** in this release. It is a genuine scope decision, not mechanical wiring:
retrofitting per-call approval onto every `NET_MUTATE` tool would regress existing UX
(today an *enabled* tool runs freely after its one-time enable gate), because the
spec's per-call gate is designed for the **new go-getter actuation path**, not for
every pre-existing tool call. That scoping (new-actuation-only vs. all-tools) is a
product decision to make explicitly before wiring — flagged, not silently skipped.
