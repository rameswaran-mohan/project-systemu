# v0.9.7 — Intent-Driven, Self-Provisioning Engine (default ON)

**Release date:** 2026-06-08
**Headline:** The runtime stops verifying frozen per-objective checklists and starts verifying the **goal**. When it's missing a capability mid-run it **pulls** for one (tool / skill / access / compute / sub-agent) through a Governor instead of failing. Adherence is now a dial (free / guided / strict). The intent engine is **default ON**; set `SYSTEMU_INTENT_ENGINE=false` to fall back to the legacy per-objective engine.

## Why (the failure pattern this fixes)

The old engine baked per-objective `success_criteria` at refine-time and then policed them literally at run-time. Three independent failure modes (impl-path mismatch, output-path mangling, delta-timing) each rejected legitimate work — the agent would loop "round-about" or park a task it had actually completed. It also had **no way to ask for a missing capability**: an under-provisioned scroll just failed, even when forging or granting one tool would have unblocked it.

v0.9.7 inverts both: **verify the goal, not the cage** (control inversion), and **let the agent pull the harness it needs at runtime** (reverse-harness).

## Pillar A — Control inversion (goal-level verification)

- A goal verifier (`systemu/runtime/goal_verifier.py`) re-derives acceptance **from the goal at verify-time** against an epoch-baseline `StateDelta`, instead of trusting frozen per-objective criteria (locked decision §0.1 #1).
- The authoritative goal is the **raw user message, preserved verbatim** (`Scroll.raw_request`), falling back to the refiner's intent only when absent (locked decision §0.1 #2). The refiner/scroll becomes an advisory prior, not a contract.
- Wired at two seams: the `COMPLETE` gate (accept when the goal is met even if some per-objective criteria weren't individually credited) and the stuck-park (accept on goal-level success before parking). Both degrade safely: any verifier error falls back to the legacy path.

## Pillar B — Reverse-Harness (runtime pull-provisioning)

- New actions `REQUEST_HARNESS` / `ASK_OPERATOR` let the agent request a capability mid-run.
- A **Governor** (`systemu/runtime/governor.py`, one per execution) arbitrates each request through a deterministic risk-scoring arbiter (`harness_arbiter.py`) → GRANT / DENY / ESCALATE, mints scoped **capability leases**, materialises the grant, and appends every (request, verdict) to a per-execution **harness ledger** (`<vault>/harness_ledger/<exec>.jsonl`). Leases are revoked at terminal state — default-deny never leaks across runs.
- **Provisioner kinds:** TOOL (forge + same-run dry-run deploy so it's callable immediately), SKILL, ACCESS, COMPUTE (extends the run's iteration budget), SUBAGENT.
- **ESCALATE rails:** an escalation snapshots the run and surfaces an operator decision card; `resume_after_grant` resumes with completed objectives intact.

## Adherence dial + Settings presets

- `free` / `guided` / `strict` (`systemu/runtime/adherence.py`). `auto` resolves chat → free, record/SOP → the per-SOP adherence chosen at save-time (locked decision §0.1 #4) else guided. An explicit operator pin (`SYSTEMU_EXECUTION_ADHERENCE`) wins.
- Under **strict**, the lenient goal-level acceptance shortcut is suppressed at both seams — the per-objective / SOP contract is honored verbatim (Strict-Replay).
- Three coherent presets pairing adherence with harness auto-grant flags: `locked_sop`, `assisted`, `autonomous`.

## Governor LLM judgment (Phase 4.1)

- The deterministic arbiter resolves all LOW/HIGH and unambiguous MEDIUM cases. Genuinely-ambiguous MEDIUM cases are handed to a conservative LLM judge (`harness_judge.py`): GRANT (confidence ≥ 0.6) → lease; DENY → deny; **any error / malformed output / low-confidence GRANT → ESCALATE, no lease**. It can only ever downgrade the default ESCALATE — it never opens a hole. Toggle: `SYSTEMU_HARNESS_LLM_JUDGE` (default true).

## Learn-from-failure (Phase 4.2)

- Skill extraction is no longer success-only. On a **failed/partial** run with real activity, `auto_skill_extractor.extract_corrective_candidate` extracts an **anti-pattern** SKILL.md (`_type: anti_pattern`: what went wrong + the corrective approach) so future runs are warned. Same confidence/activity guards as the success path; never breaks completion.

## Latent bug fixes (Phase 4.3)

- **psycopg2 web-path import:** the Postgres vault backend now imports `psycopg2` behind a guarded sentinel, and backend dispatch **degrades to SQLite** (single warning) when it's unavailable — instead of crashing the web/action-audit path.
- **Headless destructive-confirm EOF:** `notifications.confirm()` wraps `click.confirm` in `try/except (EOFError, click.Abort)` and returns the default. Destructive gates pass `default=False`, so a dead/closed stdin means **deny**, not crash.

## Default-ON graduation + cost-regression note

`SYSTEMU_INTENT_ENGINE` now defaults to **true** (`_intent_engine_enabled`). The flag and the per-feature switches remain, so any deployment can opt out.

**Cost-regression comparison (measured, live OLD vs NEW).** A live A/B was run (`_costcompare.py`): the same task executed with `SYSTEMU_INTENT_ENGINE=false` then `=true`, against identical fresh copies of the seed vault, capturing outcome + token totals + LLM-call count (token accounting via a single patch on `llm_router.llm_call`). The discriminating file task (`quotes.txt` → `summary.txt`) was run **4 trials per engine** on the free-tier model.

| Engine | Reported success | Token mean (range) | Calls mean | `summary.txt` produced |
|---|---|---|---|---|
| OLD | 0 / 4 (all parked) | 72.7K (53K–116K) | 15.0 | 0 / 4 |
| NEW | 2 / 4 reported | 85.2K (60K–145K) | 19.8 | 1 / 4 |

Per-trial NEW/OLD token ratio: 1.17, 1.06, 1.25, 1.14 → **mean ≈ 1.15×**.

Findings (honest, multi-trial):
- **No material cost regression — the headline result.** NEW costs ~1.15× the tokens of OLD, consistently and within bound (the ~1–2 extra goal-verification calls). Variance is high on **both** engines (a ~2× thrash on one trial each) and is driven by free-tier model retries, not engine logic.
- **Quality is a precision/recall *tradeoff*, not a clean win.** OLD is too strict — it parked 4/4, rejecting `quotes.txt` for "no durable evidence" *even though the file existed* (the delta-timing fragility this release targets). NEW is looser: it correctly accepted the one genuine completion (both files written) and correctly parked two incomplete runs, but **once over-accepted** — reporting success with only `quotes.txt` present (it credited the inline quote-count toward the goal's intent rather than the literal `summary.txt`). Goal-level verification optimizes for intent over literal step-completion; that occasionally over-accepts.
- **Model-capability ceiling confounds raw success rate.** The free-tier model produced the required second file in only 1/4 NEW and 0/4 OLD runs, so success-rate cannot be cleanly attributed to the engine.
- **Wall-clock is NOT a reliable engine metric** — it is dominated by free-tier latency/retries (an OLD outlier hit ~1035s). Use outcome + token totals + call-count.
- The runs also exercised the new psycopg2 fallback live (`PostgreSQL vault init failed … → falling back to file`) without crashing.
- Flipping the default to ON produced **0 new test regressions** across the full suite (2372 passed; the 8 failures are the pre-existing baseline). The new branches are defensively wrapped (Governor hoist, goal verifier, `_intent_goal_success`, harness judge, adherence resolver all fall back to the legacy path on error).

**Recommended monitoring after upgrade:** watch per-run token spend and harness-ledger volume for the first cohort of runs. If a deployment sees a regression, set `SYSTEMU_INTENT_ENGINE=false` (full opt-out) or `SYSTEMU_HARNESS_LLM_JUDGE=false` (disable just the judge) and report the workload.

## Operator-facing knobs (new in v0.9.7)

| Env var | Default | Purpose |
|---|---|---|
| `SYSTEMU_INTENT_ENGINE` | `true` | Master switch for the intent-driven engine (goal verification + reverse-harness + adherence). `false` → legacy per-objective engine |

## Known limitations / follow-ups

- **Goal-verifier precision (the Pillar A tradeoff).** Goal-level verification deliberately favors *intent* over *literal* step-completion. In multi-trial testing it correctly accepted genuine completions and correctly parked incomplete ones, but **occasionally over-accepted** (reported success when a literally-required artifact was missing but the goal's intent was otherwise satisfied). This is the flip side of fixing the old engine's over-strictness. Follow-ups to tighten precision: have the verifier enumerate the goal's *required artifacts* and require each, and/or add a confidence floor for goal-level acceptance (mirroring the harness judge's `≥0.6` GRANT floor). Strict adherence (`SYSTEMU_EXECUTION_ADHERENCE=strict`) already suppresses the lenient shortcut for compliance-critical workflows.
- **Cost-comparison sampling.** The published numbers are 4 trials per engine on a free-tier model with high run-to-run variance; treat them as directional (the ~1.15× token ratio held across trials), not precise.

## Migration

- No vault migration. The harness ledger (`harness_ledger/`) and any extracted corrective SKILL.md files are created on first use.
- **Behavioral change:** the intent engine is ON by default. Most users want this (tasks that the legacy engine parked now complete). To keep the exact legacy behavior, set `SYSTEMU_INTENT_ENGINE=false`.
- All v0.9.0–v0.9.6 features are untouched and remain available.
