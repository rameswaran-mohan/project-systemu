# v0.9.68 — R-A11b-2: discovery-before-forge auto-reuse (§5.5)

**Before forging a duplicate, the agent now reuses the tool it already has.** When the agent asks
the harness to forge a new tool, the runtime first runs one deterministic pass over the tools it
already holds. If it's asking for a tool that already exists (by exact name), it gets that tool —
no duplicate forge, no operator review card for something already vetted. If nothing matches, it
falls through to the normal forge path, now *provably* after a discovery pass (§5.5 AC1/AC2/AC3).

## What ships

- **Exact-name auto-reuse at the forge seam.** A `REQUEST_HARNESS kind=tool` runs a discovery pass
  over the vault's DEPLOYED + enabled tools; an **exact-name** match reuses the existing tool at a
  LOW auto-grant (it's already operator-vetted code) instead of forging a duplicate — no new
  PROPOSED record, no re-review. Anything that isn't an exact-name match falls through to the honest
  forge/ESCALATE path.
- **The safety invariant:** auto-reuse confers no capability the agent could not already invoke
  directly — it only answers "I need X" with "you already have X," and the reused tool still hits
  the live action gate at call time. MCP tools are structurally excluded from reuse (they aren't
  vault tool records), which removes the third-party-description attack surface by construction.
- **Both outcomes are recorded** in the harness ledger: a reuse hit (`forge_avoided`) and a
  discovery miss (`searched N tools, best score below floor`) — so a forge is only ever requested
  after discovery found nothing, and the avoidable-forge rate is measurable from day one.

## Compatibility & scope

- **v0.9.47 forge-loop safety preserved:** reuse flows the same arbitration call (both request caps
  still count), runs exactly one deterministic pass (no search→forge→search loop), writes no
  PROPOSED tool, and a capped request is still a hard deny. Full suite (5157 tests) green.
- **Conservative by design:** only an *exact-name* match auto-reuses. Fuzzy near-match surfacing
  (with operator confirmation), extend-an-existing-tool, and cross-vault semantic dedup are a later,
  purpose-built capability-slots release. A stale reuse target (disabled/re-forged after arbitration)
  does not silently forge — the agent re-requests so a genuine forge gets its own review.
- **Scope (R-A11b split):** this is R-A11b **Module B** (discovery reuse). The PLAN-6 SSRF-guard
  consolidation (R-A11b-1) and durable discovered-path persistence (R-A11b-3) ride later releases
  where they have a live consumer.
