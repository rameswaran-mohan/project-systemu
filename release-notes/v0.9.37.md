# v0.9.37 — resume-lifecycle reconciliation, brand logo, chat busy indicator

A runtime fix completing the harness request-outcome instrumentation started in
v0.9.36 (Bug 9), plus a brand logo and a dashboard fix so the Live activity
indicator fires for chat tasks too.

## Fixed — request-outcome reconciliation across the suspend→resume lifecycle

v0.9.36 reconciled non-suspend terminals; this release closes the
**suspend→approve→resume** lifecycle that every HIGH-risk capability family
(forge a tool, attach an MCP server, delegate to a sub-agent) goes through, so it
now classifies as `granted_used` instead of being invisible.

A HIGH-risk request escalates → the run **suspends** → the operator **approves**
(which materialises the capability under the *original* execution) → the run
**resumes** under a *fresh* execution id and **uses** the capability. That
lifecycle splits across two execution ledgers: the request + its escalate/grant
arbitration rows + the lease-mint live in the original (pre-suspend) ledger, while
the tool is actually called in the resumed run.

Previously the terminal finalize reconciled only the fresh resumed execution —
whose ledger has no requests — so no `request-outcome` was ever written for the
lifecycle, leaving the pull-decision taxonomy blind for all HIGH-risk families.

Now:

- The terminal finalize reconciles the resumed run's tool calls against **both**
  the resumed and the original execution ledgers, so the lifecycle resolves to
  `granted_used` / `granted_unused`.
- Outcome reconciliation collapses **multiple arbitration rows for one request**
  (the escalate row from suspend + the grant row from approve) into a **single**
  outcome, preferring the most-resolved verdict (grant > deny > escalate).
- It is **idempotent per request** — a request that already has a terminal
  outcome is never re-counted.
- Suspends now **defer** reconciliation to the terminal pass (a pending request
  is not "unresolved"), so there is no premature `escalate_unresolved` to
  double-count against the eventual `granted_*`.

No behaviour change for autonomous (non-suspend) runs.

## Fixed — Live busy dots now fire for chat / quick-answer tasks

The dashboard's "working in the background" dots next to the **Live** header lit
up for capture→execute runs (Supervisor slots) but not for chat or quick-answer
tasks, which run on an untracked daemon thread. The busy count now also includes
in-flight chat-lane tasks (via the chat cancel-token registry), so the indicator
fires for every kind of background work.

## Added — brand logo

A proper app logo replaces the placeholder bolt glyph as the dashboard favicon
and sidebar mark, and appears in the README. It falls back to the bolt glyph if
the logo asset is absent, and ships inside the wheel.
