# v0.9.9 — UI/UX Revamp: Design System + Shared Command Layer + Unified Decisions Inbox + Command-Center Shell

**Release date:** 2026-06-10
**Headline:** A four-phase UI/UX revamp that turns the dashboard into a **command center** — one design system, one command/intent layer shared by the CLI and the web, a **unified Decisions Inbox** where every human gate lands (and Approve actually *executes*), and a persistent right rail ("Needs you" + "Live") on every page. This is a **UX + internal-architecture** release: agent behavior is unchanged; the wins are consistency, a single approval surface, and the elimination of dashboard↔CLI split-brain.

> **Scope honesty:** this ships **Phases 1–4** of the revamp plus full Inbox gate routing. Phases 5 (spine consolidations) and 6 (cleanup), and the harness grant-resume runtime feature, are **deliberately deferred** (see follow-ups). Everything here is additive and regression-clean — **0 new test failures** vs the pre-existing baseline; **0 new** UI-style-lint violations.

## What landed

### Phase 1 — Design system (`systemu/interface/design/`)
A single source of truth for the dashboard's look: `tokens.py` (Midnight-Indigo palette + spacing/radius/type scales → one generated `GLOBAL_CSS`), `primitives.py` (Card · StatusPill · Button{primary|ghost|danger|success} · Input · Tabs), and `icons.py` (one Material-Symbol per concept, no shared glyphs). A custom AST lint (`tools/lint_ui_styles.py`) forbids raw hex literals and inline `.style(f"…")` and is enforced by a pytest gate with a count baseline — re-theming is now "edit tokens only."

### Phase 2 — Shared command/intent layer (`systemu/interface/command/`)
The dashboard was a thin client that hand-built `[sys.executable, '-m', 'sharing_on', …]` argv in ~8 places (split-brain). Now there is **one** path both surfaces call:
- `CommandResult` — a structured result envelope (status/summary/data/stream-ref; Rich + JSON renderers; exit codes incl. the QUEUED/75 convention).
- `dispatch(verb, args, …)` — the single dispatch contract (in-process **or** streamed via the job manager); **0 hand-built argv sites remain**.
- `GateDescriptor` / `ToolViewModel` — one descriptor for every gate, one view-model rendered identically as a CLI table and a web card.
- The missing CLI verbs so the console can finish every job headlessly: `tools enable` / `recalibrate` / `show`, a **real** `doctor --apply`, and `settings set`. Tool-enable is consolidated onto **one gated policy + one mechanism** (toggle / recover / CLI all route through it).

### Phase 3 — Decisions Inbox + mode dial
**One queue, one id-space.** Every human gate becomes a `GateDescriptor` that round-trips through the existing vault `decisions` collection. The **unified card** shows risk · inline Inspect · an explicit "what Approve does" · the highlighted safe-default with destructive options visually distinct · the dedup id. **Approve always executes** the real action (deferred steps are labeled, never silent — this also fixes the evolution silent-no-op-for-non-UPGRADE bug). Two renders over the one descriptor: the web right rail (glance + quick-Approve), the full `/inbox` page (triage + history), and the console `decisions list` / `decisions resolve`.
A **gate mode dial** (Bypass · Risk-tiered · Approve-only) is always visible in Settings with a persistent **Bypass danger banner**; a non-negotiable **floor** (package-install + destructive-recovery + the egress/fs-write/credential capabilities) always asks even under Bypass, with editable per-type overrides.

### Phase 4 — IA shell + persistent right rail
The shared frame (`dashboard._build_layout`) gained a **persistent right rail** as a true third column: "Needs you" (the Inbox glance, with quick-Approve) above "Live" (streamed run events). Active-route highlighting is now exact (segment-aware, deep pages map to their spine parent — no more `startswith` false-positives), and a global **＋New** menu fronts Record-session / Submit-task. *Live-smoke verified in a real browser* (rail renders as a fixed third column; both sections populate).

### Gate coverage — every source flows into the one Inbox
Each gate source now routes source → Inbox → Approve, **exactly-once**:
- **scroll · dep · evolution · forge · recovery** — Approve **executes** the real action. dep installs once (and a long-standing allow-list-only-no-install bug is fixed); forge is single-owner (no double-forge); recovery is fed by a 45 s daemon scan-to-queue with **stale-row auto-expiry** (`OperatorDecisionQueue.expire_by_dedup_key` + `RecoveryEngine.scan_all`).
- **harness** — **surfaces** in the Inbox and records the operator's choice (the grant-resume execution is a deferred runtime feature; see follow-ups).

## Tests & verification (honest)
- New suites across all four phases + gate routing (design-system lint gate, command-layer, gate descriptors/adapters, inbox queue, gate-mode + settings, the route-* and recovery/harness coverage suites, the Phase-4 shell helpers).
- **Full regression: 0 new failures** — the full suite shows the **same pre-existing baseline failures** (legacy web-extract drift + a handful of test-isolation artifacts) on this release as on the prior base; nothing here introduces a new failure. **UI lint: 0 new** style violations.
- **Phase-4 live smoke:** the running dashboard was driven in a browser — the right rail paints as a fixed third column, "Needs you" populates from a pending gate (risk pill + Approve button) and "Live" mirrors real streamed events.

## Known limitations / follow-ups
- **Revamp Phases 5–6 pending.** Spine consolidations (Work / Build / Shadows / Insights + the nav rename to the 6 spines + inspect-before-approve + the real side-by-side remediation card) and the final cleanup/naming pass are not in this release.
- **Harness grant-resume (execute) deferred.** Harness requests now appear in the Inbox and record the choice, but the actual run-resume is a runtime feature: it needs the `shadow_runtime` suspend contract (snapshot + park on ESCALATE) + `_apply_harness_grant` + resume-coords-at-surface + a daemon `resume_after_grant` reconciler. The Inbox resolve stays render-only for harness until that lands.

## Migration
No vault migration. Everything is additive and backward-compatible; the design system re-exports the legacy theme symbols, and the gate mode dial defaults to **Risk-tiered** (the prior behavior). All v0.9.0–v0.9.8 features untouched.
