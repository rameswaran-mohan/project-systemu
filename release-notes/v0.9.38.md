# v0.9.38 — request-outcome taxonomy across all grant kinds, terminal sub-agent runs, robust forge dry-run

Three runtime fixes hardening the harness instrumentation and the forge
sandbox, plus two small dashboard polish items (a slightly larger brand mark and
a corrected Insights nav icon).

## Fixed — request-outcome taxonomy is now correct for every grant kind (Bug 12)

The pull-decision taxonomy classified usage correctly only for plain **tool**
grants. Two coupled defects skewed every other capability family:

- `granted_used` read `outcome["tool"]` exclusively. An **MCP** grant
  materialises namespaced tools under `outcome["mcp"]["tools"][*]` (registered as
  `mcp__<server>__<tool>`), and **compute / access / skill / sub-agent** grants
  materialise no single named tool at all — so all of them were reported
  `granted_unused` even when the capability was used. Usage is now resolved
  per kind via `Governor._granted_tool_names`: tool grants match the tool name,
  MCP grants match the namespaced names, and kinds with no materialised tool are
  treated as **N/A** (usage indeterminate → `granted`, never penalised as
  `unused_grant`).
- `premature_request` fired for **every** kind whenever the capability was
  requested before any local attempt (`attempts_before < 1`). For a
  concrete-gap kind — access, MCP, compute, sub-agent — requesting immediately is
  correct, not premature. The rule is now kind-aware: only **tool** and
  **skill** requests at `attempts_before < 1` are flagged premature.

## Fixed — capped / harness-granted SUB-AGENT runs now terminate (Bug 13)

A harness-grant SUB-AGENT pull run could park indefinitely instead of finishing:

- The harness-grant SUB-AGENT observation used "decompose and proceed" framing,
  which invited the run to keep requesting more sub-agents. It now uses the same
  **terminal** framing as the native fleet branch — proceed and COMPLETE the
  objective now; request another sub-agent only for a distinct, named sub-task it
  genuinely cannot do itself — and carries `fleet.terminal = True`.
- When the per-run request cap is exceeded, a **blocking** request previously
  ESCALATED (→ suspend → approve → resume → re-request), which could loop without
  ever reaching a terminal. The cap is now a hard **DENY** for blocking and
  non-blocking alike, so a capped run continues to COMPLETE/FAIL and reconciles;
  the cap resets on the next run.

## Fixed — forge dry-run sandbox is robust on Windows (Bug 14)

Two environment defects withheld otherwise-correct forged tools at the dry-run
gate:

- **WinError 267** ("the directory name is invalid") when the dry-run subprocess
  launched with `cwd=vault_root` before the vault directory existed. The backend
  now ensures the working directory exists before launching the child.
- **Exit 9009** ("command not found") when a forged tool shelled out to a bare
  `python` / `pip` and the interpreter directory was not on the restricted-env
  PATH (the daemon often launches via an absolute interpreter). The restricted
  child PATH now includes `sys.executable`'s directory (and its `Scripts` dir on
  Windows), so bare interpreter invocations resolve.

## Polish — dashboard

- The brand mark in the sidebar is slightly larger (28 → 34 px).
- The **Insights** nav icon uses the `insights` glyph and every nav glyph now
  renders in a fixed-width centered box, so a wide glyph no longer shifts the
  column alignment.
