# v0.8.13 — chat tasks wait on tools instead of failing tool-less

A focused chat-failure hardening bundle. Chat tasks were running with no usable
tools and reporting `failure`; operators had no one-click way to approve+install
a tool's pip dependencies; the startup recovery sweep looped forever; and the
extractor occasionally crashed on a near-miss `tool_type`. All four are closed.
No new capability surface — these are reliability fixes for the file-mode runtime.

## RC#3 — chat parks on `waiting_on_tools`, then auto-runs once tools deploy

When a chat task's required tools were not runtime-ready (freshly `PROPOSED` /
`FORGED`, or with dependencies still pending approval), Stage 4 executed anyway
with no usable tools and reported `failure` — a dead end for the operator.

`direct_task` now has a **readiness gate** (Stage 3.5). Before executing, it
checks every required tool against the shared `tool_is_runtime_ready` predicate
(extracted from `ShadowRuntime._load_tools` so the gate and the loader cannot
drift). If any tool is not ready, the task is **parked** as `waiting_on_tools` —
no doomed run — with the `activity_id` and blocking tool names recorded on the
chat-history entry. The gate runs before both the synchronous and the
`route_through_supervisor` execution paths.

The **chat page** renders the new status: a warning-coloured badge reading
"WAITING ON TOOLS" plus a `needs: <tools>` meta line so the operator sees exactly
what is blocking.

The loop is closed by the **recovery sweep's Pass 2**: when a parked activity's
tools deploy and the activity heals to runnable, a new
`_resume_waiting_chat_entry` helper flips the `waiting_on_tools` chat entry to
`running` and wires an EventBus completion update, so the chat row reaches a
terminal status automatically — no operator re-submit required.

## RC#3 (cont.) — one-click per-package dependency approval

The startup dependency audit used to emit a single batched advisory notification
(`actions=["OK"]`) that an operator could only acknowledge. It is now
**actionable**: per-package `Install <pkg>` actions plus a `pkg_tool_map` in the
notification context (type `dep_approval`), so each missing package can be
approved and installed with one click. The notifications page dispatches the
`Install <pkg>` action to `approve_and_install`, then re-runs the tool's dry-run.

`approve_and_install` is now **storage-aware** (Fix 5a — repairs the Recover
button in file mode). It previously always took the sqlite path and `KeyError`'d
on `SYSTEMU_DATABASE_URL` in file mode. It now branches: docker/sqlite mode
persists to the `tool_dep_approvals` table; **file mode** persists to the JSON
`DepApprovalStore` (the store the local installer actually consults). Both then
pip-install the package and re-run the dry-run.

## Fix 6 — the unblock loop closes through a review-then-approve flow

Parking a task is only useful if there's a clear way out of it. The unblock path
is now **review-then-approve**, reusing the existing forge **spec (Gate 1) + code
(Gate 2)** review dialog rather than a blind one-click enable:

- A `dep_reminder` notification is now **actionable** — a **"Review & approve"**
  action opens the spec/code review dialog for the blocking tool (`tool_id` in
  context).
- After the operator has *viewed the spec and the generated code*, the dialog's
  **Approve & Sign Off** path now also installs the tool's declared dependencies
  (`approve_and_install`), **enables** the tool (`FORGED → DEPLOYED + enabled`),
  and heals waiting activities (`heal_activities_for_tool`) — one informed
  approval, justified because the code was reviewed. Enable stays gated on that
  reviewed approval; there is no silent auto-enable.
- The readiness gate now parks the activity as `PARTIAL` (not just records the
  chat status), so it is always within heal scope regardless of whether the
  blocking tool was newly created or reused.
- `tool_service._heal_partial_activities` now calls `_resume_waiting_chat_entry`
  the moment a tool is enabled, so the parked chat entry flips to `running`
  **live** (not only at the next startup sweep).

End to end: *chat task needs a tool → parked `PARTIAL` + "WAITING ON TOOLS" →
operator opens **Review & approve** → sees spec + code + deps → **Approve & Sign
Off** → deps installed + tool enabled → activity heals → chat entry flips to
running → task executes.*

## RC#2 — recovery sweep initialises the pipeline and stops looping

`startup_recovery_sweep`'s Pass 1 re-extracted every APPROVED-no-activity scroll
via `approve_pending_scroll` → `extract_and_process`, which raised
`RuntimeError("Pipeline dependencies not initialised …")` because the scheduler
process never called `init_pipeline`. This spammed every restart and never
cleared the zombie scroll.

The sweep now calls `init_pipeline(_config, _vault)` before re-extraction. Pass 1
is also **bounded**: a new `Scroll.recovery_attempts` counter is incremented per
attempt, and once it reaches `MAX_RECOVERY_ATTEMPTS` (2) the scroll is terminated
to the new `ScrollStatus.EXTRACTION_FAILED` state (with an operator-facing event)
rather than being retried forever.

## RC#1 — `tool_type` near-misses coerce instead of crashing

`Tool(tool_type="web")` used to raise (`'web'` is not a valid `ToolType`), which
crashed extraction. A new `coerce_tool_type` helper maps any value to a valid
`ToolType` and never raises: exact enum value/member passes through; known
synonyms map (`web`/`http`/`url`/… → `api_call`, `scrape`/`browser`/`render`/… →
`browser_action`, and so on); unknown/empty/`None` fall back to
`python_function`. A `Tool.tool_type` before-validator routes all input through
it, and the extractor's `_upsert_tool` now passes the raw model value straight
through (so the LLM's `'web'` becomes `api_call`, not a flattened
`python_function`).

The extraction prompt was cleaned up to match: the invalid `screen_capture`
dependency row is gone, and a **HARD RULE** enumerates the five valid
`tool_type` values and explicitly tells the model never to invent others such as
`'web'` or `'screen_capture'`.

## RC#4 — chat-page refresh guard locked

A regression test locks the v0.8.12 `_should_schedule_refresh` behaviour
(disconnected / missing-attr / `None` clients → not scheduled; connected →
scheduled), so the post-run refresh guard cannot silently regress. No code change
was required — the guard already satisfied every case.

## Migration

`pip install --upgrade systemu`. No data migration. Existing scrolls gain a
`recovery_attempts` field defaulting to `0`. Chat tasks that previously failed
tool-less now park as `waiting_on_tools` and auto-run once their tools deploy.
