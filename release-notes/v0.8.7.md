# v0.8.7 — Remove execute dedup + Missed-schedule alerts

Two narrowly-scoped behavior changes on top of v0.8.6:

1. **Execute clicks no longer dedup.** v0.8.6 silently returned the existing
   job when the operator clicked Execute twice for the same `(shadow, scroll)`
   pair. v0.8.7 spawns a fresh subprocess every time, honoring operator intent.
   The dedup_key is kept as an informational field (used by UI to show "N other
   runs for this shadow+scroll are in flight" — not enforced).

2. **Missed schedules don't burst-fire on restart.** v0.8.6 fired one
   backlogged event per missed schedule on dashboard restart, then resumed.
   v0.8.7 detects staleness (default: 5 minutes, configurable via
   `SYSTEMU_SCHEDULE_MISSED_THRESHOLD_SECONDS`) and SKIPS missed fires.
   Operator gets a Notification card + WARNING event per missed schedule.

## What's new

### Concurrent same-shadow same-scroll execution

- Click `▶ Execute` 3 times on the same shadow+scroll pair → 3 separate
  subprocesses (up to the concurrency cap of 3, rest queue with cancel).
- A scheduled fire while a manual run of the same pair is in-flight → both
  run concurrently.
- `find_active_by_dedup_key` is retained as an informational helper; can be
  used by future UI to surface "N other runs in flight" without blocking.

**Failure analysis:** the only silent-data-loss vector is the
`shadow.execution_log` TOCTOU race (concurrent same-shadow writers can lose
each other's appends). This is a diagnostic field, not a correctness
primitive. All other axes (Activity records, tool output files, memory
buffer, bridge events) are either isolated, append-only, or honor
last-write-wins as operator intent.

### Missed-schedule treatment

When the dashboard restarts after downtime and finds schedules whose
`next_fire_at` is older than the threshold:

- **One-time schedules**: `status="completed"`, `missed=True`. The schedule
  never fires. Operator receives a Notification: "⏰ One-time schedule missed:
  <scroll_name>".
- **Recurring schedules**: `next_fire_at` advances to the next valid future
  slot (`scheduled_at + N*interval`, smallest N such that result > now).
  `missed_fires_count` increments. Schedule resumes normally from that slot.
  Operator receives a Notification: "⏰ Recurring schedule fire missed:
  <scroll_name>. Next fire at <time>".

Each missed schedule also produces a WARNING event under
`/insights → Events` (filterable with the System/Manual toggle).

### Schedule schema additions

Three new fields on the `Schedule` model — all default-safe, no migration
needed for existing schedule files:

- `missed: bool = False` — ONCE schedules: skipped due to staleness?
- `missed_fires_count: int = 0` — RECURRING: cumulative skipped fires
- `last_missed_at: Optional[datetime] = None` — wall-clock of most recent skip

## Backward compatibility

- `JobManager.start_job(... dedup_key="")` signature unchanged. Behavior
  change: dedup_key no longer suppresses spawn.
- `Job.dedup_key` unchanged.
- `find_active_by_dedup_key` unchanged.
- v0.8.6 `tests/test_v0_8_6_bundle.py::test_skip_if_previous_run_still_active`
  is marked `@pytest.mark.xfail(strict=True)` because it asserts the removed
  v0.8.6 behavior. If anyone re-introduces dedup, that test becomes XPASS
  (= failure) and protects the v0.8.7 contract.
- Existing schedule JSON files load cleanly into v0.8.7 — pydantic supplies
  defaults for the three new fields.

## Migration

`pip install --upgrade systemu`. No `.env` changes required.

If your operational pattern depends on the v0.8.6 dedup behavior (which is
unusual — v0.8.6 was the first version to have it, and it was a default
not a feature), you'll need to add your own dedup at the call site. The
`find_active_by_dedup_key` helper is still available.
