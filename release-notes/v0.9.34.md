# v0.9.34 — real MCP client + unified structured human-input (elicitation)

systemu is now a **real MCP client**, not the old httpx shim, and gains a
**unified structured human-input (elicitation) surface**. Both are built
**against the MCP protocol primitives** rather than re-inventing them, behind a
single gated chokepoint so every external tool call is risk-tiered, consent-gated,
and treated as untrusted by default. Delivered as one combined release (phases
P0–P4), each phase put through test-first implementation, a frontloaded hermetic
end-to-end run, and a multi-agent adversarial review (every finding faced an
independent refutation panel). The consolidated P1–P4 review came back
fix-then-ship; all confirmed findings are fixed or documented below.

## What changed

1. **Real MCP transport.** Connections now go through the official `mcp` SDK
   (stdio + streamable-HTTP/SSE), isolated under `systemu/runtime/mcp/sdk/` with
   an import guard so SDK types never leak into the rest of the runtime. The
   previous URL-only httpx path is gone.

2. **One gated chokepoint for every MCP tool call.** Closes the v0.9.32
   un-gated `mcp_call_tool` hole: a call passes through L1 availability → L2
   allowlist / env-grandfather → L3 a **risk-tiered scoped-trust gate** → L4 an
   **untrusted-output guard**. Risk tiers (read-only / action / destructive) are
   derived from the tool's MCP annotations, destructive-first and **fail-closed**
   (an unknown tier is treated as the most dangerous). The gate defaults to
   asking the operator, with per-server / per-session trust scopes so routine
   read-only calls don't cause approval fatigue.

3. **Rug-pull (definition-drift) defence.** When the operator approves a tool,
   its full definition (name + sanitised description + schema) is hash-pinned. On
   every use the live definition is re-hashed and compared; a tool whose
   definition has changed since approval is **auto-disabled pending
   re-approval** rather than executed silently.

4. **Tool-poisoning + untrusted content.** Discovered tool descriptions are
   sanitised (role tags / role prefixes stripped, size-capped, untrusted-labelled)
   before they ever reach the model, and **all** MCP tool output is wrapped with
   an untrusted-content banner. The WebMCP `untrustedContentHint` annotation is
   captured for parity with the emerging web standard.

5. **Tool-exposure budget.** A server that exposes dozens of tools no longer
   floods the agent's context: at most `mcp_max_exposed_tools` (15) are surfaced
   eagerly, with a lazy `mcp_search_tools` affordance to pull in the rest on
   demand.

6. **Unified structured human-input (elicitation).** A single surface for
   "the agent needs a value from you": form-mode prompts plus a JSON-Schema
   **missing-required gap detector** that asks for required parameters before a
   tool runs instead of failing. MCP servers that issue an elicitation request
   are routed through this same surface, with secrets split out so they never
   enter the form, the model context, or logs.

7. **Connect-as-capability at runtime.** A Shadow can request an MCP server as a
   capability (`REQUEST_HARNESS kind=mcp`); on approval the Governor connects,
   discovers, hash-pins, enables per the operator's per-tool opt-in, and mints a
   lease. Revoking the lease unregisters the live namespaced tools, and the flow
   survives suspend / resume and server re-attach.

8. **Remote hardening.** Remote transports enforce TLS (`mcp_require_tls`,
   default on) and run a DNS-resolving SSRF precheck with an operator host
   allowlist (`allowed_mcp_hosts`) before any socket is opened.

## Scaffolded, not enabled this release

- **Sampling** (a server asking the client to run an LLM completion) is
  implemented as a parent-side bridge through systemu's own `llm_router` — **no
  API key ever reaches the server or any subprocess** — and the same bridge
  backs the deferred `web_act` work. It is **not wired into the live connect
  path** in this release; the server-controlled cost levers are clamped and the
  consent rail is fail-closed, and it will be activated in a follow-up once the
  consent park/resume rail lands.
- **URL-mode OAuth** (operator consents to an authorize URL out-of-band; the
  secret never enters the client) is scaffolded and contract-complete but
  likewise **not on the live path** yet; it will be enabled together with
  authorize-URL scheme validation.

## Security review

Built against the MCP protocol's own primitives and reviewed adversarially. The
one blocker found — the rug-pull pin hashing the raw description while the
use-time re-check hashed the sanitised one, which would have auto-disabled every
connected tool on its first call — is fixed (both sides hash the sanitised form)
and guarded by a behavioural regression. Other fixed findings: credentialed
stdio servers now receive resolved env-var values, TLS enforcement is wired into
the runtime connect path, server-controlled sampling `maxTokens` / prompt size
are clamped, and dict-shaped annotations are no longer dropped.

## Known limits (by design, this release)

- Sampling and OAuth are present but **dormant** (see above); enabling them is a
  tracked follow-up, not a runtime toggle.
- The remote SSRF precheck validates the resolved IP at connect time but does not
  yet pin that IP into the SDK transport, so a DNS-rebinding adversary who
  controls DNS for an operator-allowlisted host is out of scope this release.
- **WebMCP** (in-browser tool registration) and **ARD** (`ai-catalog.json`
  discovery + cryptographic publisher identity) were analysed and fold in as
  design addenda: a WebMCP-first `web_act` pivot and an ARD discovery consumer
  (P5) are planned follow-ups. Neither replaces the MCP invocation layer this
  release delivers.
