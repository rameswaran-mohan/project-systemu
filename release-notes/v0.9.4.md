# v0.9.4 — Recipe Fast-Paths (Layer 5)

**Release date:** 2026-06-07
**Layer:** 5 of 7 in the assistant-foundation arc
**Headline:** systemu now ships loadable SKILL.md-format recipes. The LLM stops re-deriving plans every time and follows proven workflows. The bundled `burrito-delivery` recipe closes the v0.9.3 user-outcome gap (LLM kept faking completion instead of following the anti-bot retry hint).

## What changes

### SkillManifest format
- `SkillManifest` Pydantic model captures the on-disk SKILL.md shape: YAML frontmatter (name, description, version, platforms, prerequisites, metadata.systemu.tags + related_skills) + markdown body.
- Toolset-routing additions: `requires_toolsets` + `fallback_for_toolsets`.

### skill_loader module
- `parse_skill_md(path)` — YAML frontmatter + markdown body parser
- `discover_skills(root)` — walks one level deep
- `check_prerequisites(manifest)` — gates by `shutil.which`
- `check_toolsets(manifest, available)` — subset check
- `load_all_skills(bundled_dir, user_dir)` — user dir shadows bundled

### Bundled starter skills
3 recipes under `systemu/skills/`:
- **`burrito-delivery`** — primary acceptance test. Prescribes DuckDuckGo-first workflow, names Yelp/TripAdvisor/Zomato as no-go anti-bot sites, writes `burrito_raw.json` + `burrito_ranked.json`.
- `find-nearby` — generic location-aware ranking recipe.
- `summarize-page` — fetch + summarize a web page.

### LLM-side tools registered via v0.9.3 v2 registry
- `skill_list_skills` — lightweight dicts of all loadable skills (`toolset="skill"`)
- `skill_view_skill(name)` — full manifest including markdown body
- These are the FIRST batch of code-registered LLM tools beyond v0.9.3's file_tools.

### CLI
- `sharing_on skill list` — shows name, version, description, required toolsets
- `sharing_on skill view <name>` — full SKILL.md body + metadata

### Elder-intake prompt nudge (carryover from v0.9.1.1 + v0.9.3 backlog)
- `systemu/prompts/elder_intake.md` now explicitly tells the LLM: when `web_extract` returns `error_type="anti_bot_blocked"`, DO NOT claim completion — issue another `web_extract` call to a DuckDuckGo/Google search URL.

### 3 new Config knobs

| Env var | Default | Purpose |
|---|---|---|
| `SYSTEMU_SKILL_LOADER_ENABLED` | `true` | Master kill switch for skill_list/view + loader |
| `SYSTEMU_SKILLS_BUNDLED_DIR` | `<package>/systemu/skills` | Bundled SKILL.md location |
| `SYSTEMU_SKILLS_USER_DIR` | `""` | Optional user-installed skills directory |

## Migration

- No vault migration. Skills are loaded from disk on demand.
- Existing tools, scrolls, user profiles, audit log are unaffected.
- The `Skill` legacy model (v0.6.x) is still supported alongside the new SkillManifest — they target different concerns (legacy = runtime-evolved proficiency; new = on-disk recipe).

## Roadmap context

Layer 5 of 7. Next: Layer 6 (Goal-Level Orchestration, v0.9.5) — `delegate.spawn_subagent` first-class tool + MCP integration + mixture-of-agents pattern.

## Known limitations

- The bundled `burrito-delivery` skill is not auto-loaded into the LLM's context yet — the LLM must call `skill_view_skill(name="burrito-delivery")` to see the recipe. Auto-loading based on intent matching is L6/L7 territory.
- `skills_user_dir` empty by default — operators who want personal skills should set `SYSTEMU_SKILLS_USER_DIR=~/.systemu/skills` and drop SKILL.md files there.

## Live test result — honest assessment

Ran the burrito acceptance test against v0.9.4 (Bangalore profile, all gates enabled):

**What worked:**
- Scroll generation: verifier strings now mention "DuckDuckGo or Google search" — the elder_intake.md nudge (T6) directly influenced Tier-1 LLM output
- web_search invocations: 2 successes (capability ledger captured)
- Verifier mechanism: rejected 2 false completion claims with specific reasons
- v0.9.0–v0.9.3 infrastructure: all green

**What didn't work:**
- The LLM did not call `skill_list_skills` to discover the bundled `burrito-delivery` recipe. Root cause: v2-registered tools (file_tools, skill_tools, capability_tools, etc.) are not currently exposed to the LLM's prompt-time tool catalog in `shadow_runtime` (~line 1989). The LLM only sees v1 vault-registered tools.
- The LLM called `web_search`, got results in working memory, then claimed completion without persisting to disk. The verifier correctly rejected.
- Zero files produced in the output directory.

**Conclusion:** v0.9.4 ships SKILL.md infrastructure (parser, loader, bundled recipes, CLI access, schemas registered in v2 registry) but the LLM doesn't auto-consult it. Full value unlocks when **v0.9.5 (L6 Goal-Level Orchestration)** ships v2 tool visibility to the LLM catalog. That visibility wiring is now a hard prerequisite for L6.
