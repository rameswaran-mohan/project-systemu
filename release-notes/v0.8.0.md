# v0.8.0 — OperatorDecisionQueue (Pattern 1)

Major architectural change: replaces silent `SYSTEMU_HEADLESS=1` auto-pick
with a vault-backed decision queue surfaced through the dashboard. When a
dashboard button spawns a CLI subprocess that needs operator input, the
subprocess now persists the question to the queue and exits with a
"queued for operator" message instead of being silently overridden to
"Skip" (the safe default).

## What's new

- **`OperatorDecisionQueue`** at `systemu/approval/decision_queue.py` —
  vault-backed (file + SQLite backends) cache of operator decisions with
  dedup-key short-circuit for safe re-attempts.
- **`SYSTEMU_DECISION_QUEUE=true`** env var enables the queue-mode branch
  in `notify_user`. When set AND no TTY is available, prompts are persisted
  to the queue instead of auto-picked.
- **Dashboard `/insights` → Pending Actions tab** now renders pending
  decision cards with action buttons. Operators click an option to resolve;
  the card auto-refreshes and the original CLI command can be re-run to pick
  up the choice.
- **`sharing_on decisions list`** and **`sharing_on decisions resolve <id> --choice X`**
  CLI commands for terminal-based resolution.
- **Migration of 3 critical caller paths** to the queue:
  - `tool_forge.forge_tool` (dedup_key: `tool_forge:<tool_id>`)
  - `shadow_decision._prompt_create_new` (dedup_key: `shadow_decision:<activity_id>`)
  - `scrolls approve` and `tools forge` CLIs now catch `PendingOperatorDecision`
    and exit with code 75 (EX_TEMPFAIL) + clean message instead of a traceback.

## Deprecated

- **`SYSTEMU_HEADLESS=1`** emits a deprecation warning at use. It still
  works (auto-picks `actions[0]`) for backwards compatibility but will be
  removed in v0.9.0. Migrate to `SYSTEMU_DECISION_QUEUE=true` for the new
  queue-routed behaviour, or unset both env vars and run from a real TTY
  for direct interactive use.

## Migration

For existing installs that relied on `SYSTEMU_HEADLESS=1` to silently
auto-pick decisions in CI / background workers:
1. Add `SYSTEMU_DECISION_QUEUE=true` to `.env`.
2. Restart the daemon.
3. Operator decisions now appear on `/insights → Pending Actions` for
   click resolution, or via `sharing_on decisions list` / `resolve`.
4. Remove the `SYSTEMU_HEADLESS=1` line once you've verified the new
   surface works for your workflow.

If you actually wanted silent auto-pick (e.g. unattended CI that's OK
with the safe default), keep `SYSTEMU_HEADLESS=1` set and ignore the
deprecation warning — the legacy behaviour is preserved through v0.8.x.

## Fixed

- **Bug #25 (v0.7.3 UAT):** Dashboard button clicks no longer silently
  auto-pick `Skip` for the operator who actually clicked the button. The
  question now surfaces on `/insights → Pending Actions` for the operator
  to answer explicitly.
- **Bug #27 (v0.7.4 UAT):** `tools forge` no longer leaves an orphan
  PROPOSED Tool record when the operator gate aborts — the gate is now
  the queue post, which runs before the Tool save, so a queued decision
  implies no orphan record yet.

## Not yet migrated (deferred to v0.8.0.1+ / v0.8.1)

- The other ~19 `notify_user` callers (evolution prompts, tool sandbox
  destructive-action confirmations, dep approval flash cards, recovery
  panel prompts) still use the legacy path. They will be migrated to the
  queue incrementally; until they are, `SYSTEMU_HEADLESS=1` continues to
  work for them.
- **Pattern 3** (decouple `AUTO_FORGE_TOOLS` from the validator-forge
  bridge) is deferred to v0.8.1 — it depends on the queue from this release
  being in place first.

## New SQL migration

`alembic/versions/20260526_0010_operator_decisions.py` adds the
`operator_decisions` table when using the SQLite or Postgres backend.
Run `python scripts/upgrade_db.py` after upgrading.
