# v0.10.5 — Two security hardenings

Two fail-safe gaps closed, both from a deep review of the project's deferred-item backlog.

## Tighter forged-network refusal

When the daemon writes a tool for itself, a tool that reaches the **network** is refused outright (there's
no OS egress jail yet to bound it). That refusal keys off a static scan of the tool's code — and the scan
was missing a few standard networking modules whose call shape it couldn't see (`http.client`, `ftplib`,
and the telnet/POP/IMAP/NNTP clients). A forged tool reaching the network through one of those was merely
*sent for approval* instead of *refused*. Now those imports are recognized as network egress, so such a
tool is refused like any other. Precise, not blunt: `http.client` counts (it's an outbound client) while
`http.server` (inbound) does not.

## Confinement re-check fails closed

The file-reference resolver has a defense-in-depth re-check that a candidate file is inside a granted
folder. When no granted-roots authority was supplied, that re-check was **skipped** — which meant an
unconfined path could slip through. It now **fails closed**: no authority to confirm confinement ⇒ the
candidate is dropped. (No caller hits this path today, so there's no behavior change in practice — it
closes a latent hole.)

## Scope

- Both are fail-safe tightenings; neither loosens anything. Full suite: 5612 passed (no classification or
  gate regressions).
