# v0.9.54 ‚Äî secrets at rest via DPAPI (S5)

**OAuth tokens and the credential file-fallback were written as plaintext JSON,
protected only by `os.chmod(0o600)` ‚Äî a NO-OP on Windows, the primary platform.
They are now DPAPI-encrypted at rest, transparently and without data loss.**

This is safety-substrate milestone **S5** of the Open-World Go-Getter build (spec
`UNIFIED-v2` ¬ß7 / ¬ß11.3). It has no jail dependency, so it ships first of the
substrate ‚Äî the lowest-cost, highest-value hardening.

## The exposure (grounded, narrowed)

The credential store's **primary** path already encrypts on Windows: `keyring>=24`
resolves to `WinVaultKeyring` (Windows Credential Manager, DPAPI-backed). So the real
plaintext exposures were the two paths keyring does **not** cover:

- **`VaultTokenStore`** (`runtime/mcp/sdk/oauth.py`) ‚Äî OAuth access/refresh tokens at
  `<vault>/connections/mcp_oauth/<server_id>.json`, written as plaintext JSON.
- **`CredentialStore` file fallback** (`runtime/credentials/store.py`) ‚Äî
  `.credentials.json`, used when keyring is unavailable, also plaintext.

Both relied on `os.chmod(0o600)`, which silently does nothing on Windows ‚Üí real
bearer secrets sat readable on disk.

## The fix

- **`runtime/credentials/at_rest.py`** ‚Äî a leaf helper (`protect_json` /
  `unprotect_json` / `is_encrypted_at_rest`). On Windows it wraps the JSON in a DPAPI
  envelope via `win32crypt.CryptProtectData` (USER-scope, UI-forbidden) ‚Äî encryption
  bound to the daemon's identity, so a different token (e.g. a future AppContainer
  child, S2) can't decrypt it. On POSIX / no-DPAPI it returns plaintext, **unchanged**
  (there the `0o600` perm is a real control) ‚Äî zero behavior change off-Windows.
- **Transparent + migrate-on-read.** `unprotect_json` reads a legacy plaintext file
  as-is; the next `save` re-writes it as an envelope. No token is ever lost during the
  upgrade ‚Äî a v0.9.53 install's existing tokens keep working and quietly encrypt on
  first re-save.
- **Fail-safe.** A corrupt/undecryptable payload raises; both stores already wrap load
  in try/except and return `{}` (a re-prompt), so a bad file can never crash boot.
- Both `VaultTokenStore` and the `CredentialStore` fallback now route their file I/O
  through the helper; the `chmod(0o600)` stays as the POSIX-secondary control.

No caller reads these files directly ‚Äî the token JSON is read only via
`VaultTokenStore.load()` and `.credentials.json` only via `CredentialStore` ‚Äî so the
on-disk format change is fully contained.

## Tests

- `tests/test_secrets_at_rest.py` (8) ‚Äî envelope round-trip; ciphertext hides the
  secret + carries the marker (Windows); legacy-plaintext migrate-on-read; the token
  store and the credential fallback both encrypt on disk, migrate a pre-existing
  plaintext file, and a corrupt store returns `{}` without raising. Windows-only
  ciphertext assertions are guarded by `is_encrypted_at_rest()` so the suite is correct
  on POSIX CI.
- Full suite green (4241 passed, 22 skipped, 1 xfailed ‚Äî baseline 4233 + the 8 new tests).

## Scope

Captured Playwright `storage_state` (the third ¬ß11.3 secret) is not built yet (a later
web-actuation track); it will route through this same helper when it lands. The child
being *unable* to reach or decrypt the store (the AppContainer/WFP side of ¬ß11.3) is
S2, not S5.
