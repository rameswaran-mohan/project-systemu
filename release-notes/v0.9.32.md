# v0.9.32 — reliability: clean recordings, operator stop control, gated shell commands

Three reliability/safety fixes. (A fourth — making `web_act` complete browser
actions end-to-end — was scoped in but pulled: the right fix is architectural,
not a patch, so it ships on its own. See "Deferred" below.)

## What changed

1. **The recorder no longer captures its own "Stop & Analyze" click.** When you
   recorded an interaction with systemu's own dashboard, the click that *ended*
   the recording leaked in as a captured step. Root cause is a cross-process
   race: the dashboard is a browser-served page and the recorder is a separate
   subprocess, so the OS-level input hook captures the click before the stop
   signal can tear the recorder down. The fix is **layered filtering**: the
   recorder learns the dashboard's origin and (1) drops web-extension events from
   that origin, (2) strips interactions whose label is a recording control
   (`Stop & Analyze` / `Cancel & Trash`) or whose window is the dashboard, and
   (3) trims the trailing stop-click within a tight window before the stop
   timestamp. Together these remove every representation of the control click
   while leaving your real final actions intact.

2. **You can stop a running task from the dashboard.** Both **shadow executions**
   and **chat submissions** now have a **Stop** control (in the running-tasks list
   and on the running chat entry). Cancellation is **cooperative** — the run
   stops cleanly at its next loop boundary rather than being force-killed — and
   the task is marked with a new terminal **`CANCELLED`** status (distinct from a
   genuine failure, so it skips the failure post-mortem and is never retried).
   Under the hood: a public `Supervisor.request_cancel`, a cancel token for the
   chat/quick lane, and a small chat-task registry.

3. **Harmful shell commands now require explicit per-command operator approval.**
   Previously the two execution lanes simply **auto-denied** anything that looked
   destructive (chat) or denied it in headless mode (shadow) — useful work just
   failed, with no way to approve it. Now a **single approval gate** at the tool
   chokepoint covers both `run_command` and `run_cli_command` on both lanes:
   provably read-only commands run freely; anything else surfaces an operator
   card with **Deny / Approve once / Always allow** (the safe default is **Deny**).
   "Always allow" remembers the exact command signature across restarts; the
   chat lane **blocks and asks** (you approve in the dashboard) while the shadow
   lane parks and resumes. The gate is a **floor** gate type — even "Bypass" mode
   can't auto-allow it — and it is **fail-closed**: if no operator is reachable,
   or the request times out, the command is **denied**, never run.

## Deferred

- **`web_act` end-to-end.** The planned "minimal fix" assumed `web_act` was a
  trusted, non-forged tool that could safely receive the LLM key in its
  sandboxed subprocess. It isn't — like nearly every seed tool it is marked
  `forged_by_systemu`, so there is no clean per-tool trust signal, and handing
  the key to subprocesses is the wrong direction anyway. The correct fix routes
  the planning LLM **through the parent process** (which already holds the key)
  so **no credential ever enters a tool subprocess** — a focused follow-up
  release. `web_act` still *perceives* pages correctly (the v0.9.31 `aria_snapshot`
  fix); only its end-to-end planning is deferred.

## Known limits (by design, this release)

- Cancellation is **per-process**: in multi-process (docker) deployments the
  dashboard can only stop tasks running in its own process.
- The command gate covers the shell tools via the tool sandbox; **MCP-routed**
  tools bypass the sandbox and are not covered by this gate.
- Cancellation takes effect at the **next loop boundary**, not mid-LLM-call.
