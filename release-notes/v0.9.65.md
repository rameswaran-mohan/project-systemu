# v0.9.65 — R-A12a: External-Event Wait primitive (durable retry timers)

**A retry now survives a daemon restart.** Until now, when the supervisor retried a
transiently-failed activity, it armed an in-process `threading.Timer` — and the activity was
marked failed *before* the timer was set. If the daemon restarted during the (5–10s) back-off
window, the scheduled resubmit was lost and the activity was **silently never retried**. This
release makes that wait durable: the retry is persisted on the run's `ExecutionSnapshot` and
fired by a background reconciler, so it survives a restart.

This is the External-Event Wait primitive half of R-A12 (the pull-wiring half, R-A12b, follows).

## What ships

- **`ExecutionSnapshot.pending_waits`** (schema **v6**, additive migration `_migrate_5_to_6`,
  mirroring S4's `external_evidence`). A durable-wait record carries `{execution_id, wait_kind,
  fire_at, attempt, max_attempts, dispatched, activity_id, shadow_id, …}`.
- **The supervisor arms a durable wait, not a `threading.Timer`.** On a retryable failure it
  writes a `retry` pending-wait to the run's snapshot (minting a minimal snapshot if none
  exists yet). No in-process timer to lose.
- **`external_wait_reconciler`** (a new APScheduler job, 15s, templated on the OAuth reconciler)
  finds due, undispatched waits, and — only for a **parked** run (not currently executing) that
  isn't terminal or cancelled — stamps the wait dispatched (before submitting, for at-most-once)
  and resubmits the retry at `attempt + 1`, so the chain terminates at `MAX_RETRIES` instead of
  looping. The resubmit **resumes** from the snapshot (skipping already-credited external
  effects / fail-closed-parking them) — safer than the old fresh restart. Waits are dropped when
  the run finished by any path, was cancelled, exhausted its attempts, or went stale (24h).
- **Cancellation expires waits (IMPL-11).** Cancelling a run clears its pending waits; a wait
  that races the cancel is a no-op — a cancelled run never resubmits.

## CONC-MAP guardrail — validated end-to-end

The reconciler is the **first new writer** of `ExecutionSnapshot` since the CONC-MAP
writer-ownership gate shipped (v0.9.64). Adding its `write_snapshot` call made
`test_writer_set_matches_conc_map` **fail** until `scheduler/jobs.py` was added to the allowlist
with the DEC-10 review — exactly the concurrency-review checkpoint the guardrail was built to
force. The reconciler respects the per-execution_id parked-run invariant it documents.

## Compatibility & scope

- **Additive** snapshot migration (v5→v6); existing runs unaffected; the retry contract matches
  the old timer's (`retry_count+1`, terminating at `MAX_RETRIES`).
- Built test-first across six tasks, then an adversarial concurrency review (skeptic-verified)
  that caught **three real defects** the green suite missed — an infinite retry loop (the
  reconciler didn't advance the attempt), spurious re-execution of a run that finished via
  another path, and a synthetic-key collision that silently dropped a retry — each fixed, with
  the review also independently confirming the resume-based retry is *safer* than the old fresh
  restart for external-effect double-submit.
- **Deferred (R-A12b):** the reverse-harness pull wiring — per-kind requirement routing, the
  bundled upfront scope card, the plan-delta gate, resume-wiring the elicitation ask path, the
  committed-effects ledger, and the avoidable-ask audit.
