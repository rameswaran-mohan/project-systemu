# v0.9.63 — R-P1: Reach v2 — resolve parked decisions from the phone

**A parked decision can now be answered from your phone.** When a run stops for an operator
decision — a tool/command approval or a short question — systemu already pushed a "needs you"
notification, but the only way to *answer* was the dashboard. This release lets you resolve
the **remotely-safe** decisions straight from allowlisted Telegram — tap an inline button or
send `/answer <tag> <choice>` — routed through the **exact same** resolution + resume rail the
dashboard uses. Durable, idempotent, audited.

Crucially, this is **not** "approve anything from your phone." A hard, fail-closed classifier
keeps money moves, irreversible/destructive actions, governor-denied actions, typed-confirm
prompts, posture changes, and anything unrecognized **dashboard-only** — a phone tap can never
resolve them.

## What ships

- **Remote resolution, same rail.** Each parked decision gets a short deterministic tag
  (`[k3f7qa]`) on its push. Tapping a button (or `/answer k3f7qa a1`) calls the identical
  `OperatorDecisionQueue.resolve` the dashboard/CLI call — the existing resume machinery then
  re-dispatches the parked run. **No second execution path.** Option keys map positionally to
  the decision's own choices, re-derived from the persisted record (never trusted from the
  wire).
- **SEC-1 — the fail-closed floor (the load-bearing control).** Every decision is stamped, at
  creation, with a persisted `resolution_class ∈ {remotely_resolvable, floor}`. Only
  **positively-recognized safe shapes** are remotely resolvable: a `command`/`tool` gate that
  *proves* it is safe (an affirmative verdict, a clean `effect_tags` list — no money/
  irreversible — and not destructive/typed-confirm), or a single-field enum-≤4 / free-text
  question. **Everything else floors** — DENY, money/irreversible, destructive, typed-confirm,
  posture/credential/forge/scroll/dep/MCP gates, multi-field forms, and anything whose safety
  evidence is *absent*. The server refuses on the **persisted bit** (default-refuse when it's
  missing), never on the outbound hint — so a forged callback cannot bypass it.
- **Redaction at the wire.** A single MASK chokepoint at the gateway redacts secrets
  (Authorization/Cookie, bearer/`sk-`/AWS/JWT/token shapes) from every outbound push and button
  label.
- **Abuse controls.** Allowlisted senders only (re-checked server-side); per-sender rate limit;
  idempotent — a resolved/expired decision answers "already resolved", and old buttons are safe
  forever. Every remote resolution writes an audit row `{ts, channel, sender, decision, tag,
  choice, outcome}`.
- **Config:** `SHARING_ON_MESSAGING_DECISION_RESOLUTION` (default on when Telegram is
  configured), `SHARING_ON_MESSAGING_PUSH_DETAIL` (`summary`/`full`), `SHARING_ON_DASHBOARD_BASE_URL`.

## Compatibility & scope

- **Additive.** No change to the dashboard resolution path or the resume rail; the
  `resolution_class` stamp rides in the already-extensible decision `context` (no migration).
  With Telegram unconfigured, nothing changes.
- **Adversarially hardened.** Built test-first across six focused tasks, then reviewed by two
  adversarial-security passes (skeptic-verified). The second pass caught a serious class of
  defect the green suite missed — because the real gate-creation paths don't persist
  `verdict`/`effect_tags`, the exclusion floors were *structurally dead*, so a governor-**denied**
  or **money-move** tool gate could have been tap-approved from a phone; a remotely-approved
  scroll/dep/forge gate would have been marked done without executing; and a tag collision could
  resolve the wrong decision. All were fixed **fail-closed** (positive-evidence-required
  classification + a narrowed remote gate set + consistent tag disambiguation), with real-path
  integration tests that build actual gates and assert they floor.
- **Deferred:** remote resolution of MCP/credential gates (need their own remote execute path);
  the live-phone tryout is operator-verifiable.
