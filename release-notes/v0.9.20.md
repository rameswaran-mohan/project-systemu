# v0.9.20 — First contact: set up right, learn it in minutes

A learning-curve release, driven by four operator field reports. The goal:
a brand-new, non-technical operator goes from install → configured →
guided → first successful task without reading anything external — and
two daily-feel bugs die.

## The expand arrow works now

The live panes' expand arrows looked broken — they weren't: the panes
re-rendered every half second and rebuilt every row collapsed, destroying
whatever you'd opened. Expansion state now survives repaints (it follows
the same event across rebuilds), and an idle pane stops re-rendering
entirely — open a row's details and they stay open.

## Chat: type first, scroll never

With any history, the chat composer sat below up to 20 cards. The composer
(mode control + textarea + Run) now renders first and autofocuses; the
Ctrl+Enter shortcut is visible instead of secret; history follows below,
newest first.

## Setup is enforced — and mostly automatic

* **At install:** the installer no longer shrugs at a blank API key
  (it re-asks, with `--no-key` as the explicit air-gapped escape), and now
  asks the two settings that silently shaped everything: **model preset**
  (balanced / quality / budget / decide-later) and **output folder** —
  both written to .env, the folder created on the spot.
* **At every boot:** a first-run engine checks readiness (key, models,
  output folder, tool catalog, profile, tour) and auto-fixes what's safe
  to fix silently — directories only, never keys, never model choices.
* **At first launch:** fresh installs are funneled to /welcome from
  EVERY screen until the key exists and the profile is saved. The key is
  still never typed into the browser — add it to .env and click
  **Re-check** (no restart needed). Existing installs that previously
  chose "later" are never retro-nagged.

## A guided tour, on your first start

Finishing the wizard flows straight into a six-step tour — Home, Chat,
Work, Inbox, Build, Settings — each explained in plain language on the
real screen, not screenshots. Leaving it unfinished keeps a "Take the
tour" pill in the header until it's done; "End tour" counts as done (a
mandatory tutorial should never hold you hostage); Settings → Help
replays it any time.

## Why your chat tasks were failing — fixed

A live-install root-cause analysis (your vault, your daemon log) found a
chain of four faults, all fixed here:

* **The quick lane couldn't use most web tools.** It built its tool
  sandbox without the dependency-approval wiring the full runtime has, so
  every tool that declares pip packages (fetch_json, web_read,
  web_extract…) was blocked "pending approval" — even though you had
  approved and installed those packages long ago. The default chat lane
  now uses exactly the runtime's installer policy.
* **Installed packages counted as missing.** The installer's
  "is this satisfied?" check was a per-process cache, not a real check —
  every daemon restart re-gated packages that were already in
  site-packages. It now checks the actual environment first.
* **Failures said `None`.** When a tool failed without a message ("tool
  'run_command' failed 3 times in a row: None"), nothing actionable
  surfaced. Failures now fall back to the command's stderr tail or exit
  code.
* **The default fast-tier model no longer exists.** OpenRouter returns
  404 for `z-ai/glm-4.5-air:free`, silently killing web_extract and every
  other tier-3 consumer on default installs. All defaults now point at a
  model proven live in the field; pick any preset in Settings as before.

## It explains itself

Every spine page now carries a one-line plain-English sublabel (Inbox:
"Approvals — questions waiting for your yes or no"); the Work empty state
links to the action instead of describing absence; the header's ＋New,
Status, and Needs-you controls explain themselves on hover.

## Verification
