# v0.9.39 — run-tree-scoped request cap and outcome reconciliation

A runtime fix completing the harness request-outcome instrumentation. Bugs 9–14
made a *single* grant classify correctly; this release makes the per-run request
**cap** and the terminal outcome **reconciliation** span the whole **run-tree**
— the suspend→approve→resume chain plus the spawned sub-agent children — instead
of a single execution.

## Fixed — the request cap + reconciliation were per-execution, now run-tree-scoped

A logical run is many executions: an escalate→suspend→approve→resume lifecycle
mints a fresh execution id on each hop, and a granted SUB-AGENT spawns child
runtimes that each run their own execution with their own ledger. Both the cap
counter and the terminal reconciliation were keyed to one execution id, so on any
multi-execution run:

- **The cap never bounded the run.** Each resumed/child execution restarted the
  request counter at 0, so a tree of executions could issue far more capability
  requests than `max_requests_per_run` — a cost/safety runaway (and the reliable
  backstop the v0.9.38 "do-not-re-delegate" framing depends on).
- **Reconciliation missed most grants.** The terminal finalize reconciled only
  the current execution (plus one immediate resume predecessor), so grants in
  earlier resume hops or in any child execution never produced a
  `request-outcome` event — the pull-decision taxonomy was blind to them.

### One run-tree identity, used by both

- **`root_execution_id`** — a stable run-tree id on the execution. A fresh
  top-level run is its own root; a resume inherits it from the snapshot (now
  persisted in `ExecutionSnapshot`); a sub-agent child inherits it from its
  parent (threaded through `SubagentFleet`).
- **Cap → run-tree-wide.** A persistent per-root sidecar holds the shared request
  counter; every capability request bumps it and the arbiter's cap compares
  against the tree-wide total, so a capped tree terminates instead of issuing
  unbounded grants. Falls back to the per-execution count when no vault is
  available, so single-execution behaviour is unchanged.
- **Reconciliation → run-tree-wide.** The same sidecar indexes every execution
  in the tree, so the single top-level terminal reconciles **all** their ledgers.
  The per-request_id collapse from v0.9.37 (Bug 11) keeps it idempotent, so a
  child that already self-reconciled is never double-counted.

No behaviour change for an autonomous, single-execution run with no children.
