# v0.10.10 — Self-diagnosis (`doctor` + `/health`) and a cross-OS capability profile (R-UX1)

When "nothing seems to be happening," you can now ask the system *why* — and it answers concretely.

## What ships

- **`sharing-on doctor`** — a one-command self-diagnosis. It checks the things that silently stop work:
  is the daemon reachable, is an LLM provider configured and reachable, is the OS keyring usable, are
  versions sane, was there a recent error — and prints the machine's capability profile. It **exits
  non-zero** when there's a real blocker (no/killed provider, a locked keyring), so it's usable in a
  script. (Bare `doctor` is the new whole-system check; `doctor <id>` keeps its existing per-scope
  recovery behavior.)
- **A `/health` page** in the dashboard rendering the same profile + a live status chip.
- **One deterministic capability profile** (`platform_profile`) that every OS-divergence renders from —
  the same schema on Windows / macOS / Linux, so behavior is never silently OS-conditional. It reports
  the keyring backend, whether capture is available, and honestly marks host-only capabilities
  (screen capture, COM/UIA, global hotkeys, host browser) as *"available via the Host Companion"* inside
  a container rather than pretending they're present. It also states plainly that the OS network-egress
  jail is **absent** (which is exactly why forged network tools are refused outright).

## Secrets: honest keyring reporting

Secret storage already preferred the OS keyring, but on a headless box `keyring` hands back a dummy
backend that only fails at write time — so the system could *report* a keyring it couldn't actually use.
That's fixed: a real backend is now detected properly, and when there isn't one, the system says so
**loudly** and falls back to its flagged at-rest file (DPAPI-encrypted on Windows, 0600 on POSIX). A
secret is never written in the clear when a real keyring is available, and a keyring failure falls back
with a warning that never contains the secret. Full suite: 5661 passed.
