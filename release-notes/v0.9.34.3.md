# v0.9.34.3 — forged-tool dry-run error is surfaced to the agent

A one-change patch on top of v0.9.34.2. It hands the agent the reason its forged
tool failed, so it can self-correct.

## What changed

When the Governor grants a `TOOL` request, the runtime forges the tool and runs
an automatic dry-run before exposing it. If the dry-run **failed**, the agent
previously received a generic, detail-free message — *"…was forged but is pending
enablement/dry-run; it is not callable this run. Use an existing tool or FAIL."* —
and was **never told why**. The error was already available (`deploy_forged_tool`
returns `{"deployed": False, "reason": <the dry-run error>}`) but the grant
handler discarded it.

An RCA of a real trial showed the cost: a model forged a tool whose declared
`parameters_schema` required `output_path` while its implementation was
`run(input_path, level=9)` (no `output_path`). The dry-run correctly raised
`TypeError: run() got an unexpected keyword argument 'output_path'` and the tool
was withheld — correct behaviour — but because the agent never saw the error, it
re-forged the *same* broken schema and ultimately failed. With the error
surfaced, it could have fixed the schema and succeeded.

The not-callable observation now includes the dry-run error plus a generic
tool-authoring repair hint (the implementation's parameters must match the
declared `parameters_schema` — same names; the schema must not require a
parameter the function does not accept). This turns the `TOOL` family from a
one-shot schema-coherence test into a fair capability probe (self-correction).

Single-file change (`systemu/runtime/shadow_runtime.py`, the TOOL grant branch)
plus a regression test. The "you may call it now" success branch is untouched; no
schema or API changes.
