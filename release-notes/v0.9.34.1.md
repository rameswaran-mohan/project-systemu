# v0.9.34.1 — MCP reachability + capture scope, plus QA bug fixes

A patch release on top of v0.9.34 (real MCP client + elicitation). It makes the
shipped MCP capability actually reachable by agents, adds an opt-in narrow/broad
capture scope, and fixes four QA-reported bugs (two of which were live in
v0.9.34). Everything is opt-in or a fix — no change to default behaviour.

## New

1. **MCP is now agent-reachable.** v0.9.34 shipped the MCP connect-as-capability
   backend (gated, arbitrated), but the agent prompt never listed `mcp` as a
   `REQUEST_HARNESS` kind — so a Shadow never asked for one and the backend was
   effectively unreachable. The agent prompt now names `mcp` and carries a
   connect-spec affordance (server_id, transport ∈ stdio/http/sse, command/args/
   env_keys, url, label, tool_filter), states that a new server needs operator
   approval, and keeps credentials as env-var **names** (never values). The
   compute request's `extra_iterations`/`extra_think` fields are now documented
   too.

2. **Capture scope (narrow vs broad).** A record session can opt into capturing
   only ONE target app/window/origin (`narrow`) instead of system-wide
   everything (`broad`, the unchanged default). Surfaced as `record --scope/--app`
   and a record-dialog toggle in the dashboard. A `CaptureScope` filter is
   installed on every collector and consulted before each event is stored:
   window/process collectors narrow by app/process name (a Windows-style
   `chrome.exe` target also matches a friendly `Google Chrome` name); the browser
   collector keeps only the allow-listed origin. The interaction stream (clicks/
   keystrokes — including the UI-introspector's element-enriched events) is kept
   under narrow Phase 1 because those events can't yet be attributed to an app;
   narrow still strips the obvious cross-app window/process/origin noise. Scope is
   recorded in `session.json`.

## Fixed

3. **MCP overflow discovery was dead** (QA Bug 6; live in v0.9.34). When a
   server exposed more tools than the per-run budget, the runtime advertised an
   `mcp_search_tools` affordance to reach the rest — but no handler was
   registered, so calling it returned "tool not found". A handler is now
   registered; it searches the full enabled-tool set by keyword and returns the
   matches (callable by their namespaced name).

4. **In-run MCP tool calls failed** (QA Bug 8; live in v0.9.34). The connect
   path never persisted the transport spec, so the stateless call path fell back
   to `http://<server_id>` and could not reach a stdio server (`unhandled errors
   in a TaskGroup`). The full reconnect recipe is now persisted — secret-free
   (env-var **names** only; values are re-resolved from the parent env at call
   time, never written to the on-disk connections store).

5. **Granted COMPUTE was a no-op** (QA Bug 7). A compute grant defaulted to +0
   iterations when the request omitted an amount, so a self-extension request
   changed nothing. An absent/zero request now defaults to a sensible non-zero,
   ceiling-bounded bump.

6. **"Needs you" rail crowded out the live feed.** The dashboard inbox section
   grew unbounded and pushed the live-events pane below the fold; it now has a
   fixed-height internal scroller (mirroring the live pane).

7. **Test-suite collection fix** (QA Bug T): a stray `tests.` import prefix in the
   MCP runtime-connect tests broke collection in the standard layout.

## Known limits (by design, this release)

- **Capture-scope Phase 1** narrows window/process/browser-origin events; the
  global input-hook is not yet narrowed per-app (its events carry no
  foreground-app context). The interaction stream is kept rather than dropped, so
  narrow recordings stay useful; Phase 2 will correlate input events to the
  foreground window. Title-only narrowing requires an app target (the title
  refines within an app).
- MCP sampling and OAuth remain scaffolded but dormant (unchanged from v0.9.34).
