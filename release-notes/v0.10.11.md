# v0.10.11 — Your table is now yours to curate (R-B2 / T2a)

The **On the table** page has shown you everything systemu can see it has — services, tools,
files, keys — since it landed. It was read-only. Now you can curate it.

## What ships

- **Remove** anything off your table with the × on its card. Removing is honest about what it
  does: for a real thing (a credential, a connected server, a tool) you get a plain note that
  **the actual object is still active** and where to manage it — the table is your *view*, not the
  store, so removing a card never silently revokes a key or disconnects a server.
- **Undo** — every removal leaves a 10-second "Removed X · **Undo**" banner. One click brings it
  straight back.
- **Pin** what matters with the ★. Pinned items sort to the top of their zone and stay pinned
  across restarts.
- **Search** filters the whole board as you type (name, detail, or kind); **collapse** any zone
  you don't care about right now.

## Under the hood: single-writer discipline preserved

The read-only page had exactly one writer for the table's durable store (the 60-second
reconciler). Adding curation could have made the UI a second concurrent writer of the same file —
so instead, **pins and removals are written to their own UI-owned sidecars** (`pins.json` /
`tombstones.json`) that the reconciler only *reads*. The projection snapshot (`items.json`) still
has a single writer. No file has two writers, so a pin can never race the reconciler. The
concurrency map and its CI guardrail were updated to match. Pure curation logic is unit-tested
(remove/undo round-trip, pin survives reconcile, search, still-active honesty); the board
rendering is operator-verifiable.

## Scope

This is the *curate-what's-there* half of table curation. The **"+ Put on the table"** add
palette (declare a service, name a credential, forge a tool — deep-linking into the existing
flows) and a "restore removed" view are the next slice (T2b).
