# v0.9.52 — resume a chat task parked on a command-approval gate

**A chat task that ran a destructive shell command (`run_command`) and parked on the
operator approval would hang forever after you resolved it. It now resumes.**

## The bug

A chat task runs through the `shadow_runtime` ReAct loop. When it calls a destructive,
non-allowlisted command, `tool_sandbox._maybe_gate_command` posts an approval decision
(`kind="gate"`, `gate_type="command"`) and raises `PendingOperatorDecision`; the task
parks. But:

- the command-gate decision carried **no resume coordinates** (`execution_id`,
  `chat_submission_id` were absent), and
- **both** resume dispatchers — the `resume_on_decision` EventBus subscriber and the
  cross-process poll in `jobs.py` — bail on `kind != "structured_question"`.

So when you resolved the approval, **nothing resumed the parked task** — it stayed
`pending_decision` until the daemon restarted. (Confirmed against the live vault: the
resolved decision had `kind: gate`, `gate_type: command`, all resume coords `None`,
`resume_dispatched: None`.) Pre-existing; unrelated to the v0.9.51 param-synth work.

## The fix (park + resume)

Park-and-resume was chosen over a block-poll so it frees the execution thread and
works for the workflow lane too, matching how `structured_question` questions already
resume.

1. **Resume coords** — a new `execution_id` context-var (sibling of the existing
   `chat_submission_ctx`); `_maybe_gate_command` stamps `execution_id` +
   `chat_submission_id` onto the command-gate decision so it's addressable.
2. **Resume dispatch** — `resume_on_decision` and the `jobs.py` poll now handle
   `kind="gate"/gate_type="command"`: they derive `activity`/`shadow` from the run's
   execution snapshot and re-submit with `resume_from_execution_id` on **Approve**, or
   mark the activity `FAILED` on **Deny** (no re-ask loop).
3. **Honor on re-run** — a single-use resume-approval in `CommandApprovalStore`
   (`mark_resume_approved` / `consume_resume_approved`) that `_maybe_gate_command`
   consumes, so the resumed run runs the command exactly once and re-asks for any
   *later* command.

## Tests

- `tests/test_command_gate_resume.py` — the one-shot store, command-gate Approve
  (resume + honor), Deny (finalize, no loop), a `structured_question` regression, and
  a non-command gate guard.
- Full suite green (4233 passed). The **end-to-end resume** (a live task continuing
  after approval) is operator-verifiable only — the gate covers the units.

## Not included

The nicegui `Type is not JSON serializable: function` (F5) crash is cosmetic (you can
still resolve the decision) and independent of this hang; deferred.
