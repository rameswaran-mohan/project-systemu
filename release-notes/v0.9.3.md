# v0.9.3 — Capability Ledger + Tool Registry v2 (Layer 3)

**Release date:** 2026-06-07
**Layer:** 3 of 7 in the assistant-foundation arc
**Headline:** systemu now tracks its own capabilities AND ships a Hermes-style code-side tool registry — while preserving the auto-forged vault tool path.

## What changes

### Capability Ledger
- **`Capability` model** + sidecar JSON at `vault/capabilities/_usage.json` tracking name, kind (tool|skill), registered_at, last_used_at, invocations, successes, failures, last_error.
- **`capability_ledger` module**: `register/record_invocation/get_capability/list_capabilities/get_stats` — atomic writes via tempfile + rename, in-process lock.
- **`tool_sandbox` hook**: every tool call (v1 OR v2) records its outcome into the ledger. Gated by `SYSTEMU_CAPABILITY_TRACK_OUTCOMES=true` (default).
- **LLM tools**: `capability_list_my_capabilities`, `capability_get_stats`, `capability_last_used` (toolset="capability", read-only).
- **CLI**: `sharing_on capability list/show/stats`.

### Tool Registry v2 (Hermes-style)
- **`tool_registry_v2`**: code-side registry with module-level `register()` decorator-compatible API, `get/list/list_by_toolset`, AST-scan discovery from `systemu/runtime/tools/*.py`, **`check_fn` TTL cache** (default 30s, configurable via `SYSTEMU_CHECK_FN_CACHE_TTL_SECONDS`).
- **`Tool.check_fn_name: Optional[str]`** — slot for vault-stored v1 tools to opt into the same availability gating (lookup at runtime by name).
- **Per-context whitelists** in registry: `verifier_fork` (read-only), `curator` (skill/memory), `fact_extractor` (write_user_fact), `delegate_child` (empty — parent_whitelist minus delegate, runtime-composed).
- **Module-level singleton**: `from systemu.runtime.tool_registry_v2 import registry`.

### File toolset (first code-registered batch)
- `systemu/runtime/tools/file_tools.py` registers `read_file`, `write_file`, `search_files` via `registry.register(...)` at module load. All in `toolset="file"`, `max_result_size_chars=100_000`.

### Tool hygiene peer modules
- `systemu/runtime/tool_hygiene/`: `path_security.safe_resolve` (rejects traversal + escapes), `url_safety.is_url_safe` (allowlist scheme, blocks loopback + RFC1918), `output_limits.cap_output` + `is_likely_binary`, `ansi_strip.strip_ansi`.

### Integration
- `ToolSandbox.execute()` consults v2 registry FIRST, falls back to v1 (vault-based) for auto-forged tools — both paths coexist, no v1 break.
- `ShadowRuntime.__init__` discovers v2 tools at startup via `_discover_v2_tools()` (idempotent).
- `_resolve_tool_whitelist(context)` returns empty set for unknown contexts (defensive — runtime treats empty as "allow nothing").

## Operator-facing knobs

| Env var | Default | Purpose |
|---|---|---|
| `SYSTEMU_CAPABILITY_LEDGER_ENABLED` | `true` | Master kill switch for the ledger read paths |
| `SYSTEMU_CAPABILITY_TRACK_OUTCOMES` | `true` | Record invocations from tool_sandbox |
| `SYSTEMU_CHECK_FN_CACHE_TTL_SECONDS` | `30` | TTL for cached `check_fn` results in registry v2 |
| `SYSTEMU_TOOL_OUTPUT_MAX_CHARS_DEFAULT` | `100000` | Default output cap (used by tool_hygiene.output_limits) |

## Migration

- No vault migration required — capability sidecar created on first write.
- Auto-forged vault tools (v1) continue to work unchanged.
- Existing v0.9.0/v0.9.1/v0.9.2 features untouched.
