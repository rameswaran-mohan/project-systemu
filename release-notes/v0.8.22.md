# v0.8.22 — make existing fixes reach users

A delivery-layer release. v0.8.21 shipped real fixes for the bakery-RCA (the `web_extract` toolkit
and the stuck-loop guard), but operators who tried the same prompt through chat on an upgraded
install still hit the same loop. Three structural defects in the layers *between* a fix and the
user were swallowing the improvements — and v0.8.21's own release note flagged the largest one:
*"a package upgrade alone does not rewrite already-seeded vault tools."* v0.8.22 closes those three
gaps. No new tools, no new user-facing surfaces — just the plumbing that makes the toolkit reach
the agent and the agent's questions reach the operator.

## What went wrong (systems RCA)

Re-running the bakery prompt on a `pip install --upgrade systemu==0.8.21` install reproduced the
failure. Three independent defects:

- **D1 — vault upgrade-rot.** The two new v0.8.21 tools (`web_extract`, `extract_records`) live in
  the *package*, but a vault initialized on v0.8.19 still holds the v0.8.19 tool set. A `pip
  install -U` upgrade never rewrites the vault, so the agent on an upgraded install never sees the
  new toolkit at all. v0.8.21's own release note acknowledged this — but left the fix for later.
- **D2 — extractor invents tool names.** The activity extractor was given the *list* of existing
  tools but only their bare names; the LLM, unable to judge fit from a name alone, invented
  plausible-but-fictional names (e.g. `search_places_nearby`) instead of binding to `web_extract`.
- **D3 — pending decisions look like failures.** When a chat-submitted run pauses on a v0.8.19 R3
  `request_choice` (e.g. the v0.8.21 stuck-loop *Hint / Partial / Cancel* prompt), the chat-history
  entry was marked `status="failed"` with no interactive surface. The card existed on the
  `/insights` page, but the operator on the chat thread had no way to know — let alone resolve it.

## The fix — A + B + C

### A — vault auto-migration on daemon start

`systemu/runtime/vault_migrator.py` runs once per daemon boot, between the v0.7-c skill_migrator
and scheduler construction. Tools-only scope; never touches forged tools.

- **Identity by NAME**, not by the `forged_by_systemu` flag. Mis-flagged vaults (seen on several
  early installs) are auto-healed during UPDATE — the flag is cleared back to `false` so future
  runs treat them as seed tools.
- **Fast-path on `.seed_version` match** — if the vault is already on the current package version,
  the migrator returns immediately with no work.
- **ADD / UPDATE / SKIP** — new seed tools added; existing seed tools whose body or schema
  changed are updated; forged tools (genuine operator extensions, by NAME) are skipped.
- **Wild Card rewiring by `id == "shadow_wildcard"`** — the directory hash is unpredictable, so the
  migrator looks up the shadow by its stable id.
- **Off-switch:** `SYSTEMU_VAULT_AUTO_MIGRATE=off`. Safe-fail: any unexpected error logs and
  proceeds with the existing vault.
- **Atomic `.seed_version` write** (tempfile + `os.replace`) marks completion only after a clean
  run.

Net effect: `pip install -U systemu` is now enough — the next daemon start picks up the new
toolkit silently.

### B — toolkit-aware activity extractor

`systemu/pipelines/activity_extractor.py` + `systemu/prompts/extract_skills_tools.md`.

- **Body-schema fallback in `_enrich_tool_for_catalog`** — when the tool index payload is sparse,
  the extractor now reads the tool's full description and parameter schema from disk before
  showing them to the LLM. The LLM sees real tool fit signal, not just a name.
- **STRONG prompt steering** — a new `## v0.8.22 — PREFER existing tools` section instructs the
  extractor to reuse existing tools rather than forge new ones whenever an existing tool can do
  the job. Replaces the previous gentle nudge.
- **`forge_rationale` persisted** on every forged tool — the extractor must record *why* a new
  tool was needed instead of reusing existing ones. Future audits can grep this field to spot
  bad forge decisions.

Net effect: `web_extract` is selected for the bakery prompt instead of a fictional
`search_places_nearby`.

### C — inline pending-decision card in chat

`chat_submission_id` is threaded from the chat dispatch site to the decision queue, so any
R3 question raised during a chat-tied run is recognizably *part of that chat*.

- `systemu/runtime/chat_submission_ctx.py` — a `ContextVar` (the standard Python pattern for
  per-task context propagation).
- `systemu/runtime/shadow_runtime.py` — `execute()` accepts `chat_submission_id` and resets the
  ContextVar in a `try/finally` covering all 15 return paths.
- `systemu/interface/notifications.py` — the three R3 producers (`notify_user`,
  `request_credential`, `request_choice`) merge the ContextVar value into `OperatorDecision.context`
  via a shared `_v0822_merge_chat_id` helper.
- `systemu/approval/decision_queue.py` — `post()` and `resolve()` emit EventBus events
  (`operator_decision_posted` / `operator_decision_resolved`) carrying the `chat_submission_id` so
  any UI can refresh on demand.
- `systemu/pipelines/direct_task.py` — the sync chat-dispatch path catches
  `PendingOperatorDecision` and writes `status="pending_decision"` to the chat-history entry
  (redacted snapshot — `decision_id`, `dedup_key`, `options` only, no secret values).
- `systemu/interface/components/pending_decision_card.py` — thin wrapper around
  `insights.render_decision_card`, the same primitive the `/insights` page uses. Single source of
  truth — eliminates a regression class seen in v0.8.21 around R3 hint-shape rendering.
- `systemu/interface/pages/chat_page.py` — `_render_entry` calls the inline card when status is
  `pending_decision`; the page subscribes to EventBus so a decision resolved from another tab
  refreshes the chat thread (auto-unsubscribed on disconnect).

Net effect: the chat thread shows the R3 card inline. The operator resolves it in chat and the
run continues — no `/insights` round-trip required.

## Compatibility

- The migrator is purely additive on the vault side: forged tools are never touched (identity by
  NAME), and the off-switch (`SYSTEMU_VAULT_AUTO_MIGRATE=off`) preserves the prior behavior.
- The extractor change is steering and enrichment only; the tool-forge code path is unchanged.
- The chat card reuses an existing renderer; no new R3 schema or queue contract.
- No new external dependencies.

## Known limitations and follow-ups

- The queued (Supervisor) chat-submission path does not yet thread `chat_submission_id`
  end-to-end. The sync chat path (the common single-machine `local` setup) is fully covered. A
  follow-up will extend `Supervisor.submit` and the worker that calls `runtime.execute`.
- Banked from v0.8.21: `_finalize_stuck` cancel-summary wording polish; `tools_tried` should
  filter out succeeded tools. Non-blocking; deferred to a v0.8.23 cleanup batch.

## Migration

```
pip install --upgrade systemu==0.8.22
```

No configuration changes required. On first daemon start after upgrade, the vault auto-migrator
silently brings the toolkit up to date — no `--force` re-seed needed. To opt out, set
`SYSTEMU_VAULT_AUTO_MIGRATE=off`.
