# v0.9.34.4 — forge dry-run creates test files (file-reading tools validate)

A one-function patch on top of v0.9.34.3. It lets the forge dry-run validate the
most common tool shape: read a file, produce output.

## What changed

The forge dry-run rewrites a forged tool's path-like arguments into a unique tmp
directory, but it created the **directory** and **never the files**. So any forged
tool that **reads a file** failed its dry-run with *"input file does not
exist: …"* and was correctly-but-wrongly withheld — even when the tool was
entirely correct. (RCA: a model self-corrected its schema to a coherent
`run(input_path, output_path)`, but every attempt still failed because the
dry-run handed it a non-existent input path; an earlier hash tool passed only
because it happened to take *text*, not a path.) The `pathy_keys` set also omitted
input-side names, so input paths were inconsistently sandboxed.

`_sandbox_paths` now:
- recognises input-side keys (`input_path`, `input_file`, `infile`, `in_path`,
  `src`, `source`, `data_path`) alongside the existing output-side ones, and
- **creates a small test file** at each sandboxed file path, so a tool that reads
  the path finds content (the dry-run only checks the tool doesn't crash, not
  output correctness — any bytes suffice; output tools simply overwrite it).

Directory-style keys (`*_dir`, e.g. `output_dir`) get a real **directory** rather
than a file, so a tool that creates or writes into the directory still works.

This makes every file-reading forged tool dry-runnable — not just one benchmark
shape — so the forge pipeline can validate both content-taking and file-reading
tools. One function (`systemu/pipelines/tool_dry_run.py::_sandbox_paths`); the
success path is unaffected; +3 regression tests.
