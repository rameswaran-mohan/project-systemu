# v0.8.11 — honest dep-messages + resilient chat refresh + locked Wild Card

A focused bug-fix bundle: three carried-over defects from the v0.8.10 line are
closed. No new capability and no change to the seeded Wild Card's contents.

## RC1 — missing-dependency failures name the real package, never "unknown"

When a tool dry-run failed on a missing dependency, the operator-facing message
could surface the literal `unknown` (e.g. "undeclared missing package 'unknown'"
or "pip install unknown"), which is unactionable.

The dep-failure path now resolves the most honest package name in priority
order:

1. the tool result's own `missing_packages` list, when present;
2. the tool's declared manifest `dependencies`, as a fallback;
3. a clear human phrase — **"a required package (see tool manifest)"** — as the
   worst case.

The misleading `unknown` literal is gone everywhere on this path. The install
hint follows suit: a real package token yields `pip install <pkg>`, while a
human fallback phrase (or an empty list) yields "see the tool's manifest" rather
than a nonsensical `pip install <phrase>`.

The recovery engine's `DEP_PENDING` card and the runtime dep-failure builder
both apply this fallback, and the `classify_dry_run_error` callers (recovery
engine + scheduler jobs) now forward the structured `missing_packages` from
dry-run evidence so the classifier prefers the real list over regex sniffing
the moment a producer writes one.

## RC2 — chat post-run refresh survives navigating away mid-task

Running a task in **Chat** spawns a background thread that, on completion,
schedules a NiceGUI `ui.timer` to refresh the history panel. If the operator
navigated away (or the page client was deleted) before the task finished, the
timer failed **asynchronously** inside NiceGUI's connected-state check — a
failure a synchronous `try/except` around the timer could not catch, surfacing
as an unhandled "client deleted" error.

The post-run refresh is now pre-guarded: it checks the client's
`has_socket_connection` flag before scheduling the timer and skips cleanly when
the client is gone. The task work itself always completes; the completed result
shows correctly on return, with no unhandled NiceGUI error.

## Wildcard — single seeded Wild Card's wiring is verified and locked

The fresh seed ships exactly one Wild Card shadow. v0.8.11 adds regression tests
that **lock** its wiring against build pollution:

- exactly one Wild Card (`shadow_wildcard`) in the shadow-army index;
- its `tool_ids` match the curated tools index exactly (no drift);
- its `skill_ids` match the curated skills index exactly (no drift);
- its body loads cleanly through the `Shadow` model;
- a **determinism guard** asserts exactly one `shadow_shadow_*` directory on
  disk, so a polluted build can never ship extra Wild Cards (the failure mode
  guarded against from the v0.8.10 line).

No change to the Wild Card's contents or persona — these tests verify and freeze
the already-correct seed. The `.gitignore` comment for the
`!shadow_shadow_wildcard/` exception was clarified (comment only).
