# v0.9.51 — the holistic param-synthesizer + context-grounding (forge dry-run)

**Theme: one recursive engine synthesizes valid, materialized test parameters for
*any* tool schema and grounds them in the operator's real input files — dissolving
the per-shape whack-a-mole that kept file/list/nested tools from passing their forge
dry-run.**

## The problem

A forged tool only deploys after its dry-run proves it *runs*. The dry-run needs
representative inputs, which the harness had to synthesize from the tool's JSON
Schema. The old synthesizer was flat and scalar-only: it handled a top-level
`output_path` string, but a **list of paths** (`files_to_add`), a **path nested
inside an object**, a **map of paths**, or a wrapped `{type, properties, required}`
schema each needed their own special-case — and each missing case meant a correct
file tool failed its dry-run on phantom inputs (or got stuck). Every new tool shape
was a new bug. This is the class behind the **`create_password_protected_zip` /
multi-file** tasks that kept failing: a *correct* tool starved of valid inputs.

**Scope (honest):** v0.9.51 guarantees the dry-run feeds a *correct* tool valid,
materialized inputs. It is **not** a fix for a tool whose forged code is itself
wrong — e.g. the docx / `msoffcrypto` case, where the flash model doesn't know the
library API and writes broken code. That tool now gets perfect inputs and still
fails on its own code (correctly); that's a model-knowledge gap, a separate problem.

## The engine — `systemu/pipelines/fixture_synth.py`

ONE recursive schema-walk engine that emits a structurally-complete,
constraint-valid instance from an arbitrary JSON Schema and materializes a **real,
format-valid fixture file at every path-shaped string leaf, at any nesting depth**:

- **Recursive in structure** — object / array / array-of-objects /
  `additionalProperties` map / `$ref` (+ cycle guard) / `anyOf` / `allOf`. "A path
  inside a list inside an object" is the *same code* as a top-level path.
- **One path oracle** — key tokens (incl. plurals like `files_*` and ext-keys like
  `*_xlsx`), JSON-Schema `format`, `contentMediaType`, and the description all feed
  one decision, applied at every string leaf. It deliberately leans toward "path"
  (a missed path false-*fails*; a path-for-text false-positive still runs).
- **A list-of-paths gets N distinct real fixtures** (the v0.9.49 shape), each a
  parseable docx / xlsx / pdf / png / zip / json.
- **Leaf generators** — formats (email/date/uuid/uri), `enum`/`const`, numeric
  bounds (`minimum`/`exclusiveMinimum`/`multipleOf`), required→non-empty.

`dry_run_tool` now synthesizes its params via this engine instead of the flat
LLM-first `_generate_test_params` + `_sandbox_paths`. Params are **deterministic** —
the LLM is consulted only for the semantic *skip* judgment ("this tool has real side
effects, don't dry-run it").

## Graceful degradation

A constraint the engine can't satisfy without a regex engine (a `pattern`-bound
required string) is **flagged** in `SynthResult.unresolved`; if the dry-run then
fails, `dry_run_tool` degrades it to a non-doomed **operator_verify** instead of a
hard "failed" — the "never false-fail on a synthesis gap" contract. (The
execution-time `_is_format_parse_failure → operator_verify` router is unchanged.)

## Context-grounding

The forge now extracts the operator's **real input-file references** from the
forging Scroll (`raw_request` / narrative / intent) and persists them on the tool as
`Tool.grounding_inputs`. The synthesizer copies any that still **exist on disk**
into its sandbox for matching INPUT path leaves — so a content-dependent tool
(`extract the table from report.docx`) is dry-run against **real content**, not an
empty fixture. Fully graceful: a bare filename or stale path is a no-op (synthetic
fallback); output/destination leaves stay synthetic.

## Live-events Re-trigger

When the operator expands a live event that ties back to an activity — a
failed/cancelled/stuck run they're investigating — a **"↻ Re-trigger this task"**
button now re-runs it in place. The shared `render_event_details_body` gained a
`context=` param so the affordance reaches every surface it serves (/chat Live
Events, right-rail, insights); a new DRY `work.py::rerun_workflow_by_activity`
routes through the one canonical `Supervisor.submit` resubmit path (dedups
in-flight).

## Stale dry-run failures no longer permanently block a task

A `dry_run_status: failed` verdict used to be permanent: a tool that failed under
older code (e.g. before this release's deterministic engine stopped injecting a junk
`dry_run` key) stayed cached as `failed`, and any task that needed it was condemned
on the stale verdict — the harness never re-checked whether the tool actually still
fails.

The root fix is **demand-driven re-validation**: before the reaper finalizes a
parked task, it gives each of that task's required `failed` tools **one fresh
dry-run under the current code** (`revalidate_blocking_failed_tools`). A tool that
recovers → `DEPLOYED` makes the task satisfiable, so it's no longer reaped and the
existing enable/heal sweep re-dispatches it. It runs on **every reconcile tick (not
just startup)**, so an always-on daemon recovers too; it's **demand-driven** (only
tools blocking a real parked task) and **bounded** to once per `(activity, tool)` per
session, so a genuinely-broken tool re-validates at most once per task that needs it
and never loops. A complementary startup sweep + a `systemu_version` stamp on the
evidence handle the proactive "just upgraded" case.
