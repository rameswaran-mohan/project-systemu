# v0.9.49 — a declined or un-forgeable tool ends the task cleanly, not in a hang

If a task needed a tool you **declined at the forge gate** (or that could never be
forged), the task used to sit parked `waiting_on_tools` **forever** — and "Enable
& run" on the blocked card was a dead-end, since there was nothing to enable. This
release makes that a **clean terminal** with a clear message instead of a silent
hang.

## Fixed

- **Declining a forge gate finalizes the tasks that needed that tool.** When you
  Skip/decline a tool's forge review, every parked task that required it is now
  finalized immediately with a message naming the declined tool — *"Tool 'X' was
  declined at forge review; the objective needing it can't be completed"* —
  instead of hanging.
- **"Enable & run" is honest about an un-forged tool.** The readiness gate no
  longer silently no-ops when a required tool was never forged or can't deploy; it
  finalizes the task and surfaces the reason (including a failed tool's dry-run
  error).
- **The daemon-side reaper covers the declined case too.** As a fallback, the
  reconciler finalizes any parked task whose required tool can never become
  available — declined, failed validation, or missing — while leaving a tool whose
  forge is still **pending** alone (you may still approve it).

## Under the hood

One shared, idempotent finalizer (`finalize_unsatisfiable_activity`) now backs the
forge-reject event, the readiness gate, and the reaper — so "this tool can never
become available, finalize the task" is decided in exactly one place, with an
"any required tool unavailable" rule that correctly handles a task that has one
working tool and one declined tool. Each fix landed test-first (~26 new tests).

No eval/paper changes.
