# v0.9.24 — Pick your provider, per tier; validated before it runs

Wave 14. You can now choose *which* LLM provider serves each tier — and
systemu validates that choice before it runs, honors it end-to-end, and is
honest when a model later breaks.

## Choose a provider per tier

Setup and Settings now let you pick a provider — **OpenRouter, Google,
OpenAI, Anthropic, local Ollama, or Auto** — independently for each of the
three model tiers, with the credential each provider needs:

- `sharing_on setup` walks **all three tiers individually** (provider →
  model → its credential), reusing a key when the same provider repeats.
  Flags for automation: `--tier{N}-provider`, `--tier{N}-model`,
  `--anthropic-key`, `--openai-key`, `--ollama-url`.
- **Settings → LLM Routing Tiers** gains a per-tier provider dropdown and a
  "Same provider for all tiers" toggle, plus a read-only **provider
  credential status** panel (✓/✗ per provider, `.env`-sourced — keys are
  never typed in the browser).
- **Auto is the default and byte-identical to before** — existing
  OpenRouter installs keep working untouched.

## Validated before it runs

Your model choices are checked at the boundary and recorded as a trust
anchor (`data/validated_models.json`): OpenRouter against its live catalog,
Google/OpenAI/Anthropic with a 1-token ping, Ollama against its local
model list. A rejected model or missing key is named precisely and
**blocks** with a fix-it pointer — never silently swapped for something you
didn't choose.

## Honest when a model breaks

The blanket auto-fallback from v0.9.23 is now principled:

- A model you **validated** that later stops working (provider deprecation,
  catalog rename) → degrades to the tier's fallback so your task still
  finishes, **and raises a loud, visible flag** ("re-validate in Settings").
- A model that was **never validated** that the provider rejects → a
  **config error** that blocks and points you at setup/Settings. No silent
  substitution.

## Under the hood

Provider credentials are now first-class config (anthropic/openai/ollama
keys + the Ollama URL), so the router and Settings read one source; a
cross-provider key-leak path (OpenAI falling back to the OpenRouter key)
was removed; and forcing a not-yet-servable provider on the live path now
refuses clearly instead of silently degrading to OpenRouter.

> **Native Anthropic / Ollama live calls** are gated behind a clear "not
> yet servable on this path" message in this release — the full provider
> invocation migration is the tracked next step. OpenRouter, Google, and
> OpenAI tiers run end-to-end now.

## Verification

40+ new tests across config / router / validation / setup / Settings, all
keyless. Full suite green.
