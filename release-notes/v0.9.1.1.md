# v0.9.1.1 — Hotfix (output_dir + tool timeout + web_extract scraping)

**Release date:** 2026-06-07
**Type:** Hotfix for v0.9.1 issues caught in the live burrito test.
**Layer:** Still Layer 4 (Durable-Outcome Contracts) — patches to make the L4 acceptance criterion reachable on real runs.

## What changes

### Verifier output_dir precedence

The verifier now respects `user_profile.default_output_dir` (set during `sharing_on user init`) ahead of `config.output_dir` (from `SYSTEMU_OUTPUT_DIR`). Before this fix, a user who initialised their profile with a non-default output dir but didn't also set `SYSTEMU_OUTPUT_DIR` would see the verifier check `~/Documents` while the LLM wrote files to the profile path — false rejection on every claim.

### Tool timeout — per-tool slot + 60s default

- **Per-tool override**: `Tool.timeout_seconds: Optional[int] = None` model field — tools that need more wall-clock can declare it (web tools should set ~90s).
- **Default bumped 30s → 60s.** Configurable via `SYSTEMU_TOOL_DEFAULT_TIMEOUT_SECONDS`. The historical 30s default was too aggressive for web tools.

### web_extract — browser-realistic UA + anti-bot retry hint

The default `User-Agent` was `systemu/0.8 (+github.com/...)` — a dead giveaway that was being 403'd by every major site (Yelp, Reddit, TripAdvisor, Google review pages, etc.). Now:

- UA bumped to a current Chrome-on-Windows string
- Added `Accept-Language`, `DNT`, `Connection`, `Upgrade-Insecure-Requests` headers
- Caller-supplied headers still win (unchanged contract)
- `fetch_json.py` left alone — targets legitimate JSON APIs (Nominatim, Overpass) that expect polite UAs, not browsers

When a site still returns `401/403/406/429/451` (Cloudflare / PerimeterX / Akamai Bot Manager class), web_extract now returns `error_type="anti_bot_blocked"` with an actionable hint pointing the LLM to retry against a search engine URL (DuckDuckGo, Google) instead of banging on the same blocked URL.

**Live impact** (verified by re-running the burrito test): URLs that previously returned 403 now succeed for DuckDuckGo / Google / Wikipedia. Hard-blocked sites (Yelp, TripAdvisor) still 403 because they require JS rendering — but now the LLM gets a concrete retry suggestion in the error.

## Operator-facing knobs

| Env var | Default | Purpose |
|---|---|---|
| `SYSTEMU_TOOL_DEFAULT_TIMEOUT_SECONDS` | `60` | Wall-clock budget per tool call when the Tool itself doesn't override |

## Migration

- Existing tools without `timeout_seconds` set continue to use the (now 60s) default.
- Existing scrolls and user profiles are unaffected.
- No vault migration required.

## Acceptance — partial

Re-running the burrito live test on v0.9.1.1 shows: scroll generates with concrete verifier strings (Bangalore baked in from user_profile), web_extract reaches network with the new UA, anti-bot hint reaches the LLM, verifier correctly rejects false completion claims. The LLM still sometimes chooses to fake completion instead of using the retry hint — driving the LLM to actually follow the hint is prompt-engineering / L5 (Recipe Fast-Paths) work, deferred to v0.9.4.
