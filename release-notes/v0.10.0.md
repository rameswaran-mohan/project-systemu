# v0.10.0 — G-DEMO v0: receipts, not self-report

The go-getter milestone. systemu can now **act through a connected tool and then independently verify
the effect happened** — turning "the tool said it did it" into "the effect was machine-checked against
an independent read of the real resource." This release lands the automated acceptance proof of that
loop end-to-end (§12A: *open a GitHub issue from a bug note → independently read the created issue back →
a machine-verified receipt*), completing the R-A14a tier-2 MCP-only actuation milestone.

## What this release adds (on top of v0.9.78 core + v0.9.79 hard-DENY)

- **The §12A G-DEMO v0 acceptance fixture — the whole receipt path, proven end-to-end.** A hermetic,
  CI-automatable acceptance test drives a **real** MCP `create_issue` over stdio and a **real**
  independent read-back, and asserts the persisted receipt is genuinely **machine-verified**
  (`confirmed = true`, via the independent `api_readback` seam) — not merely that the run reported
  success. No network, no real token, no DNS (a public-IP-literal host); runs with the verification net
  **off** (the default), because the MCP actuation modality carries its own per-actuation obligation.
- **It closes an integration blind spot.** Before this, no test joined the two live seams — the real
  MCP-call path and the real independent read-back — so the full chain (MCP structured result → output
  guard → evidence synthesis → host-pinned https read-back → receipt) had never run together. Standing
  it up immediately surfaced and fixed a real integration quirk (the connected server's structured
  result had to carry the resource URL at the top level for the read-back to find it).
- **The money-move guarantee is proven in the same fixture.** A money-moving MCP effect drives the exact
  same path yet stays **claimed, never machine-verified** (fail-closed — a money-move is only ever
  credited through an independent, provably-fresh read-back, which this tier does not yet have). A
  money-move can never be faux-verified from a self-reported result.
- **The receipt is proven load-bearing, not decorative.** A negative control shows that if the
  independent read-back returns a *different* resource, the receipt correctly stays **claimed** — so
  `confirmed = true` genuinely depends on the read-back matching the real created resource, not on a
  coincidence.

## The release gate stays green

- The forged-network **hard-DENY CI gate** (v0.9.79) binds this and every release until the OS-kernel
  egress jail ships: a forged/untrusted network tool is refused before it can run.
- Full suite: **5550 passed**. The acceptance fixture and its money-move + wrong-read-back negative
  controls are permanent CI.

## Scope

- This is the **CI-automatable** half of the acceptance matrix (M-API on a local mock service). The
  **live cold-install** variant — a real connector and a real token against a live API — is an
  operator tryout, not run in CI.
- Still net-**off** by default; the autonomous verification net's arming remains an operator-gated step.
