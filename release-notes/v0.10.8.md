# v0.10.8 — Money-moving MCP effects can now earn a receipt (safely)

When the daemon acts through a connected MCP tool, a **money-moving** effect has always been held to the
strictest bar: it is credited only if the system can independently, *provably-freshly* read the effect
back — never on the tool's own word. Until now that proof was impossible for MCP tools, so a money-moving
MCP effect stayed **claimed, never verified** (a deliberately safe default).

This release adds the missing proof for the case where it's possible: a money-moving effect whose
result is readable at a **client-chosen idempotency key**. Before the effect runs, the system
independently checks that the key is **not yet present**; after, it reads it back and confirms it **is** —
so the effect provably happened *this run* (not a replay). Only then is it credited, through the same
hardened, host-pinned, HTTPS read-back everything else uses.

## Why this is safe

- **No new trust path.** Crediting still flows solely through the existing hardened verifier; this only
  supplies the freshness proof that path already required. A replayed (already-present) key is rejected,
  and a proof of one resource can't credit a different one.
- **The read-back target is operator-curated**, not tool-declared — so a compromised tool can't redirect
  the check to its own server to fake freshness.
- **Dormant by default.** No money-moving MCP tool is configured yet, so the live behavior is
  byte-identical to before until an operator curates one. Full suite green.

## Scope

- Built test-first: the tests exercise the full path (fresh→credited, no-proof→claimed, replay→rejected,
  wrong-resource→rejected), so the mechanism is validated and ready the moment a real tool lands.
