# v0.10.21 — A task parked on a tool approval no longer stays stuck after you approve it

Found by running the app: you ask a chat task to do something, it needs to run a tool that
requires approval (e.g. a web search), a card appears, you choose **Always allow** — and the task
just… sits there. The dashboard log shows, over and over:

```
[ResumeOnDecision] decision dec_… missing resume coords — skipping
```

## Why it happened

When a run parks on an approval gate, the resumer figures out *which* run to wake up by reading
that run's **resume snapshot**. But a tool gate raised on the agent's **very first action** parks
*before* any snapshot has been written — so there were no coordinates to resume from, and the
resumer skipped it every 15 seconds forever. Coupled to that, a tool gate's "Always allow" was only
recorded *as part of* that resume step, so when the resume bailed, the standing approval was never
saved either — meaning even re-running the task would hit the same gate again.

## The fix

The run now carries its `activity_id` + `shadow_id` as run-scoped context (the same mechanism the
command gate already used for `execution_id`) and **stamps them into the gate decision at park
time**. The resumer reads the coordinates straight from the decision — no snapshot required — so a
task parked on its first tool call resumes cleanly once you approve. If a snapshot does exist it's
still used; if not, the run simply re-runs from the start (safe: the gate is a *pre*-execution
check, so nothing ran before it), and because the approval is now recorded *before* the re-run, the
tool passes the gate the second time.

Approval recording is also decoupled from the resume dispatch: a coords-less gate (e.g. a decision
that predates this fix, mid-upgrade) now records the standing **Always allow** and stops re-logging,
so a manual re-run succeeds. The same fix applies to command gates parked on a first action.

**Safety preserved.** A *refused* snapshot (newer schema than this build) still declines rather than
starting fresh — a run that may have committed effects is never silently re-run (DEC-9). And a
coords-less **Approve once** deliberately records *nothing* (a manual re-run re-asks) so a dangling
single-use approval can't later be spent by an unrelated call.

Pure-logic changes; new unit tests cover the carriers, the coord stamping, resume-without-a-snapshot,
the coords-less standing-allow rescue, the Approve-once no-dangling-bridge guarantee, and the
refused-snapshot DEC-9 guard. Full edit-safe suite green. Adversarially reviewed.
