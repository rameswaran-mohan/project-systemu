# v0.9.26 — Anthropic & Ollama actually run

Completes the provider story from v0.9.24–0.9.25. Those releases let you
*select and validate* any provider, but the live call path could only
invoke OpenAI-shape providers (OpenRouter / Google / OpenAI) — native
Anthropic and local Ollama were gated with a "not yet servable" message.
Now they run.

## What changed

The live `llm_call` routes native-shape providers through the unified
`provider.call() → LLMResponse` path that already existed for all five
providers, and normalizes the result into the exact same shape the
OpenAI path returns:

- **Ollama** (local, no key) — calls your running daemon over HTTP.
  **Live-validated** this release: a real `qwen2.5:0.5b` call ran through
  the full path and returned its answer + token counts.
- **Anthropic** — calls the native SDK (`messages.create`). Same code
  path; needs your `ANTHROPIC_API_KEY` and the `[anthropic]` extra.

The OpenAI-shape path (OpenRouter / Google / OpenAI) is **completely
untouched** — the native branch is additive, so there's zero behavior
change for existing installs.

## Honest details

- JSON mode for native providers is recovered by the existing extraction +
  repair loop (they don't take OpenAI's `json_object`/`tools` params).
- A **validated** native model that later breaks **flags loudly and fails**
  rather than silently degrading to the OpenRouter budget model — that
  would switch provider *and* cost behind your back. A **never-validated**
  model still blocks with a fix-it pointer.

## Verification

New live-path tests (routing decision, response normalization, native
error classification) + a real Ollama call. Full suite green.
