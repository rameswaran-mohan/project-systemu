# v0.8.18 — tool credentials & connections (declare → ask → store → inject → degrade)

Tools that need a paid or keyed API now have a first-class path from declaration to secure use.
Previously, a tool that depended on a third-party API key (weather, geocoding, a paid search
backend) had **no way to ask for that key** — and the user was never consulted. Autoforged keyed-API
tools either failed silently the first time they ran or shipped with a hardcoded placeholder string
that produced a 401 on every call. There was no store, no injection point, and no graceful skip.

## Root cause

The runtime modelled tools as if every capability were free and keyless. A `Tool` could declare its
type, its dry-run status, and its implementation — but not the fact that it **requires a secret to
function**. So there was no point in the pipeline where the system could notice "this tool can't run
without `OPENWEATHER_API_KEY`", surface that to the operator, collect the value, store it safely, and
inject it at call time.

The result was the worst possible failure mode: an autoforged "get the weather" tool would be wired
up, pass its dry run (which never hit the live API), then fail on the first real invocation with a
cryptic upstream 401 — or, if the model had hallucinated a placeholder key into the implementation,
fail the same way with no signal that a credential was even involved. The operator had no breadcrumb
and no place to fix it.

## The capability

**`requires_credentials` on tools.**
`Tool` gains a `requires_credentials: list[CredentialRequirement]` field (default `[]`, fully
backward-compatible). A `CredentialRequirement` declares the env-var-shaped `key`, a human `label`,
an optional `signup_url`, an `auth_type` (`api_key` default, or `none`), and a `free_tier` hint. The
`key` is validated to be env-var-shaped at construction.

**`CredentialStore` — OS keychain with a 0600-file fallback.**
Secrets are written to the OS keychain via `keyring` (macOS Keychain, Windows Credential Manager,
Secret Service on Linux). Where no keyring backend is available, the store falls back to a
`0600`-permission JSON file under the vault dir. `mask_secret()` and `status()` only ever surface a
`last4` tail and a `present` flag — never the raw value.

**`CredentialResolver` — keyring → env → missing.**
Resolution has a strict precedence: a value connected via the store (keyring) wins, then the process
environment is consulted as a fallback, and only then is the credential reported missing. `auth_type
== "none"` always resolves present. The resolver can list `missing()` requirements for a batch and
`promote_to_env()` resolved secrets so the tool implementation reads them the same way it always has.

**Gate-4 in the tool runtime.**
`ToolRegistry.execute()` now runs a credential gate before loading and calling a tool. If a required
credential is missing and the run is interactive, it raises a pending credential request through the
decision queue; if headless (or the operator skips), it returns a structured **degraded** result
instead of crashing.

**Pre-flight, batched ask.**
Before an activity runs, the pipeline scans the tools it will use and posts a **single batched**
credential request for everything that's missing — so the operator is asked once, up front, rather
than interrupted mid-run for each tool in turn.

**A Connections section in Settings.**
Settings now has a Connections view that enumerates every declared credential requirement across the
tool vault, shows whether each is connected (with a masked `last4`), links the signup URL, and lets
the operator paste and save a key straight into the store.

**Graceful degradation — reuses the v0.8.17 shape.**
A missing credential produces the same `success: False` / `degraded: True` / `note` envelope
introduced for degraded web search in v0.8.17, with `error_type: "tool_credential_missing"`. The
Shadow runtime treats it as a soft, non-retryable skip — no spinning to MAX_ITERATIONS — and the
`note` tells the model the tool is unavailable until the operator connects the key.

**Autoforge declares credential needs.**
The forge tool-spec prompt now instructs the model to populate `requires_credentials` when it builds
a keyed-API tool, so newly forged tools arrive with their credential needs declared instead of a
hardcoded placeholder.

## UX safeguards

- **Pre-flight, not mid-run.** The operator is asked before the activity starts, never interrupted
  partway through a multi-step run.
- **Batched.** One request covers every missing key for the activity, not one prompt per tool.
- **Silent on success.** If everything resolves (already in keyring or env), nothing is surfaced —
  no noise.
- **Seed tools never block.** The curated seed/Wild Card capability set is keyless by design, so a
  fresh install is never gated on a credential prompt.
- **Ask-once.** Requests dedup by key (`cred:<KEY>`) and by activity (`creds:<activity_id>`), so the
  operator isn't re-asked for a key they already declined or supplied.

## Security

- **Secrets never leak.** No raw credential value is ever written to logs, the event log,
  transcripts, or the model context. The degraded result and every status surface carry only the
  `label`, the masked `last4`, and a `present` flag.
- **Keyring at rest.** Values live in the OS keychain; the file fallback is `0600` and lives under
  the vault dir.
- **Masked everywhere.** `mask_secret()` renders values as `<redacted:cdef>`; `status()` exposes only
  the last four characters.

## Back-compat

Existing environment keys keep working unchanged. `OPENROUTER_API_KEY`, `SYSTEMU_TAVILY_API_KEY`,
`SYSTEMU_EXA_API_KEY`, and the rest are picked up by the resolver's **env fallback** exactly as
before — zero migration required. Tools with no `requires_credentials` (every tool that predates this
release) behave identically; the field defaults to an empty list.

## Migration

```
pip install --upgrade systemu==0.8.18
```

`keyring` installs automatically with the upgrade. No vault reseed is required — existing vault
contents, activities, and executions are fully compatible, and existing env-var API keys keep working
via the env fallback.

**Connect paid-tool keys (optional, only if you use keyed tools):**
open **Settings → Connections** and paste a key for any tool that declares a requirement. The value
goes into your OS keychain.

**Optional credential policy:**

```
SYSTEMU_CREDENTIAL_POLICY=prompt     # default — ask the operator once (via queue dedup), then degrade if skipped
SYSTEMU_CREDENTIAL_POLICY=degrade    # never prompt; skip keyed tools with a degraded result
```

Restart the daemon after changing the policy.
