# v0.9.5 — Goal-Level Orchestration (Layer 6)

**Release date:** 2026-06-07
**Layer:** 6 of 7 in the assistant-foundation arc
**Headline:** L6 delivers the architectural prerequisite L5 was missing — v2-registered LLM tools are now visible to the agent — plus delegation, peer-review, and MCP wrappers.

## What changes

### T0: v2 tool visibility (the L4/L5 unlock)
- New helper `_build_llm_tool_catalog(vault, config)` in `shadow_runtime.py` enumerates v2-registered tools and adds them to the LLM's prompt-time catalog alongside v1 vault tools.
- check_fn-gated tools are excluded from the catalog (so the LLM doesn't waste turns on unavailable tools).
- v1 wins on name collisions — auto-forged vault tools are intentional.
- **The bundled v0.9.3 file_tools (read_file/write_file/search_files) + v0.9.4 skill_tools + v0.9.5 delegate/mixture/mcp tools are now actually callable from the LLM.**

### T1: delegate.spawn_subagent (Hermes pattern)
- `systemu/runtime/tools/delegate.py` registers `spawn_subagent` in the v2 registry under `toolset="delegate"`.
- Depth-capped by `Config.delegate_max_depth` (default 3). Child inherits parent's tool whitelist minus `delegate`/`spawn_subagent` (no recursion).
- `dynamic_schema_overrides` (Hermes pattern) surfaces live `max_depth` + `max_turns_per_child` + `max_concurrent_children` in the tool description so the LLM sees the current limits at every render.
- For v0.9.5 the child is a single Tier-1 LLM call (planning + Q&A). Full recursive ShadowRuntime as the child runtime is v0.9.6+ work.

### T2: 3 delegate Config fields + `ToolEntry.dynamic_schema_overrides`
- `Config.delegate_max_depth` (default 3, `SYSTEMU_DELEGATE_MAX_DEPTH`)
- `Config.delegate_max_concurrent_children` (default 2, `SYSTEMU_DELEGATE_MAX_CONCURRENT_CHILDREN`)
- `Config.delegate_max_turns_per_child` (default 20, `SYSTEMU_DELEGATE_MAX_TURNS_PER_CHILD`)
- `ToolEntry.dynamic_schema_overrides: Optional[Callable[[], Dict[str, Any]]]` — zero-arg callable applied to the LLM-facing schema at render time.

### T3: MCP wrapper (no OAuth — deferred to L7)
- `systemu/runtime/mcp/client.py` ships `mcp_call_tool(server, name, params)` — a lightweight httpx-based wrapper around the MCP server `/tools/call` endpoint.
- `parse_servers()` parses the comma-separated `SYSTEMU_MCP_SERVER_URLS` env var.
- Registered as `mcp_call_tool` in the v2 registry, `toolset="mcp"`, `is_action_tool=True`.
- OAuth-protected MCP servers (Hermes `mcp_oauth_manager` pattern) are deferred to v0.9.6+ — operator brings their own non-auth-protected server URLs for now.

### T4: mixture_of_agents tool (Hermes peer-review pattern)
- `systemu/runtime/tools/mixture_of_agents.py` registers `mixture_of_agents` in the v2 registry under `toolset="verification"`.
- Spawns N Tier-1 LLM peer reviewers with diverse lenses (default: correctness, completeness, edge-cases), aggregates yes/no verdicts, returns majority.
- Conservative tie-break: ties → no.
- LLM exceptions count as no.
- For use BEFORE claiming completion on high-stakes or irreversible decisions.

### Elder-intake nudge (carryover from v0.9.4 backlog + new in v0.9.5)
- `systemu/prompts/elder_intake.md` now instructs the LLM: before generating objectives, call `skill_list_skills` to scan available recipes. If one matches the user's intent, call `skill_view_skill` to load it. Follow the Procedure section step by step.
- Combined with T0's tool visibility, this should make the burrito test actually consume the `burrito-delivery` recipe and produce real files on disk.

## Migration

- Existing v0.9.0–v0.9.4 features untouched.

## Acceptance criterion

The burrito test should now produce `burrito_raw.json` and `burrito_ranked.json` on disk because:
1. The LLM sees `skill_list_skills` in its tool catalog (T0).
2. The elder_intake nudge instructs the LLM to scan available skills first.
3. The LLM calls `skill_view_skill(name="burrito-delivery")` and loads the recipe.
4. The recipe prescribes the DuckDuckGo-first workflow + explicit `write_file` calls.

## Live test result — honest assessment

Ran the burrito acceptance test against v0.9.5 (Bangalore profile, all gates enabled):

**What worked:**
- T0 unit tests confirm `_build_llm_tool_catalog(vault=None)` returns 8 v2-registered tools cleanly
- Standalone Python check confirms all 8 v2 tools (file_tools + skill_tools + delegate + mixture + mcp) are reachable
- elder_intake nudge guard tests pass (prompt instructs the LLM to call `skill_list_skills` first)
- web_search invocations: 3 successes (up from 2 in v0.9.4)

**What didn't work:**
- The LLM in the burrito test never called `skill_list_skills`, `skill_view_skill`, `spawn_subagent`, `mixture_of_agents`, or any other v2 tool
- The output directory remains empty (no burrito_raw.json / burrito_ranked.json)
- Root cause: T0's wiring at `shadow_runtime.py:1744` augments `tool_index`, but at scroll-execution time the LLM's actual prompt-time tool catalog is assembled from the activity's `required_tool_ids` (vault-stored tools only) via the activity-extractor → scroll-refiner pipeline. The v2 augmentation is at the wrong scope.

**Conclusion:** v0.9.5 ships the infrastructure (delegate, mixture, MCP wrapper, dynamic_schema_overrides, Config knobs) but the **end-to-end v2 tool visibility from registry to LLM prompt** is incomplete. The runtime needs a second augmentation point in the scroll-execution pipeline where `required_tool_ids` → LLM tool list happens. This becomes a hard L7 prerequisite.

The L1–L4 infrastructure (verifier, capability ledger, audit log, episodic memory) continues to work correctly. L5 + L6 ship the right shapes — they just don't have the wiring to reach the LLM yet. v0.9.6 L7 will close this last gap.
