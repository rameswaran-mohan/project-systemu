# v0.9.2 — Episodic Memory (Layer 2)

**Release date:** 2026-06-07
**Layer:** 2 of 7 in the assistant-foundation arc
**Headline:** systemu now remembers across sessions. Every run summarizes itself; future runs can search and recall.

## What changes

- **SessionSummary** stored after every completed (or partial) run — intent, outcome, key facts surfaced, files produced, tags. Tier-1 LLM generates the summary; you can disable via `SYSTEMU_SUMMARIZE_AFTER_RUN=false`.
- **Vault** gains `append_session_summary`, `query_session_summaries`, `search_session_summaries` with backend-aware implementations:
  - File backend: JSONL at `vault/episodic/sessions.jsonl`
  - SQLite backend: `session_summaries` table + FTS5 virtual table with porter+unicode61 tokenizer
  - Postgres backend: `session_summaries` table + tsvector column + GIN index, search via `plainto_tsquery`
- **Two new session-tool Python functions** (CLI-callable in v0.9.2): `session_search(query)` and `session_recall(session_id)` — read-only utilities for surfacing the episodic log. LLM-side tool dispatch wiring lands in v0.9.3 once the v2 tool registry ships; this release provides the underlying machinery and CLI access.
- **CLI commands**: `sharing_on session list`, `sharing_on session show <session_id>`, `sharing_on session search <query>`.
- **5 new Config fields** for operator tuning (all `SYSTEMU_*` env vars).
- **Runtime hooks** in `shadow_runtime` (3 finalize sites: success, fail, stuck) and `direct_task` (chat-resolve) trigger summarization automatically. Best-effort — a flaky LLM never blocks the user's task.

## Operator-facing knobs

| Env var | Default | Purpose |
|---|---|---|
| `SYSTEMU_EPISODIC_MEMORY_ENABLED` | `true` | Master kill switch (also disables read paths) |
| `SYSTEMU_SUMMARIZE_AFTER_RUN` | `true` | Trigger summarization on run-finalize |
| `SYSTEMU_EPISODIC_SEARCH_DEFAULT_LIMIT` | `5` | Default result count for session_search |
| `SYSTEMU_EPISODIC_SUMMARY_MAX_CHARS` | `800` | Truncate the LLM-generated outcome_summary |
| `SYSTEMU_EPISODIC_TAGS_MAX_COUNT` | `8` | Max tags per session |

## Migration

- No vault migration required — file backend creates `vault/episodic/sessions.jsonl` on first write.
- sqlite/postgres backends auto-create the `session_summaries` table + indexes on vault open.
- Existing scrolls, user profiles, action audit logs are unaffected.
- Existing tools and verifier behavior untouched.

## Privacy

Session summaries include the intent + outcome + tags + files produced. They are stored locally under `vault/`. The Tier-1 summarizer LLM receives the intent + chat_result + files_produced list — operators should be aware that, depending on what their tasks include, this may surface to whichever LLM provider their `tier1_model` points at.

Disable per-run via `SYSTEMU_SUMMARIZE_AFTER_RUN=false` or globally via `SYSTEMU_EPISODIC_MEMORY_ENABLED=false`.
