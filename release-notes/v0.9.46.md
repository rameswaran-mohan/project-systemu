# v0.9.46 — amend-then-approve: edit a capability request before granting it

When the agent pulls a capability and the request escalates to you, you can now
**edit the request's spec before approving it** — narrow an access scope, fix a
forge parameter, trim a subagent budget — instead of a blunt yes/no on whatever
the agent asked for.

## New — Edit a harness capability request

Every harness **capability** card (tool / skill / access / compute / subagent /
mcp) now shows **Deny · Approve · Edit**. "Edit" opens the request's spec in a
JSON editor; **"Approve amended"** grants the *edited* spec. The dead "Edit spec"
stub (which silently behaved like Approve) is gone — Deny/Approve only, plus the
real Edit affordance.

The edited spec is granted through a single arbitrated path (`Governor.grant`):

- It is **re-arbitrated** (with the per-run request cap neutralized — your approval
  isn't a new request). A **hard-safety DENY** (e.g. an MCP server pointed at an
  IP literal — SSRF) is blocked; an ordinary escalation is granted on your
  authority.
- If your edit **raises the risk band** (e.g. access `read → write`), it requires
  a typed **`CONFIRM`** and is recorded — so a scope increase is never silent. A
  same-or-lower-risk edit grants with no extra friction. This guard is enforced at
  the grant point, not just in the UI.
- The grant is recorded with full provenance (original vs amended spec, original vs
  fresh risk band, whether a band increase was confirmed).

## Under the hood

- `Governor.grant` is now the single front door to a capability grant — the daemon
  reconciler no longer hand-builds a forced verdict, so **every** grant (edited or
  not) is arbitrated. Unedited "Approve" is behavior-preserving.
- The amend affordance is surfaced on the **Inbox** (the primary review surface),
  not just secondary panes.
- COMPUTE/SUBAGENT grants stay clamped to policy ceilings; ACCESS remains an
  advisory lease (amending it narrows the recorded intent, not an enforced sandbox).

No eval/paper changes.
