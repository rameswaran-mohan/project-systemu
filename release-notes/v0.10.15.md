# v0.10.15 — "You may already have this" at the forge gate (R-CAP1, forge governance)

Before the assistant forges a brand-new tool, it now checks whether you already have one that does the
same kind of thing — and says so, so you can extend the existing tool instead of accumulating
duplicates.

## What ships

- **A same-slot dedup advisory.** When the forge confirmation appears, if an existing tool already
  occupies the proposed tool's *capability slot* (e.g. both `create:issue`), the card adds a heads-up:
  *"this shares the create:issue capability with existing tool(s): X — consider extending one instead
  of forging a duplicate."*
- **It informs, it never blocks.** The advisory only adds a line to the confirmation — the choices are
  unchanged, and you can still forge. If the check can't run for any reason, the forge proceeds exactly
  as before. It reads a fresh in-memory view and never writes anything.

## Where R-CAP1 stands

This completes the safely-shippable core of Capability Slots (§5.5.1): a searchable capability index
(v0.10.13), relevance-ordered live tool selection (v0.10.14), and now duplicate-aware forging. Every
piece is additive and adversarially reviewed.

The remaining governance pieces are deliberately deferred because each has a real precondition:
**hard forge-routing** (forcing select-or-extend-or-justify) needs an operator-confirmed-exit flow;
the **differential dry-run** dedup check must run inside the network-egress jail (not yet on the main
build), so it stays static-only until then; **structured tool lineage** needs a model field; and the
**avoidable-forge metric** rides a later measurement release. These are documented, not dropped.

Full suite green; adversarial review clean.
