# v0.8.15 — browser tools run in a subprocess (sync-Playwright fix)

A focused runtime fix. Shadows that needed the web — including capture-built
shadows — were failing because every browser tool errored out. This restores
them. No new capability surface.

## What was broken

The seeded browser tools (`web_navigate`, `web_click`, `web_extract_text`,
`tool_type=browser_action`) use Playwright's **synchronous** API. At shadow
runtime in local mode, `ShadowRuntime` executes tools through the `ToolRegistry`
**in-process fast path**, which runs them via `loop.run_in_executor(...)` —
inside the running asyncio ReAct loop. Playwright's sync API refuses to run in
that async context and raised:

```
Tool web_navigate failed: It looks like you are using Playwright Sync API inside the asyncio loop.
```

The error was returned as a failed result (not raised), so the call never fell
through to the subprocess path — the tool simply failed, the shadow couldn't
browse, and execution gave up after retries. Any shadow needing a browser tool
failed identically; a capture-built shadow (e.g. one that had to navigate the
web) just happened to surface it.

Telling contrast: the **dry-run** path already runs tools in a **subprocess**
(no asyncio loop), which is exactly why these tools *passed dry-run* but *failed
at runtime*.

## The fix

Browser tools now take the same subprocess path at runtime that they already
take during dry-run. A fresh subprocess has no asyncio loop, so the sync
Playwright API works normally.

- `ToolSandbox.execute_tool` gained a `tool_type` parameter and a
  `_must_use_subprocess(tool_type, deps)` predicate. When a tool is
  `browser_action` (or declares `playwright`/`selenium`/`playwright-stealth`),
  the in-process fast path is skipped and the tool runs through the existing
  subprocess backend. All other tools keep the fast path unchanged.
- `ShadowRuntime` passes `tool_type=<tool>.tool_type.value` (the string
  `"browser_action"`, not the enum repr) so the routing actually fires.
- `PLAYWRIGHT_BROWSERS_PATH` was added to the subprocess env allow-list
  (`_SAFE_ENV_KEYS`) so a custom browser location propagates to the child
  (default-location chromium already resolves via `USERPROFILE`/`HOME`).

Non-browser tools, dry-run, Docker/SSH/WSL backends, and the async fast path for
everything else are unchanged.

## Migration

`pip install --upgrade systemu==0.8.15` and restart the daemon. After upgrading,
web shadows (capture- or chat-based) execute their browser tools instead of
failing with the Playwright sync-API error. (Future follow-up: optionally
rewrite the seeded browser tools to Playwright's async API; not required for this
fix.)
