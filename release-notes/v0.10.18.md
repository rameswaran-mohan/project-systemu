# v0.10.18 — Measuring redundant tool-forging (R-A13.5, avoidable-forge)

The system now measures how often it forged a new tool that duplicated one it already had — a
deterministic health metric, no AI judgment involved.

## What ships

- **`sharing-on debug avoidable-forge`** — for every tool the system forged, it re-runs the
  capability-slot lookup with hindsight: does an existing tool already occupy that tool's slot (would
  it have bound instead of forging a duplicate)? It reports the rate plus which forged tools duplicate
  which existing ones.
- **Deterministic + read-only.** It's a post-hoc replay over the capability index — same inputs, same
  numbers, every time (no model call, no randomness). It writes nothing.

## Why it matters

This is the first metric of the measurement program the roadmap has been accreting toward. The
avoidable-forge rate is the tripwire that will decide whether the deliberately embedding-free tool
selection (shipped in v0.10.13–15) ever needs revisiting — if redundant forging stays low, the simple
lexical approach is validated by data. It's reported alongside the avoidable-ask rate (the next slice,
which decides the quick-lane ask policy).

An adversarial review (2 lenses × refute-by-default panels) checked the metric's soundness (no
misleading double-count, honest about what it can and can't see) and its read-only/deterministic
guarantees. Full suite green.

## Scope

This is the avoidable-**forge** half. The avoidable-**ask** half needs a recorded corpus of the
questions the system asked (with the context it had at the time), so its fixture format and the
replay are the next slice.
