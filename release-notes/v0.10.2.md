# v0.10.2 — Forge no longer executes unproven tool code to check it

A security hardening. When the daemon writes a tool for itself, it runs a quick
**conformance check** to confirm the tool's `run()` can be called the way the
registry calls it. That check used to *execute* the freshly-generated code to read
the signature — which meant any top-level statement in that not-yet-vetted code ran
in the daemon at write time, before the tool had ever passed a single safety gate.

Now the check reads the `run()` signature **statically**, without executing the tool
at all. A generated tool that (accidentally or maliciously) reached the network or
touched the filesystem at import time can no longer do so during the conformance
check — that only happens later, through the normal execution path, which is gated
by the forged-network hard-DENY (v0.9.79).

## What changed

- The forge conformance check parses the tool source and inspects `run()`'s
  parameters directly (positional, keyword-only, `**kwargs`, defaults, `async def`)
  instead of importing the module. Same conformance verdicts as before — proven by
  the existing conformance tests plus new parity tests — with **zero execution** of
  the generated code.
- A regression test asserts that module-level code in a generated tool is never run
  during the check.

## Scope

- Behavior-preserving for legitimate tools (all 198 forge tests green; full suite
  green). This closes a Lane-1 spine-audit follow-up finding.
