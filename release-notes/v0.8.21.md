# v0.8.21 — extract from pages + stop spinning on dead ends

A capability + control release. The agent can now turn a fetched page (or a raw URL) into structured
records, and the runtime no longer rides a stuck loop to MAX_ITERATIONS when a tool keeps failing or
no objective is being credited. Two new seed tools, one new runtime guard, one new live-editable
Settings section. No API change to existing tools, no new external dependencies.

## What went wrong (RCA)

The "find top 5 bakeries near me" run failed at `MAX_ITERATIONS=30` with `status="partial"`. The agent
could fetch a Justdial directory page (HTTP-level success), but had no tool to turn that page's
listings into a structured `[{name, address, ...}]` list. With no extraction primitive and no
early-stop signal, it spun: same two tools, same dead-end, 30 iterations, no objective credited.

## The capability fix — extract from a page

Two purely additive seed tools, with descriptions explicitly steering the agent to the right one:

- **`extract_records`** — HTML or text → records, via the Tier-3 LLM. Defended in four layers:
  1. sanitize untrusted markup (drop scripts/iframes/etc.),
  2. truncate to a fixed token budget,
  3. wrap the page in an *untrusted-frame* system prompt so prompt-injection in the page can't
     override the operator's schema,
  4. validate the LLM output against the caller-supplied JSON schema before returning.
  POC-validated against the same live URL that failed the bakery run.
- **`web_extract`** — URL → records. Composes v0.8.20's polite-headers fetch with `extract_records`,
  adds lightweight pagination (`max_pages` 1–5; follows `rel=next` and `?page=N` patterns), and
  detects soft-blocks / empty bodies before spending an LLM call. The Justdial 14-byte
  `<HTML></HTML>` response now returns a degraded shape immediately instead of feeding emptiness
  to the model.

Tool descriptions steer the agent: use **`web_extract`** when you're starting from a URL; use
**`extract_records`** *after* `web_read` / `fetch_html` when you already have content in hand.

## The control fix — stuck-loop guard

A new guard in `shadow_runtime` watches for two stuck-loop signatures, with a **hybrid trigger**
(either is enough):

- **a)** N iterations without crediting any objective, or
- **b)** the same tool failing N consecutive times.

When triggered, behavior depends on the surface:

- **Operator-attended runs** pause via the v0.8.19 R3 `request_choice` prompt with three options —
  *Hint* (operator types a one-shot nudge that gets folded into the next iteration's user payload),
  *Accept partial* (gracefully finalize with whatever is in hand), or *Cancel run*.
- **Headless runs** degrade to a structured `FAIL` with
  `error_type:"stuck_loop"`, `stuck_on_objective`, `tools_tried`, and a human-readable `reason`.
  They never just hang on the iteration cap.

The progress counter resets the moment an objective is credited, so a productive run is never
interrupted.

## Settings page — live-editable thresholds

A new **Stuck-loop guard** section between Behaviour and Storage:

- Checkbox: *Enable stuck-loop guard*.
- Int input: *Iterations without progress* (1–30).
- Int input: *Consecutive tool failures* (1–10).

Saves write to `.env` *and* patch `os.environ` for the live process — the next iteration picks up
the new threshold without a daemon restart.

## Compatibility

- The two new tools are purely additive; no existing tool changed.
- Stuck-guard defaults are safe-by-default: counter resets on every objective credit, default
  thresholds (5 iterations without progress, 3 consecutive same-tool failures) are generous, and
  `SYSTEMU_STUCK_GUARD=off` disables the guard entirely.
- No new external dependencies — reuses the already-bundled Tier-3 LLM and `jsonschema`.

> **Note for existing vaults:** the two new tools land in `systemu/vault/tools/...` in the package,
> but existing vaults keep their *copied* tools until they are re-seeded. A fresh `sharing_on init`
> from a clean directory (or `--force`) picks up `extract_records` and `web_extract`; a package
> upgrade alone does **not** rewrite tools already seeded into an existing vault.

## Migration

```
pip install --upgrade systemu==0.8.21
```

No new dependencies and no configuration changes are required. The stuck-guard is on by default with
safe thresholds; tune or disable it from the Settings page. To pick up `extract_records` and
`web_extract` in an existing vault, re-seed it (`sharing_on init` into a fresh directory, or
`--force`) — a package upgrade alone does not rewrite already-seeded vault tools.
