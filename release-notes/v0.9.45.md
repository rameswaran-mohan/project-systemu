# v0.9.45 — "ask the operator for a value" works in every lane (+ rail height tweak)

The root fix for operator-input: a free-text "ask the operator" question now
renders a real answer box and the agent receives the typed value — instead of
capturing a button label and re-asking.

## Fixed — free-text ASK_OPERATOR no longer loops

When the agent needs a value it can't infer (e.g. *"find the SHA of any given
number"* → *"What number?"*), the workflow lane surfaced a generic harness gate
(Deny / Approve / Edit spec) with **no field to type the value into**. The
reconciler then treated the button label as the operator's answer, the agent still
had no value, and it re-asked until the run-tree request budget ran out — a
futility loop.

Now a free-text input request **synthesizes a one-field schema**, so the card
renders the SAME answer form the missing-parameter and MCP-elicitation paths
already use. The operator types the value, the reconciler extracts the **clean
value** (e.g. `42` — not the raw form JSON or a button label), and the agent
continues in **one round-trip**.

This converges all four "ask the operator" paths — quick-lane `ASK_USER`,
missing-parameter, MCP elicitation, and now free-text `ASK_OPERATOR` — onto the
same answer-capture widget. Capability requests (tool / skill / access / …) are
untouched (the spec preview is preserved).

## Also

- A **parked** run (awaiting your input) now logs as WARNING, not ERROR — it no
  longer looks *failed* in the Live feed.
- The decision-card Context panel is sanitized before serialization, ending the
  recurring `nicegui … Type is not JSON serializable: function` error.
- **Right rail:** "Needs you" height reduced (220→150px) and "Live" increased
  (280→350px) — same total, more room for the live feed.

No eval/paper changes.
